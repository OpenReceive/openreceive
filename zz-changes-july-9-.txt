OpenReceive — DX & Contract Recommendation
Date: 2026-07-09
Author: review synthesis (Claude), grounded in code inspection. Review-only; no code changed.
Framing constraint (from maintainer): the host app owns order creation and checkout/cart
logic. OpenReceive is ONLY the payment layer (routes, tokens, settlement, swaps). Every
recommendation below is chosen to make that boundary crisper, not blurrier.


================================================================================
TL;DR
================================================================================
The proposal and the second review are directionally right. But the highest-leverage
change is NOT docs polish — it is fixing the amount-authority contract, which is currently
inconsistent across four shipped surfaces and quietly violates the project's own stated
security invariant. Fix that first; it is the design rail everything else hangs on. Then do
the packaging + quickstart simplification. Do NOT build a no-orderId / prepare checkout mode.

Two design rails to hold everything against:
  1. The host owns order creation.          (OpenReceive never mints orders.)
  2. OpenReceive never trusts a client price. (The host must resolve the price server-side.)


================================================================================
FINDING 1 (P0 — the rail): amount authority is broken across four surfaces
================================================================================
This was verified in code, not assumed. Four shipped surfaces disagree about who sets the
price on create-checkout:

  A. STATED INVARIANT — client price must never be trusted:
       packages/js/node/src/get-order-amount.ts:5
         "The shipped create-checkout route MUST NOT trust a client-supplied price."
       docs/guides/routes.md:101  (repeats the MUST NOT)

  B. NODE HANDLER — silently trusts the client when the hook is omitted:
       packages/js/http/src/handler.ts:112-116   (only a console.warn)
       packages/js/http/src/handler.ts:190-192   ("fall back to the (trusted) client amount")

  C. RUBY HANDLER — same permissive fallback + warn (parity, but in the wrong direction):
       packages/ruby/openreceive-server/lib/openreceive/server/request_handler.rb:79-83
       packages/ruby/openreceive-server/lib/openreceive/server/request_handler.rb:271-280 (apply_client_amount)

  D. BROWSER CLIENT — the shipped happy path posts ONLY { order_id }, no amount, and
     explicitly relies on the server pricing it:
       packages/js/browser/src/internal/checkout.ts:485-488
       packages/js/browser/src/internal/checkout.ts:581-587

  E. OPENAPI SPEC — marks `amount` REQUIRED on the create body:
       spec/openapi/openreceive-http.v1.yaml:223-225

Consequences:
  - E contradicts D directly: the shipped browser client's happy-path body ({ order_id })
    would FAIL the documented OpenAPI schema (amount required).
  - B/C contradict A: the "MUST NOT trust client price" invariant is downgraded to a warning,
    and routes.md:110 softens it further to "acceptable for local development." A dev who
    forgets the hook ships a route that lets a tampered client pay 1 sat for a $50 order —
    silently, with only a log line.

RECOMMENDATION — make server-side pricing MANDATORY at the shipped route layer:
  1. Require the pricing hook in the shipped handler. Omitting it should THROW at
     construction (both engines), not warn. Delete the client-amount fallback branch
     (handler.ts:190-192; Ruby apply_client_amount path).
  2. The create-checkout ROUTE body becomes { order_id, memo?, description_hash?, metadata? }
     — no `amount`. If a client sends `amount`, reject 400 (fail loud). The route obtains the
     price ONLY from the hook.
  3. Update OpenAPI: drop `amount` from required and from CreateCheckoutRequest properties
     (spec/openapi/openreceive-http.v1.yaml:220-241). Update golden/test vectors that post an
     amount on create. The browser client (D) already conforms — no change needed there.
  4. Rewrite routes.md "Amount authority" (docs/guides/routes.md:99-111): remove the
     "acceptable for local development" language; the hook is required, full stop.
  5. Keep the amount-authority seam at the ROUTE layer only. The low-level service method
     (getOrCreateCheckout) still accepts an explicit amount — that is the correct, trusted
     entry point (the hook's return flows through it, and advanced/testing code can call it
     directly). Client trust is removed from the HTTP surface, not from the library.

IMPORTANT — this does NOT kill client-priced / tip-jar / donation checkouts:
  The hook still receives `clientAmount` (marked UNTRUSTED, get-order-amount.ts:20-21). A host
  that genuinely wants a payer-chosen amount reads and validates clientAmount INSIDE the hook
  and returns it. So "trust the client" becomes an explicit, visible, host-owned decision
  instead of a silent framework default. That is exactly rail #2.

Why this is P0: it is the one change that touches security, the OpenAPI contract, the browser
behavior, the docs, the golden vectors, and Ruby parity simultaneously. Doing it first means
the rename and the quickstart rewrite are written ONCE against the correct contract.


================================================================================
FINDING 2 (P0, same change): rename getOrderAmount -> resolveOrder
================================================================================
"getOrderAmount" reads like a calculator. The hook conceptually means: "load the host's order
and return its payment terms." Rename to `resolveOrder` (Ruby: `resolve_order`). Fold this into
the P0 change so the contract and the name land together.

  Signature/semantics after the change:
    resolveOrder: async ({ orderId, clientAmount, metadata, request }) => AmountSource | null
      - return { usd } | { sats } | { amount: {...} }  -> authoritative price
      - return null                                    -> 404 (order not found / rejected)
      - throw                                          -> 400 (validation)
    (Today a null return would throw a 500 TypeError via the `in` operator — codify null->404.)

  Ruby: mirror as resolve_order, keyword form ->(order_id:, client_amount:, metadata:, request:).

Naming note: prefer `resolveOrder` over bare `order` — `order` over-implies returning the order
entity; the hook returns payment terms (+ existence), not the cart.


================================================================================
FINDING 3 (P1): one canonical import surface
================================================================================
Verified: the umbrella `openreceive` package exports ./node, ./react, ./vue, ./svelte,
./angular, ./elements, ./provider-data, ./contracts — but NOT the server adapters, and does
not depend on them (packages/js/openreceive/package.json:12-64). So today the Node happy path
still teaches 3-4 separate @openreceive/* packages on line one (node + express + http + react).

RECOMMENDATION:
  - Add subpath exports: openreceive/express, openreceive/fastify, openreceive/next, each
    re-exporting createOpenReceive + its adapter, so:
        import { createOpenReceive, openReceiveExpress } from "openreceive/express";
        import { Checkout } from "openreceive/react";
    and the install line becomes:  npm install openreceive express react react-dom
  - PEER-DEP CARE (do not regress bundles): the express/fastify/next adapters pull in
    @openreceive/node (postgres, sqlite, swap providers). Wire them as optional/peer deps so a
    browser-only or react-only consumer of `openreceive` never drags the server graph into a
    client bundle. Subpath exports are the right DX; just keep the server side opt-in.
  - This is "finishing a direction the repo already started," not inventing one — but it IS
    real packaging work (new deps + exports + peer-dep design), not a one-line re-export.


================================================================================
FINDING 4 (P2): rewrite quickstart-node.md around the already-good <Checkout orderId> flow
================================================================================
The frontend story is already excellent: <Checkout orderId> creates the checkout, captures the
capability token, and polls automatically (packages/js/react/src/checkout.ts:32+). The quickstart
reads as hard mostly because it (a) hides the host-order-creation step and (b) explains machinery
the developer must not touch before they ever see a paid checkout.

New quickstart shape (shorter, but NOT by hiding the host-order step):
  1. Install:            npm install openreceive express react react-dom
  2. Configure:          receive-only OPENRECEIVE_NWC (+ a defaults table for the rest)
  3. YOUR app creates and persists an order   <-- make this an explicit numbered step.
                                                  This is the invisible prerequisite today; it
                                                  is hidden behind loadYourOrder() at
                                                  quickstart-node.md:87. The shipped router mounts
                                                  NO create-order route (by design — rail #1), so
                                                  a reader who pastes the quickstart cannot
                                                  succeed until this step is shown.
  4. Mount:              openReceiveExpress({ service, resolveOrder, onPaid })
  5. Render:             <Checkout orderId={order.id} />

Cuts:
  - Remove the entire "What is the order access token" section (quickstart-node.md:120-139).
    "You never manage it" is enough here; move the detail to security.md / routes.md. Token
    handling is fully automatic on both server (httpOnly cookie + hash storage) and client
    (browser replays it) — verified.
  - Demote the "Prefer to call the methods directly?" coda (quickstart-node.md:160-165) to an
    advanced page. One canonical story only.
  - Collapse the three amount return shapes to { usd } in the happy path; keep { sats } /
    { amount: { fiat } } as an advanced escape hatch.
  - Shrink the config block to OPENRECEIVE_NWC + a short defaults table (store=sqlite,
    prefix=/openreceive, tokens=auto, currencies=USD are already the real defaults).


================================================================================
FINDING 5 (P3): settlement sweep — make it an explicit opt-in helper, not an adapter default
================================================================================
Position updated after reading the code. Settlement is already best-effort swept on both
checkout creation and status reads via a single global cursor:
    packages/js/node/src/service/checkouts.ts:101-109  (scheduleBestEffortSweep on create)
    (and on every getOrder status read)
The cursor deliberately reaches back over ALL open invoices, so one user's organic traffic
settles another user's now-paid-but-unwatched order (the "closed the tab then paid" case is
covered as long as ANY traffic flows). So a background timer is NOT required for correctness
under normal traffic — only a fully idle system needs it.

RECOMMENDATION (softened from "auto-sweep on by adapter default"):
  - Do NOT hide a setInterval inside every adapter by default. It is wrong on serverless/Next
    (no long-lived process) and adds DB load for a case organic traffic already handles.
  - Add an explicit opt-in helper, e.g.  openreceive.startSweeper({ intervalMs })  returning a
    stop handle the host can clear on shutdown.
  - Keep it OUT of the quickstart happy path; document in advanced/ops that fully idle
    deployments (no organic traffic) should run it.


================================================================================
EXPLICITLY NOT DOING (yet): no-orderId / prepare / prepareCheckout checkout mode
================================================================================
Both reviews and the maintainer constraint converge here. A <Checkout /> that works without an
orderId (a `prepare` hook that mints an order as a side effect of checkout creation) is
tempting but it pulls order creation back INSIDE OpenReceive — violating rail #1 — and opens an
idempotency hole (remount / double-submit / StrictMode double-render creates duplicate orders).
The clean boundary "host owns carts/orders; OpenReceive owns payment" is the project's biggest
strength. Keep <Checkout orderId={...} /> as the single primary mental model. Revisit a
prepareCheckout escape hatch only AFTER the order-owned flow feels excellent, and only if real
users ask.

SEQUENCED PLAN
================================================================================
P0  Fix amount-authority contract + rename to resolveOrder (one atomic change):
      - require the hook in shipped handlers (throw if missing); delete client-amount fallback
        (JS handler.ts:190-192; Ruby apply_client_amount).
      - create-route body = { order_id, memo?, description_hash?, metadata? }; reject client amount.
      - rename getOrderAmount -> resolveOrder / resolve_order; null->404, throw->400.
      - update OpenAPI (drop amount required+property), golden/test vectors, routes.md amount section.
      - preserve client-priced use case via host honoring clientAmount inside resolveOrder.

P1  Umbrella adapter exports: add openreceive/express|fastify|next (createOpenReceive + adapter),
    with adapters as optional/peer deps so client bundles stay clean.

P2  Rewrite quickstart-node.md: one story — create order (your app) -> resolve price -> render
    <Checkout orderId> -> fulfill idempotently. Cut the token section and the direct-methods
    coda; move them to security.md / routes.md. Add a minimal runnable example that IS the
    quickstart pasted verbatim.

P3  Add openreceive.startSweeper({ intervalMs }) opt-in helper; document idle-system caveat.
    No default adapter interval.
