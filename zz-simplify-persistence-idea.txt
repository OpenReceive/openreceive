Yes—exactly. I overstated it with “bring your existing database.”

The cleaner proposition is:

OpenReceive has no storage configuration whatsoever. No database URL, Redis URL,
migrations, tables, adapters, or internal persistence.


CORE IDEA
=========

The application already owns its orders. OpenReceive communicates with the wallet
and swap providers, but it does not own payment records or workflow persistence.

The host application stores the payment hash and the time it was paid on its
existing order, using its own ORM and database conventions. OpenReceive never
receives a database connection and never knows how the order is represented.

The universal application-facing record is simply:

    order.payment_hash
    order.paid_at

paid_at is null before payment. Once OpenReceive verifies settlement, the host sets
paid_at once and never changes it again. The integrating developer does not need to
store or understand BOLT11 invoices, millisatoshis, invoice creation times, invoice
expiry times, wallet state, or settlement workflow state.

For a swap, the host may additionally store one opaque swap recovery token:

    order.openreceive_swap_recovery_token

The payment hash lets the host recognize the order when OpenReceive reports it paid.
The recovery token lets OpenReceive query an unresolved swap provider workflow.

This makes the boundary simple:

    OpenReceive manages communication with wallets and swap providers.
    The host application manages orders and payment records.


ORDINARY PAYMENT FLOW
=====================

Conceptually, checkout creation becomes:

    const checkout = await openreceive.createCheckout({
      orderId
    });

    // Application code using its existing database/ORM:
    await orders.setPaymentHash(orderId, checkout.paymentHash);

    // Return the checkout only after saving its payment hash.
    return checkout;

The sequence matters. The host must save the payment hash before exposing the checkout
to the payer. If the host write fails, the checkout is abandoned and must not be
displayed. That prevents a payer from completing a payment the application cannot
correlate with an order.

The settlement flow becomes:

    onPaid: async payment => {
      await orders.setPaidAtOnce(payment.paymentHash, payment.paidAt);

      // Optional: payment.details contains all additional safe information
      // OpenReceive has about the settled payment.
    }

That is the entire application responsibility. OpenReceive has already verified the
payment. The host performs one idempotent terminal transition:

    unpaid -> paid

Once paid, the order never becomes unpaid again.

OpenReceive does not look up or modify the order. It reports a verified payment by
payment_hash, and the host performs the update using its own data model.

The lifecycle is:

    Host creates/prepares order
              |
              v
    OpenReceive creates checkout
              |
              v
    Host stores payment_hash; paid_at is null
              |
              v
    Checkout is returned to payer
              |
              v
    OpenReceive discovers verified settlement
              |
              v
    onPaid({ paymentHash, paidAt, details? })
              |
              v
    Host sets paid_at once; it never changes again


WHAT THE HOST SHOULD STORE
==========================

The host-facing minimum is deliberately two fields:

    payment_hash
    paid_at

For example:

    orders
    ------
    id
    price
    payment_hash
    paid_at

The host already knows its own order price. It should not copy wallet amounts,
invoices, or invoice timing fields merely to satisfy OpenReceive. Those are internal
payment mechanics and belong inside OpenReceive's checkout response and wallet
verification logic.

When onPaid supplies paymentHash and paidAt, the host finds the matching order and
sets paid_at only if it is still null. The update must be idempotent so repeated
delivery is harmless. A non-null paid_at is the entire durable paid state.

Everything else is optional. Applications may retain more audit or checkout data if
they want it, but OpenReceive must not require it.


SETTLEMENT EVENTS
=================

The normal settlement callback should be intentionally small:

    {
      paymentHash,
      paidAt,
      details?
    }

paymentHash and paidAt are the only required fields. details is an optional object
containing all additional safe information OpenReceive has about the settled
payment. It may include normalized wallet transaction data, amounts, invoice and
checkout context, timestamps, metadata, fiat quote context, swap context, and how
settlement was detected. Fields unavailable from a particular wallet or provider
are omitted.

Applications do not need to read or store details to fulfill an order. Applications
that want an audit record, analytics, receipts, or advanced workflows can use or
persist it. The object must never include the receive-only NWC connection string,
wallet secrets, provider credentials, raw capability secrets, or other private
configuration.

OpenReceive is responsible for checking the expected payment and proving settlement
before calling onPaid.

Internally, the stable wallet identity remains payment_hash and authoritative
settlement still requires settled_at or state == "settled". A preimage remains
corroborating information, not settlement authority. These are OpenReceive
implementation rules, not concepts imposed on the host application.

The onPaid contract must be explicitly at-least-once:

  - OpenReceive may deliver the same settlement repeatedly.
  - Events may be observed out of order.
  - A payment may be reported after the host considered an invoice expired.
  - A failed callback will be attempted again when reconciliation rediscovers it.
  - The host must make marking paid and fulfillment idempotent.

The host's order-level rule is simply that paid_at is write-once. Duplicate callbacks
do not change it, ship twice, send two receipts, or make the order unpaid again.


NIP-47 RECONCILIATION
=====================

NIP-47 provides three relevant mechanisms:

  1. payment_received notifications
  2. lookup_invoice by payment_hash or BOLT11 invoice
  3. list_transactions over a creation-time range

Notifications are passive wake-up hints. They are optional and relay delivery is
not a durable settlement mechanism. OpenReceive must confirm settlement through a
wallet status lookup or transaction scan and require settled_at or state ==
"settled".

lookup_invoice is useful for a known unpaid payment_hash, but it is an optional NWC
capability and must be detected from the wallet's advertised methods.

list_transactions is the broad recovery path. It should:

  - request incoming transactions;
  - use overlapping inclusive time windows;
  - deduplicate by payment_hash;
  - page with at most 20 transactions, following NIP-47 guidance;
  - permanently cache only final settled facts if the host wants caching;
  - treat pending results as mutable and short-lived.

An important NIP-47 detail is that list_transactions ranges and ordering use
transaction creation time, not settlement time. A cursor based only on the most
recent settlement can miss an older invoice that settles later.

Recovery can therefore happen in either direction:

  - OpenReceive lists wallet transactions and emits each verified settlement by
    payment_hash. The host ignores hashes that do not belong to it.

  - The host takes unresolved payment_hash values from rows where paid_at is null and
    asks OpenReceive to reconcile them.

OpenReceive does not need its own record of which orders remain unpaid. The host
already has that information.


SMALL PUBLIC PRIMITIVES
=======================

The normal storage-free API should remain order/payment oriented:

    createCheckout({ orderId })
    checkPayment({ paymentHash })
    watchPayments({ onPaid })

Invoice lookup, transaction ranges, BOLT11 handling, millisatoshi normalization,
expiry, reuse, and wallet capability detection are internal OpenReceive mechanics.
Advanced low-level APIs may expose them when genuinely needed, but the normal host
integration should not require them.


SWAPS: TWO INDEPENDENT RECOVERY PLANES
======================================

Swaps introduce a second external ledger. There are two different questions:

    Did the merchant's Lightning wallet receive the payment?
    What is happening inside the swap provider workflow?

They have different recovery identifiers:

    Wallet settlement: payment_hash
    FixedFloat workflow: FixedFloat order ID + FixedFloat order token

The payment hash is enough to prove successful final settlement. Once FixedFloat
pays the shadow Lightning invoice, OpenReceive can discover that payment through
NWC and call onPaid. The host can fulfill the order without consulting FixedFloat.

The payment hash is not enough to inspect an unresolved FixedFloat swap. FixedFloat
documents its /order request as requiring both:

    {
      "id": "TESTID",
      "token": "TESTTOKEN..."
    }

The FixedFloat API does not document a list/search endpoint by payout BOLT11,
payment hash, or client reference. Although an order response contains the shadow
BOLT11 as to.address, the caller cannot retrieve that order without already knowing
the order ID and token.

Therefore a database lost down to payment_hash can reconstruct wallet settlement,
but it cannot reconstruct an unresolved FixedFloat order, refund, underpayment, or
emergency workflow.


OPAQUE SWAP RECOVERY TOKEN
==========================

OpenReceive can avoid internal persistence by turning the provider credentials into
one opaque recovery value for the host to store. The application-facing result is:

    {
      paymentHash: "...",
      swapRecoveryToken: "or_swap_v1_..."
    }

The recovery token is an authenticated, server-encrypted envelope. Internally it
contains approximately:

    {
      version: 1,
      provider: "fixedfloat",
      providerOrderId: "TESTID",
      providerToken: "TESTTOKEN...",
      paymentHash: "...",
      orderId: "...",
      payInAsset: "USDT_TRON",
      expiresAt: 1234567890
    }

Only these values are required to poll FixedFloat:

    provider identifier
    provider order ID
    provider order token

The payment hash and order ID are included in the sealed envelope to bind the token
to the intended payment/order and prevent substitution. They are not required by
FixedFloat's /order endpoint itself.

The FixedFloat order token is sensitive and must never be exposed in plaintext to
browser code, logs, screenshots, source maps, or public payloads. If an opaque
recovery token is browser-carried, it must use authenticated encryption and be
treated as a capability. Hosts may instead keep it entirely server-side.


MINIMAL HOST RECORD WITH SWAPS
==============================

The host-facing order data remains very small:

    payment_hash
    paid_at
    openreceive_swap_recovery_token   null when no swap is active

The host already has its order ID, price, and any business status. It does not add wallet amounts,
BOLT11 invoices, provider IDs, or provider tokens to its schema. For swaps it stores
one extra opaque string rather than provider-specific columns.

The two values cannot safely be collapsed with FixedFloat today:

  - NWC settlement is identified by payment_hash, which maps the verified payment
    back to the host order.

  - FixedFloat polling requires the independently generated order ID and token, so
    their encrypted representation must be retained somewhere.

No cryptographic transformation can recover independently generated FixedFloat
credentials from the NWC payment hash. The FixedFloat token is created after the
shadow invoice, and FixedFloat does not accept a documented client-supplied
idempotency/reference value that could bridge the two systems.


STATELESS SWAP POLLING
======================

Polling becomes:

    const swap = await openreceive.getSwap({
      recoveryToken: order.swapRecoveryToken
    });

OpenReceive decrypts the recovery token, selects the named provider, extracts the
provider order ID/token, calls FixedFloat /order, and returns the current normalized
state.

OpenReceive does not need to persist:

    provider status
    deposit transaction
    payout transaction
    refund state
    deposit amount
    deposit address
    provider expiry
    emergency information

FixedFloat returns those fields again from /order. The host may cache terminal
results or retain an audit snapshot, but caching is not required for polling.

The deposit address and exact deposit amount need only be stored if the host wants
to render payment instructions without polling or retain a complete audit trail.


SWAP CREATION ORDERING
======================

A safe storage-free swap creation flow is:

  1. OpenReceive creates the shadow Lightning invoice.
  2. OpenReceive creates the FixedFloat order using that BOLT11 as toAddress.
  3. FixedFloat returns its order ID, token, deposit address, amount, and expiry.
  4. OpenReceive seals the provider ID/token into swapRecoveryToken.
  5. The host persists payment_hash and swapRecoveryToken.
  6. Only after persistence succeeds are the deposit address and amount shown to
     the payer.

If persistence fails, the swap instructions must not be shown. The unused provider
order can expire without receiving customer funds.

If FixedFloat /create times out before returning a deposit address, a provider order
may exist, but its deposit address was never shown to the payer. No payer can fund
that orphan order through OpenReceive. A retry may therefore create another unused
provider order, but it does not risk customer funds. A conservative delay can avoid
excessive retries.

This is simpler than attempting to persist an internal "creation needs reconcile"
workflow when FixedFloat offers no API for recovering the lost order ID/token.


SWAP COMPLETION
===============

Successful swap completion does not depend on provider polling:

  1. The payer sends the requested asset to FixedFloat.
  2. FixedFloat exchanges it and pays the shadow Lightning invoice.
  3. NWC exposes the settled incoming Lightning transaction.
  4. OpenReceive verifies settled_at/state == "settled".
  5. OpenReceive calls onPaid with payment_hash.
  6. The host finds and fulfills its order.

FixedFloat reporting DONE is not settlement authority. Provider completion remains
"finalizing" until the wallet confirms the shadow Lightning invoice settled.


REFUNDS AND EMERGENCIES
=======================

Underpayments, late deposits, overpayments, and refund choices require the swap
recovery token because they are provider workflows rather than wallet settlement.

A refund operation could be:

    openreceive.refundSwap({
      recoveryToken,
      refundAddress,
      confirmationToken
    })

OpenReceive decrypts the recovery token and sends FixedFloat the order ID, order
token, refund choice, and address.

A short-lived confirmation token can itself be cryptographically signed/sealed to
avoid server-side nonce storage. It can bind:

    payment hash
    provider order identity
    refund address
    expiration timestamp

Without persistence, strict single-use enforcement is not possible locally.
Therefore repeated refund submissions must be handled safely against the provider's
current order state, or the host must persist a one-time refund/fulfillment guard in
its own order record.


LOSS AND RECOVERY
=================

If only the swap recovery token is lost:

  - NWC can still prove whether the shadow invoice settled.
  - A settled order can still be fulfilled by payment_hash.
  - OpenReceive cannot inspect or manage an unresolved FixedFloat order.
  - An operator must use the FixedFloat dashboard/support path.

If only cached provider state is lost:

  - Nothing essential is lost.
  - OpenReceive polls FixedFloat again using the recovery token.

If OpenReceive restarts:

  - It has no state to reload.
  - The host supplies stored payment hashes for reconciliation.
  - The host supplies stored recovery tokens for swap polling.

If the host database is being backfilled:

  - NWC lookup/list can reconstruct settled invoice facts by payment_hash.
  - FixedFloat unresolved state can be reconstructed only if the recovery token (or
    raw provider ID and token) survived.




WHAT OPENRECEIVE NO LONGER OWNS
===============================

Under this proposal OpenReceive does not own:

    invoice tables
    checkout tables
    swap tables
    database migrations
    database or Redis connections
    storage adapters
    idempotency rows
    settlement-action leases
    provider-state history
    transaction-scan cursors
    order-to-payment mappings
    fulfillment state

The host owns its orders, payment attempts, invoice reuse decisions, fulfillment
guards, and any audit history it wants to retain.

OpenReceive remains responsible for:

    receive-only NWC communication
    exact amount normalization
    invoice creation
    settlement verification
    NIP-47 notification handling
    lookup/range reconciliation primitives
    swap-provider API communication
    provider response normalization
    sealing and opening swap recovery tokens
    enforcing that provider completion is not wallet settlement
    emitting at-least-once onPaid events


POSITIONING
===========

The resulting developer proposition is:

    Configure a receive-only NWC connection.
    Do not configure storage for OpenReceive.
    Store payment_hash and nullable paid_at on your order.
    If you enable swaps, also store one opaque swap_recovery_token.
    When onPaid reports paymentHash and paidAt, set paid_at once.
    A non-null paid_at is terminal and never changes.

Or, more concisely:

    OpenReceive manages wallet and swap-provider communication, not payment records.

This preserves reliable payment correlation and recoverable swap polling without
asking users for an OpenReceive database, Redis instance, migration step, schema,
or persistence adapter.
