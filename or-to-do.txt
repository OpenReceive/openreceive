Rules

Outstanding
-----------

1. Add package-level README/install files for the primary JS package artifacts.

   Current gap:

     tools/package/build-artifacts.mjs copies package README.md files into
     generated package artifacts when they exist, but packages/js/* currently
     has no package-level README.md files.

   Scope:

     - packages/js/node/README.md
     - packages/js/express/README.md
     - packages/js/browser/README.md
     - packages/js/elements/README.md
     - packages/js/react/README.md

   Constraints:

     - Keep each README package-specific and concise.
     - Link to docs/01-quickstart-node.md and docs/05-frontend-checkout.md
       instead of repeating full quickstarts.
     - Preserve the NWC secret boundary: no browser/mobile secrets, no live
       checkout in pure frontend code.
     - Keep package manifests private until explicit publishing approval.
     - Update package artifact or release checks if README inclusion should be
       mandatory for these packages.

2. Reconcile workflow skeleton paths before enabling public release workflows.

   Current gap:

     docs/12-release-process.md names .github/workflows/*.yml, while the
     checked safe workflow skeletons and validator currently use
     .github/workflows-disabled/*.yml.

   Scope:

     - Decide whether v0.1 should keep disabled skeletons or move safe,
       read-only workflows into .github/workflows/.
     - Keep docs/12-release-process.md, tools/validate/check-workflows.mjs,
       and the actual workflow directory aligned.
     - Keep publish disabled until explicit maintainer approval.
     - Do not introduce secrets, deployment credentials, npm publish,
       docker push, or gh release creation into public workflows.

3. Preserve the specific deferred ecosystem package shortlist.

   Current gap:

     README.md and docs/ currently say that new SDK ecosystems and native
     mobile UI kits are deferred, but they do not preserve the old master
     plan's specific package targets for later ecosystem work.

   Scope:

     - Go: audit or create nwc-client-go, then add Go core/adapters.
     - Rust: wrap rust-nostr for receive checkout, then add Rust core plus
       Axum/Actix-style adapters.
     - .NET: create or adopt Nwc.Client, then add .NET core plus ASP.NET Core
       integration.
     - Kotlin/JVM: create nwc-client-jvm, then add Ktor/Spring server
       integration.
     - Elixir: create nwc_client/open_receive packages and a Phoenix adapter.
     - Swift/Apple: audit NostrKit or create NWCClient, add Swift server/client
       pieces, SwiftUI checkout UI, and a Vapor adapter.
     - iOS demo: prefer a SwiftUI app backed by the existing Node/Express API
       first; keep Swift/Vapor API proof as later work.
     - Android: add Android client/Compose checkout UI after backend
       conformance is stable.
     - Flutter/Dart: add checkout widgets/client helpers only; do not use Dart
       or Flutter as a consumer-app wallet-secret runtime.
     - React Native: add a mobile checkout package only after the display-safe
       payload contract is stable.

   Constraints:

     - Do not start this before the current JS/Express release surface is easy
       to install, diagnose, package, and document.
     - Every backend SDK needs shared-vector conformance and live-wallet smoke
       behavior that skips clearly without OPENRECEIVE_NWC.
     - Mobile packages are display clients only. They must never store or
       receive NWC secrets.
     - Do not add send-payment features to OpenReceive receive-checkout APIs.
