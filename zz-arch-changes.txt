Below is the standalone spec in plain text (self-contained, since the agent won't see this thread), followed by the diffs worth handing over. The diffs are split into apply-as-is (self-contained, will patch cleanly) and reference (depends on the surrounding refactor, so treat as a pattern, not a literal patch).
One design decision baked into all of it, stated up front so the agent applies it uniformly: snake_case is the single convention for the entire developer-facing surface — HTTP requests, HTTP responses, the objects the SDK returns, and callback payloads. Internal domain objects inside service.ts stay camelCase and are mapped to snake_case at the public boundary. This is what makes res.json(checkout) correct with zero hand-mapping and lets an app pass req.body straight into createCheckout without renaming fields.

DIRECTIONS — OpenReceive v0.1 DX redesign
Repo: github.com/OpenReceive/openreceive

Status: pre-release, no external users. Breaking changes are authorized and expected. Update spec, JS packages, demos, and docs together so npm run test:ci stays green.
GLOBAL RULE

Use snake_case for the entire developer-facing surface: HTTP request bodies, HTTP response bodies, the JS objects the SDK returns from its methods, and the arguments passed to callbacks such as onPaid. Internal domain objects within service.ts may remain camelCase, but they must be converted to snake_case at the public method boundary. There must be exactly one casing a consumer ever sees.
DO NOT

Do not redesign the core model. Order contains Checkouts; a Checkout contains Invoices. All three are load-bearing and stay. The work is on the surface, not the model.
Do not add SDK-generated refresh URLs (e.g. checkout.links.refresh) or make the frontend component poll a URL OpenReceive invented. The app owns its routes. The app supplies a statusUrl; the component polls that.
Do not search for or "fix" createInvoice or refreshInvoiceStatus. They do not exist in the repo. The current verbs are createCheckout, getOrder, getCheckout.
Do not add application-domain statuses (manual_review, replaced, etc.) to OpenReceive types. Settlement statuses only.
Do not introduce a new "Session" noun or a new top-level method to solve display-checkout selection. Add a field to the existing getOrder result.

WORK ITEM 1 — Unify wire casing to snake_case and delete the dual-casing normalizer (highest priority)

Today the Node SDK returns camelCase, the OpenAPI response schemas are snake_case, the OpenAPI request schemas are camelCase, and the browser reconciles the two with a record.foo_bar ?? record.fooBar fallback on every field. Collapse this to one convention.

Strategy (low risk): keep service.ts internal builders (serializeInvoice, buildCheckout, buildOrder, groupCheckouts) camelCase exactly as they are. Add three boundary mappers — toWireInvoice, toWireCheckout, toWireOrder — that convert the internal camelCase object to the snake_case public object. Wrap the return value of every public method (createCheckout, getOrder, getCheckout) in the matching mapper. This guarantees no internal camelCase accessor is missed, because internals are untouched.
Change the exported types (OpenReceiveInvoice, OpenReceiveCheckout, OpenReceiveOrder, OpenReceiveCreateCheckoutRequest) to snake_case to match what the mappers emit and what the wire spec already says.
In spec/openapi/openreceive-http.v1.yaml, change CreateCheckoutRequest and CreateCheckoutAmount to snake_case (order_id, description_hash, expires_in_seconds). Responses are already snake_case; leave them.
In packages/js/browser/src/internal.ts, delete the camelCase fallbacks in normalizeCheckoutSnapshot and normalizeCheckoutInvoiceSnapshot (every ?? record.<camel> and ?? record.bolt11). After the SDK reliably emits snake_case, those branches are dead. Also make CheckoutState uniformly snake_case (lightningUri to lightning_uri, expiresInSeconds to expires_in_seconds) and update the readers inside the browser package.
Update examples/hello-fruit so clients read snake_case. This also fixes a latent bug where App.tsx reads invoice.order_id off a camelCase object and gets undefined.
Acceptance: grep for ?? record. in browser/internal.ts returns nothing. One casing end-to-end.

WORK ITEM 2 — Stop overloading the word "Invoice"

"Invoice" currently names two different things: the BOLT11-level object (OpenReceiveInvoice) and the checkout snapshot (packages/js/browser/src/index.ts re-exports CheckoutSnapshot as Invoice, and the demo types the wire checkout as Invoice).

Remove the CheckoutSnapshot as Invoice alias. Export the checkout type as Checkout. One word per concept: a Checkout is the payable thing everywhere; an Invoice is the BOLT11 detail inside a checkout, everywhere.
Update the demo's CreateOrderResponse type and imports (App.tsx).

WORK ITEM 3 — Centralize display-checkout selection into getOrder

Remove order.paidCheckout ?? order.activeCheckout from every caller and from the docs. Compute it once, server-side.

In service.ts buildOrder, add display_checkout = paid_checkout ?? active_checkout (computed from the internal paidCheckout ?? activeCheckout before mapping). Keep paid_checkout and active_checkout for advanced cases.
Add display_checkout to the Order schema in the OpenAPI spec and to the OpenReceiveOrder type.
The quickstart status helper becomes: get the order, return order.display_checkout. No ?? in app code.

WORK ITEM 4 — Make a real Checkout component the happy path for all four frameworks and fix the callback contract

The component prop must be checkout, not invoice. The simple path is an app-supplied statusUrl. The callback is the escape hatch and must return a full checkout or null, never an empty object.

React (packages/js/react/src/index.ts): rename the invoice prop to checkout. Document statusUrl as the default. Change the refreshStatus signature from (state) returning Partial<Snapshot> to (orderId: string) returning Promise<Checkout | null>.
Vue, Svelte, Angular already ship real components (Checkout.svelte exists; Vue and Angular accept an options object with statusUrl). Promote the component-plus-statusUrl form to the headline in each package README and in the quickstart. Demote the createOpenReceive<Framework>CheckoutBinding factories to a documented advanced escape hatch.
Shorten the factory names you keep: inside @openreceive/svelte the symbol can be createCheckoutBinding rather than createOpenReceiveSvelteCheckoutBinding; the package already namespaces it.
Remove every ?? {} from docs and code. The callback returns null to mean no change.
Target happy path, identical across frameworks: a Checkout component that takes checkout, statusUrl, onSettled, onStartOver.

WORK ITEM 5 — Add a metadata passthrough to createCheckout

Add optional metadata (a string-keyed record) to the createCheckout request type and the OpenAPI CreateCheckoutRequest.
In service.ts, merge it into the stored invoice metadata in the checkoutMetadata builder, without overwriting reserved keys (order_id, checkout_id, superseded, amount_spec, memo, description_hash, expires_in_seconds).
Round-trip it back out in onPaid.

WORK ITEM 6 — Enrich the onPaid payload

You fulfill a checkout snapshot, so the callback needs the checkout identity. The hook already carries the full invoice row internally; surface a clean snake_case object: order_id, checkout_id, invoice_id, payment_hash, amount_msats, metadata (plus the existing invoice, source, transaction). Document that onPaid may fire more than once, so fulfillment must be idempotent on checkout_id.
WORK ITEM 7 — Clarify the idempotent-supersede semantics of createCheckout (optional)

createCheckout already behaves correctly: for the same order_id it returns the paid checkout if one exists, reuses or renews an open checkout when the amount matches, and supersedes plus creates a new checkout when the amount changes. This is the mechanism behind the cart-edit flow. Do not change the behavior. Optionally rename to checkoutForOrder to signal it, and document the supersession. If renamed, keep the verb consistent everywhere.
WORK ITEM 8 — Rewrite docs/guides/quickstart-node.md

Lead with the shortest path to a rendered, payable checkout: set env, createOpenReceive with onPaid, one createCheckout, an app status endpoint that returns getOrder, and a Checkout component fed checkout and statusUrl.
Make the React, Vue, Svelte, and Angular sections show the same component-plus-statusUrl shape. The only difference should be import path and template syntax.
Move the boundary philosophy, the "onPaid runs more than once" note, and the "fulfill from the paid checkout, not the live cart" subtlety into a short "How settlement works" section below the happy path.
Add a "Cart changes" section: re-call createCheckout with the new amount; it supersedes the old checkout; old checkouts stay settlement-watchable; if a superseded checkout is later paid, fulfill from order.paid_checkout.

ACCEPTANCE CRITERIA

One casing convention (snake_case) across spec, SDK return values, browser, demos, and callbacks. No ?? record.<camel> fallbacks remain in browser/internal.ts.
No symbol named Invoice refers to a checkout. grep for "as Invoice" returns nothing.
order.display_checkout exists; paidCheckout ?? activeCheckout appears nowhere in docs or examples.
All four framework quickstart snippets use a Checkout component with checkout and statusUrl; binding factories appear only under an advanced heading. No ?? {} anywhere.
createCheckout accepts metadata; onPaid receives order_id, checkout_id, invoice_id, payment_hash, amount_msats, metadata.
The Hello Fruit demos run end-to-end and npm run test:ci passes.

function toWireCheckout(m: CheckoutModel): OpenReceiveCheckout {
  return {
    checkout_id: m.checkoutId,
    order_id: m.orderId,
    status: m.status,

  };
}

function toWireOrder(m: OrderModel): OpenReceiveOrder {
