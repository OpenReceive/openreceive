1. Decide and fix the backend publish story — this is the big one. Your release tool (tools/release/npm-release.mjs) has a hardcoded PUBLIC_PACKAGE_NAMES list of seven frontend packages. @openreceive/node and @openreceive/core are marked private: true and are excluded. But OpenReceive's entire model is "the server creates and verifies the invoice; the browser never sees the NWC code." So a developer who runs npm install @openreceive/react gets checkout UI that talks to a backend they have no supported, published way to build — the half that actually does the work (createOpenReceive, the NWC service, the stores, the CLI) only exists as unpublished source in your repo. At the "publish to npm" milestone this is the thing most likely to make adoption stall. Either promote node + core to public and add them to the release list, or, if frontend-only is the deliberate v0.1 scope, say so loudly in the README and quickstart and give people a clear story for the server side. Right now it's neither.
... make them all public

2. Replace the two bespoke build pipelines with one standard bundler. You currently maintain two hand-rolled TypeScript-compiler-API builders: packages/js/provider-data/build.mjs and tools/package/build-artifacts.mjs. Between them that's ~600 lines doing JSON-import inlining, CommonJS rewriting, import.meta.url shimming, .ts→.js extension rewriting in declarations, and bin-shebang regex surgery. That's all stuff tsup, unbuild, or tsc project references do for free and far more predictably. The deeper problem is that locally everyone runs from .ts source (via tsx + allowImportingTsExtensions), while the published dist is reconstructed in a temp dir and deleted (build:packages is --dry-run only). So the artifact your users consume is never visible during development, and bugs in the transform layer only surface after publish. Moving each package to a normal npm run build that emits a committed-shape dist means local === published, deletes both custom builders, and makes the whole thing legible to new contributors.

6. Clean up the public API surface. Two things stood out in packages/js/node/src/service.ts. First, the exported types mix casing within a single object: OpenReceiveCreateInvoiceRequest has orderUuid (camel) sitting next to amount_sats, amount_msats, and description_hash (snake), and OpenReceiveInvoice is entirely snake_case (invoice_id, payment_hash, order_uuid). A consumer has to memorize which field is which convention. Pick camelCase for the JS surface and map to the wire contract internally, or commit to snake_case everywhere — just not both in the same type. Second, the create-invoice request accepts amount, amount_sats, amount_msats, and fiat as overlapping optionals with no type-level guarantee that exactly one is set; a discriminated union would catch that at compile time instead of runtime. Minor related nit: @openreceive/react's peer range is "react": "*" — tighten it (you're on React 19 types, so >=18 at least).

3. Rename public JS API fields before anyone depends on them

Because you have no backward-compatibility burden, I would fix naming now. The current invoice request mixes camelCase and snake_case: orderUuid, amount_sats, amount_msats, optionalInvoiceDescription, description_hash, and expiry. That is awkward in TypeScript.

I would make the public JS API idiomatic camelCase and keep wire/schema naming separate:

await openreceive.invoices.create({
  orderId: order.id,
  amount: { fiat: { currency: "USD", value: "24.99" } },
  memo: "Order #1234",
  expiresInSeconds: 900,
});

Return this:

{
  invoiceId: string;
  orderId: string;
  bolt11: string;
  paymentHash: string;
  amountMsats: string;
  status: "pending" | "paid" | "expired" | "failed";
  expiresAt: Date;
}

Also, I would rename orderUuid. The docs say it is the app’s unique order key and should be used for idempotent fulfillment, but it does not need to be a literal UUID. Better names: orderId, checkoutId, or idempotencyKey.

My preferred shape:

{
  orderId: string;
  idempotencyKey?: string; // default: orderId
}

That makes the idempotency model explicit.

8. Use typed config errors with actionable hints

There is already an OpenReceiveServiceError with status, code, and body. I would extend that discipline to all setup/runtime failures:

class OpenReceiveConfigError extends Error {
  code:
    | "missing_nwc"
    | "invalid_nwc"
    | "wallet_preflight_failed"
    | "store_unavailable"
    | "unsafe_memory_store_in_production"
    | "missing_price_provider";

  docsUrl?: string;

OpenReceiveConfigError: missing_nwc

11. Rename browser-side createInvoice

The browser package exports createInvoice, copyInvoice, createLightningUri, checkout controller helpers, QR helpers, wallet-opening helpers, provider helpers, and status helpers. The name createInvoice on the browser side can be misleading because the browser must not hold NWC and cannot actually create a wallet invoice directly. The README explicitly says browser/mobile/static sites never receive NWC credentials and need a backend route.

I would rename browser helpers to make the boundary obvious:

requestCheckoutInvoice(...)
requestInvoiceFromServer(...)
createCheckoutController(...)

Example:

const checkout = createCheckoutController({
  createInvoice: () =>
    requestCheckoutInvoice("/api/openreceive/invoices", {
      orderId,
    }),
});

This nudges users toward the correct security model.

12. Unify framework component props

The docs show React, Vue, Svelte, and Angular usage, but the prop vocabulary differs across examples: invoice, snapshot, options, onSettled, etc. I would define one cross-framework contract:

type CheckoutViewProps = {
  invoice: OpenReceiveInvoice;
  status?: OpenReceiveInvoiceStatus;
  providers?: OpenReceiveProvider[];
  onCopy?: () => void;
  onOpenWallet?: (providerId: string) => void;
  onSettled?: (invoice: OpenReceiveInvoice) => void;
  theme?: "system" | "light" | "dark";
};

Then every wrapper mirrors it:

<OpenReceiveCheckout invoice={invoice} onSettled={...} />
<OpenReceiveCheckout :invoice="invoice" @settled="..." />
<OpenReceiveCheckout {invoice} on:settled={...} />

Framework wrappers should feel like thin skins over the same model.

18. Keep generated contracts behind an intentional subpath

The README says the spec is the source of truth, and @openreceive/core exports generated contracts. The core package index exports generated contracts alongside runtime utilities. That is useful for internal consistency, but it can leak too much implementation surface.

I would expose generated schemas intentionally:

import { schemas } from "openreceive/contracts";

But keep normal users on:

import { createOpenReceive } from "openreceive/node";

Generated models should not become the de facto public API unless you truly want users to depend on them.

21. Replace loose logging with stable event types

The current logging type is flexible: a log entry has level, event, optional message, optional error, and arbitrary extra keys. That is convenient internally, but external developers benefit from stable event names.

I would expose:

createOpenReceive({
  onEvent(event) {
    console.log(event.type, event);
  },
});

With a discriminated union:

type OpenReceiveEvent =
  | { type: "wallet.preflight.started" }
  | { type: "wallet.preflight.succeeded" }
  | { type: "invoice.create.started"; orderId: string }
  | { type: "invoice.create.replayed"; orderId: string; invoiceId: string }
  | { type: "invoice.lookup.started"; paymentHash: string }
  | { type: "invoice.settled"; orderId: string; invoiceId: string }
  | { type: "settlement.handler.failed"; orderId: string; attempt: number };

Keep logger as an adapter, but document onEvent as the supported debugging interface.
