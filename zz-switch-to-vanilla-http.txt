OpenReceive Re-Architecture Spec — Route-Shipping Packages (Node + full Ruby port)
PART 0 — Thesis and ground truth
OpenReceive today is a library of methods: createOpenReceive(config) returns a service (createCheckout/getOrCreateCheckout, getOrder, getCheckout, order, sweepPendingInvoices, swap ops, rate ops); the host app hand-writes HTTP routes that call those methods. The re-architecture ships the routes themselves (rodauth-style) that operate on the host's database, while authentication stays 100% the host's. We are doing a full Ruby port (two settlement engines), so this spec defines one language-neutral contract that both a Node implementation and a Ruby implementation must satisfy bit-for-bit.
Source-of-truth files the agent MUST read and treat as authoritative (do not re-derive or guess these):

spec/schemas/*.json — invoice, error, swap-order, rate-quote, payment-event, invoice-storage.
spec/test-vectors/*.json — settlement-detection, idempotency, fiat-to-msats, amount-boundaries, invoice-lifecycle, swap-*, transaction-scan-pagination. These are the cross-language conformance oracle.
spec/openapi/openreceive-http.v1.yaml and spec/asyncapi/openreceive-events.v1.yaml.
AGENTS.md — the non-negotiables. Every one is a hard invariant below.
packages/js/node/migrations/001_create_openreceive_invoices.postgres.sql — the current schema.
packages/ruby/openreceive/lib/openreceive.rb — the Ruby core primitives that ALREADY exist (money math, settlement detection, NWC parse/normalize, idempotency, in-memory KV store, error classes). The port reuses these; do not rewrite them.

PART 1 — Invariants both ports MUST preserve (conformance contract)

NWC secret is server-only. The receive-only NWC code never reaches browser/mobile/logs/tests/source maps/error bodies. Route handlers must never echo it. Redaction helper already exists in both languages (redact_nwc_uri in Ruby; equivalent in JS).
Receive-only. No route ever exposes a send-payment method. Only make-invoice + list-transactions (incoming) NIP-47 methods are used.
Settlement authority. A payment is settled only when settled_at is present OR transaction_state/state == "settled". A preimage alone is corroboration, not proof. Passive NWC notifications are hints; the authoritative signal is a backend status refresh (list_transactions scan). No background daemons — refresh is request-driven (sweepPendingInvoices / status reads).
Money math is exact integers/decimals. amount_msats is the only money field in public payloads (millisatoshis, integer, bounds [1000, 9007199254740991]). Fiat→sats uses decimal math with ceil rounding ((fiat * 1e8 / btc_fiat_price).ceil, then *1000 for msats) — see Money.quote_fiat_to_sats (Ruby) and the JS pricing module; both must match spec/test-vectors/fiat-to-msats.usd.json. Never use binary floats for fiat.
Idempotency. Create operations are keyed by scope = (namespace, operation, idempotency_key) with a canonical-JSON request_hash = sha256:<64hex>. Same scope + same hash ⇒ replay original; same scope + different hash ⇒ 409 CONFLICT. Canonical JSON = keys sorted, see Idempotency.canonical_json. Both languages must produce identical hashes for identical logical requests (add a cross-language vector).
Replay/duplicate-safe settlement. Applying settlement twice is a no-op; settled_at is write-once (||=).
Capability-token protection for reads (NEW, this spec). Order/checkout reads are guarded by an unguessable per-order token so anonymous payers can poll their own order without a login and cannot read anyone else's. Defined in PART 2.
Amount authority stays with the host (NEW, this spec). The shipped create-checkout route must NOT trust a client-supplied price. The host supplies a resolveAmount hook that returns the authoritative amount for an order; the route uses that, not the raw client amount. (Derived from the current demo, where /create_order computes the total server-side before calling OpenReceive.)

Any change to a schema requires updating its test vectors in the same change (AGENTS.md).
PART 2 — Security model: three tiers, the authorize hook, capability tokens
The Devise/rodauth "everything behind login" model is wrong here; a content site needs anonymous checkout. Routes fall into three tiers:

Tier 1 — anonymous-capable (create checkout, list/quote rates): open by default; protected by (a) rate-limit hook, (b) the host authorize hook, (c) the host resolveAmount hook so price can't be forged.
Tier 2 — capability-scoped (read order/checkout status, run swap quote/start on your own order): requires a valid per-order capability token OR an allow decision from authorize.
Tier 3 — privileged (sweep pending invoices, admin/refund operations): requires authorize to allow; fails closed — if no authorize policy is configured, Tier-3 routes return 403 and log a loud configuration error at mount time.

The authorize hook (single most important contract). Identical shape in both languages:
authorize(context) -> boolean            # true = allow, false = deny (deny -> 403 UNAUTHORIZED)
context = {
  action:   one of
            "checkout.create" | "order.read" | "checkout.read" |
            "swap.options" | "swap.quote" | "swap.start" | "swap.refund" |
            "invoice.sweep",
  request:  the raw framework request object   # host reads THEIR OWN session/cookie/JWT/header here
  resource: { order_id?, checkout_id?, amount_msats?, attempt_id? }   # present when known
  token:    the presented capability token, or null
}
Rules:

OpenReceive NEVER inspects the host's session. It only calls authorize and obeys the return value. Inversion of control: the host hooks into OpenReceive, not the reverse.
Default authorize if the host provides none: allow Tier 1, allow Tier 2 iff a valid capability token is presented, DENY all Tier 3. This makes the package usable with zero auth config on a content site, and safe by default.
Content-site host example: authorize = ({action, token, resource}) => action==='checkout.create' || (validToken(token, resource.order_id)).
User/Account host example: authorize = ({action, request, resource}) => { const u = currentUserFromMySession(request); if (action==='checkout.create') return !!u; return orderBelongsTo(u, resource.order_id); }.

Capability tokens. On checkout creation, mint a high-entropy token (≥128 bits, URL-safe). Return it to the caller in the create response as order_access_token. Store only a hash (sha256) in the DB. Reads must present the token (header Authorization: Bearer <token> or X-OpenReceive-Order-Token; the existing browser client already forwards custom headers, so no client-contract break). Verify by hashing and comparing to the stored hash for that order_id. Tokens are per-order (all checkouts under an order share the order's token). This is the mechanism that makes the package work identically for no-User and User apps.
Rails specifics that let you ride host auth without knowing it:

config.parent_controller (default "ActionController::Base"; hosts set "ApplicationController"). Engine controllers inherit from it, so they automatically get the host's CSRF protection, authenticate_user!, and current_user. The authorize proc can then call current_user because it's inherited.
CSRF: inherited from the parent controller; do NOT ship your own. For token/API callers the host skips CSRF as they already do.

Node specifics: the router mounts behind the host's existing CSRF and body-parser middleware. OpenReceive ships no CSRF and no auth — only the authorize/resolveAmount/rateLimit seams.
PART 3 — Canonical HTTP route contract (the thing being shipped)
One contract; every Node adapter and the Rails engine implement it identically. Wire style matches the existing browser packages (POST + JSON body, snake_case, an action-multiplexer on the order route). Mounted under a host-chosen prefix (default /openreceive).
POST  {prefix}/checkouts                      Tier 1   action=checkout.create
  req:  { order_id, amount|sats|usd, memo?, description_hash?, metadata? }
        (exactly one of amount{btc|fiat} / sats / usd; amount bounds enforced)
  flow: authorize(checkout.create) -> resolveAmount(order_id, client_amount) ->
        getOrCreateCheckout -> mint/att capability token (first time for order)
  res 201: { checkout: <CheckoutSnapshot>, order_access_token }   # token only on first mint
  errors: 400 INVALID_REQUEST, 403 UNAUTHORIZED, 409 CONFLICT (idempotency),
          429 RATE_LIMITED, 503 WALLET_UNAVAILABLE

POST  {prefix}/orders/{order_id}              Tier 2/3 action=order.read | swap_*
  req:  { order_id, action?="status" }
        | { order_id, action="swap_quote", pay_in_asset }
        | { order_id, action="start_swap", pay_in_asset }
        | { order_id, action="refund_swap", attempt_id, refund_address, refund_nonce, confirm? }
  flow: authorize(order.read | swap.quote | swap.start | swap.refund; token) ->
        service.order(body)
  res 200: status -> <OrderStatus>; swap_quote -> {quote}; start_swap|refund_swap -> {attempt}
  note: unknown action -> 400 (never silently treated as status)
  errors: 400, 403, 404 NOT_FOUND, 409, 429

GET   {prefix}/checkouts/{checkout_id}         Tier 2   action=checkout.read
  flow: authorize(checkout.read; token) -> service.getCheckout
  res 200: <CheckoutSnapshot>;  404 NOT_FOUND

GET   {prefix}/orders/{order_id}/swap-options  Tier 2   action=swap.options
  res 200: <SwapOptionsResponse> { enabled, options[] }

GET   {prefix}/rates                            Tier 1 (public)
  query: base?, currencies?   -> res 200: <RateQuote[]> (per spec/schemas/rate-quote)

POST  {prefix}/admin/sweep                      Tier 3   action=invoice.sweep
  flow: authorize(invoice.sweep) [FAILS CLOSED] -> service.sweepPendingInvoices
  res 200: { scanned, settled, expired, ... }
CheckoutSnapshot, OrderStatus, SwapOptionsResponse, SwapAttempt, and the error body all conform to the JSON Schemas in spec/schemas/. Error body shape: { code, message, retryable?, request_id?, details? } with code ∈ the enum in error.schema.json. HTTP status mapping (already implied by the Ruby error classes): INVALID_REQUEST→400, UNAUTHORIZED→403, NOT_FOUND→404, CONFLICT→409, RATE_LIMITED→429, WALLET_UNAVAILABLE→503, INTERNAL→500, INVOICE_EXPIRED→409/410, TIMEOUT→504.
The OpenAPI file currently defines only shared shapes ("app-owned controllers"). Change: promote it to define these full paths, so it becomes the executable contract both ports are tested against.
PART 4 — Shared service/orchestration contract (both languages implement)
The route layer is thin; it calls a service object. Both languages expose the same logical service (names may be idiomatic per language). Signatures (SDK-facing, camelCase in JS / keyword args in Ruby; HTTP bodies are snake_case):
createOpenReceive(options) -> service
  options: { nwc, storeUri, namespace, priceCurrencies?, swap?, logging?, operation? }

service.getOrCreateCheckout(req) -> CheckoutSnapshot     # idempotent create
service.getCheckout({ checkoutId }) -> CheckoutSnapshot
service.getOrder({ orderId }) -> OrderStatus
service.order(body) -> OrderStatus | {quote} | {attempt}  # action router, snake_case body
service.swapOptions({ orderId }) -> SwapOptionsResponse
service.swapQuote({ orderId, payInAsset }) -> SwapOption
service.startSwap({ orderId, payInAsset }) -> {attempt}
service.refundSwap({ attemptId, refundAddress, refundNonce, confirm? }) -> {attempt}
service.sweepPendingInvoices() -> ReconcileSummary
service.listRates(input) / service.quoteRates(input) -> RateQuote[]
service.close()
Orchestration responsibilities (identical semantics both languages): validate request → idempotency check → resolve fiat quote if needed (pricing) → build NIP-47 make_invoice request → call NWC client → persist invoice row via store → apply state transitions → return snapshot. order/status reads perform a bounded list_transactions scan to refresh settlement (see PART 6). Swaps follow the swap-* test vectors exactly (dual-payment, supersede-then-paid, emergency-refund, settlement-any-invoice, shadow-active-selection, start-idempotency).
PART 5 — Store interface (port surface) + migrations
The store is a KV-style interface. Node has it (postgres-store.ts, sqlite-store.ts); Ruby has the in-memory version (InMemoryInvoiceKvStore) and needs a durable one. Required methods (names from the JS Postgres store; Ruby in-memory store already mirrors most):
ensureSchema()
putIfAbsent(record) -> {status: created|replayed, row}     # enforces idempotency scope + uniqueness
put(record) / update transitions
get(invoiceId) -> row?
getByPaymentHash(paymentHash) -> row?
getByIdempotencyScope(scopeKey) -> row?
listByOrderId(orderId) -> row[]
listByCheckoutId(checkoutId) -> row[]
listOpen({ now, limit }) -> row[]        # for sweep/reconcile
getMeta(key) -> {value, rev}?            # transaction-scan cursor etc.
casMeta(key, value, expectedRev) -> {status: ok|conflict, row}   # optimistic concurrency
Uniqueness constraints (already in migration 001): invoice_id PK, payment_hash unique, invoice unique, and unique idempotency scope index on (namespace, operation, idempotency_key). Ruby's in-memory store already enforces all of these — port that logic to SQL/ActiveRecord.
Migration changes:

Keep openreceive_invoices and the namespaced openreceive_schema_migrations tracking table exactly as-is.
Add migration 002 introducing capability tokens. Preferred: add order_access_token_hash TEXT (nullable, set on first checkout per order) plus an index on (order_id, order_access_token_hash). Alternative: separate openreceive_order_tokens(order_id PK, token_hash, created_at). Choose the column approach unless multi-token-per-order is needed.
Author both dialects: 002_*.postgres.sql and 002_*.sqlite.sql. Structure migrations/ to allow migrations/mysql/ later without restructuring.

PART 6 — Settlement state machine (must be bit-identical across languages)
States (from schema + migration): transaction_state ∈ {pending, settled, expired, failed, accepted}; workflow_state ∈ {draft, invoice_created, verifying, settlement_action_pending, settlement_action_completed, expiry_pending_verification, expired_closed, failed_closed, cancelled}; settlement_action_state ∈ {pending, completed, failed}.
Transitions (Ruby InMemoryInvoiceKvStore and JS state/transitions.ts are the reference; they already agree — keep them agreeing):

mark_verifying: invoice_created|expiry_pending_verification → verifying iff not settled.
mark_settled(settled_at): set transaction_state=settled; workflow_state=settlement_action_pending unless already settlement_action_completed; settled_at ||= … (write-once).
mark_expired_closed: iff not settled → expired / expired_closed.
mark_failed_closed: iff not settled → failed / failed_closed.
mark_expiry_pending_verification: iff not in {settled, expired, failed} → expiry_pending_verification.
mark_settlement_action_completed / _failed: manage settlement_action_state + settlement_action_completed_at (write-once).

Settlement detection predicate (both languages, already implemented — Settlement.settled?/expired?/failed?): settled iff settled_at present OR state=="settled" OR transaction_state=="settled". Validate against spec/test-vectors/settlement-detection.json in BOTH languages.
Reconcile/scan: sweepPendingInvoices and status reads run a bounded list_transactions scan with a persisted cursor in meta (via casMeta), page limit and window padding from config (transaction_scan_*). Must satisfy spec/test-vectors/transaction-scan-pagination.json.
