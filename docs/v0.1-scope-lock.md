# v0.1 Release Boundary

v0.1 is no longer a blank-slate scope lock. It is the current release boundary
for the packages that already exist in this repository.

## Primary Supported Path

The first end-developer path to polish is:

- `@openreceive/node` for receive-only NWC, package-owned persistence, CLI
  setup, store initialization, doctor checks, and one-shot poll support.
- `@openreceive/express` for mounted HTTP routes, auth hooks, gated lookup,
  protected poll, backend settlement verification, and settlement actions.
- `@openreceive/browser`, `@openreceive/elements`, and `@openreceive/react`
  for display-safe checkout UI/state. These packages never receive NWC secrets.

This path must be easy to install, diagnose, package, and test before adding
new SDK families or demos.

## Supporting Packages

These packages are in the repo and should stay under test:

- `@openreceive/core`
- `@openreceive/provider-data`
- `@openreceive/testkit`
- `@openreceive/next`

They support the primary JS path, but the release promise is still the
Node/Express backend plus web checkout packages.

## Thin Or Initial Surfaces

- `@openreceive/vue`, `@openreceive/svelte`, and `@openreceive/angular` are thin
  typed bindings around the shared web component and browser-owned checkout
  model. Do not describe them as full native component systems yet.
- `openreceive` Ruby and `openreceive-rails` are initial proof work. Keep tests
  green, but do not publish or expand the Rails lane until Rails smoke and
  trusted live-wallet proof gaps are closed.

## Deferred

Do not add significant new work in these areas until the current release
surface is hardened:

- new SDK ecosystems
- new demos
- ecommerce plugins
- native mobile UI kits
- broad provider-data packages for other ecosystems
- edge/serverless live checkout promises
- send-payment features in OpenReceive receive-checkout APIs

## Contract Rules

- `spec/` remains the source of truth.
- `amount_msats` is the public millisatoshi field.
- Invoice storage requires `operation` because idempotency scope is
  `merchant_scope + operation + idempotency_key`.
- Settlement requires backend lookup with `settled_at`, `state == "settled"`,
  or `transaction_state == "settled"`. A preimage alone is not final proof.
- Browser, mobile, and static frontend code never receive NWC secrets.

## Gate

Before treating the boundary as synchronized, run:

```sh
npm test
npm run validate
npm run check:generated
npm run typecheck
npm run test:js
npm run test:package-smoke
npm run test:ruby
npm run check:release
npm run test:ci
```

`npm run test:live:nwc` and `npm run test:live:ruby:nwc` are optional trusted
live-wallet checks and must skip clearly when wallet secrets are absent.

## Parallelism

Shared schemas, vectors, generated contracts, storage semantics, settlement
rules, and checkout behavior remain lead-owned. Parallel work is acceptable
only on isolated polish lanes with separate files and no contract drift.
