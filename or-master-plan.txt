

Brand And Naming
----------------

Use the product name OpenReceive.


The product, packages, docs, org, domains, and public assets should use the
correct spelling: OpenReceive / openreceive.

Recommended public names:

  GitHub org:        openreceive
  Main repo:         openreceive/openreceive
  Apex/private app:  openreceive.org, private Rails/Kamal app, out of scope
  Docs:              docs.openreceive.org
  Hosted demos:      demos.openreceive.org
  Playground:        play.openreceive.org
  Registry CDN:      registry.openreceive.org
  Demo subdomains:   django-demo.openreceive.org, express-demo.openreceive.org
  Runtime model:     no required OpenReceive daemon; package runner inside app backend

If the misspelled domain was purchased, keep it only as a redirect.

License
-------

Use the MIT License for the OpenReceive repository and first-party packages
unless a specific package has a documented compatibility reason to use another
permissive license. Third-party NWC, crypto, framework, and UI dependencies
must be compatible with MIT distribution.

Existing openreceive.org app:

  openreceive.org is already a private Rails app deployed separately. Its
  production edge, host inventory, capacity notes, IPs, load balancer details,
  and deployment topology belong in private ops notes, not in the public master
  plan. That app lives in a private repo and must stay completely outside this
  open source project.

  Hard boundary:

    - Do not put open source demo code in the private Rails app.
    - Do not put demo deployment config in the private Rails repo.
    - Do not require the private Rails app to route, proxy, or host demos.
    - Do not attach demo containers to the private app's Kamal deployment.
    - Do not put private landing pages, SEO pages, product pages, admin code,
      analytics code, or app-specific business logic in the public repo.
    - Do not require the public repo to contain the private site's renderer,
      CMS, routes, or import job implementation.

  The open source project may use subdomains such as docs.openreceive.org,
  demos.openreceive.org, and django-demo.openreceive.org, but those should be
  deployed as separate services with separate DNS records, compose projects,
  environment files, and source control from the private apex Rails app. They
  must not share the private apex app infrastructure for stable public demos.
  Use a separate demo edge/node. The private app can link to demos manually if
  desired, but it is not part of their deployment path.

  Documentation is the one intentional bridge: canonical technical docs live
  in the public GitHub repo as markdown/data, and the private openreceive.org
  app may import or render those docs. The import job, SEO presentation,
  landing pages, and private site code stay in the private repo.


Product Definition
------------------

OpenReceive should make this promise:

  "Add Bitcoin Lightning receive payments to any app with a tiny backend
  integration and a polished frontend checkout."

The core product must support:

  - Creating a BOLT11 invoice.
  - Looking up invoice settlement.
  - Safely polling until settled or expired.
  - Listening to NWC notifications as an accelerator.
  - Converting fiat amounts to millisatoshis.
  - Locking a quote at invoice creation.
  - Showing the customer QR/copy/open-wallet UI.
  - Showing payer-side route suggestions that may be able to pay the invoice.
  - Pushing passive settlement updates to the frontend.
  - Providing an idempotent boundary where a merchant-supplied settlement
    action may run only after backend verification.

The reference flow:

1. Merchant creates or obtains a receive-only NWC connection string.
2. Backend stores that NWC string server-side only.
3. Backend SDK calls make_invoice through a native NWC client.
4. Backend stores invoice_id, payment_hash, invoice, amount_msats, expiry,
   quote, metadata, transaction_state, and workflow_state.
5. Frontend receives only non-secret invoice display data.
6. Frontend renders QR, copy button, lightning URI, countdown, payment routes,
   and transaction/workflow state.
7. Backend listens for notifications where practical and also polls
   lookup_invoice through a background job, queue worker, scheduled task, or
   long-running framework worker.
8. Backend confirms settlement by lookup_invoice result with settled_at or
   transaction_state/state == "settled". A preimage is corroborating proof, not
   a sufficient settlement signal by itself.
9. Backend emits invoice.settled to the matching browser session.
10. If configured, backend runs the app-owned settlement action exactly through
    OpenReceive's idempotent action boundary. OpenReceive records that the
    action completed or failed, but never records what the action did.

There is no pure-frontend OpenReceive checkout mode. A browser app, mobile app,
static site, or desktop shell can render the checkout UI, but invoice creation,
NWC access, settlement lookup, polling, notification handling, and merchant
settlement actions must always happen in a server-side component controlled by
the merchant.


Product Wording Boundary
------------------------

Marketing, docs, demos, provider-registry copy, and README text must be
conservative about what OpenReceive does.

Do not say or imply:

  - OpenReceive lets customers pay with fiat.
  - OpenReceive lets customers pay with crypto.
  - OpenReceive processes fiat, crypto, on-chain, card, bank, or exchange
    payments.
  - OpenReceive is a processor, exchange, broker, wallet, custodian, or money
    transmitter.

Use wording like this:

  "OpenReceive creates one Lightning invoice and can show payer-side route
  suggestions for wallets, exchanges, swap services, or fiat onramps that may
  be able to pay that invoice."

Provider routes are suggestions, not payment guarantees. The payer chooses and
uses a third-party wallet, exchange, onramp, or swap service outside
OpenReceive.


Non-Negotiable Rules
--------------------

Security and correctness:

  - NWC secrets never go in browser code.
  - NWC secrets never go in mobile apps shipped to users.
  - Pure frontend apps are not supported for live checkout.
  - Static sites must call a small API, CMS/plugin backend, merchant server, or
    explicitly proven serverless/durable backend before showing an invoice.
  - Receive-only NWC is the default and recommended mode.
  - OpenReceive receive-checkout APIs never send payments.
  - Standalone nwc-client-* protocol libraries may support pay_invoice for
    general NWC users, but OpenReceive checkout adapters hide or refuse send
    methods by default.
  - The frontend never unlocks a product by itself.
  - Notifications are never final proof by themselves.
  - Settlement truth comes from lookup_invoice; NWC notifications can provide
    the same data faster but cannot run merchant settlement actions by
    themselves.
  - Store payment_hash for every invoice.
  - Store preimage when available, but never treat preimage alone as final
    settlement proof for incoming invoices.
  - Use idempotency keys for invoice creation.
  - Use exact decimal/fixed-point money math, not floating point.
  - When local expiry is reached, perform a final lookup and only close the
    invoice as expired when wallet truth confirms expiry or the grace policy is
    exhausted.
  - Redact wallet secrets from logs, errors, telemetry, and screenshots.
  - Public demos use rate limits and low invoice amounts.

Architecture:

  - One canonical product contract.
  - One canonical provider registry.
  - One canonical price conversion algorithm.
  - One canonical payment state machine.
  - One canonical live-wallet harness with wallet profiles, using Rizful as the
    first fixture rather than the protocol center of gravity.
  - One canonical fruit-sticker demo product.
  - One conformance suite used by every SDK.
  - No required OpenReceive daemon, sidecar, gateway, or extra service.
  - Every backend SDK owns native NWC integration directly or through a
    recently maintained NWC client dependency.
  - Generated types are allowed; duplicated business rules are not.

Release strategy:

  - Build one excellent end-to-end flow before broad package expansion.
  - Start with native SDK mode for real framework developer experience.
  - Build or adopt NWC clients per language, but only adopt third-party clients
    that are recently maintained, permissively licensed, and pass conformance.
  - Do not publish 20 half-working libraries before the spec is stable.


Storage Contract And Invoice Lifecycle
--------------------------------------

The storage contract must be defined before the first SDK adapter. OpenReceive
depends on idempotency and settlement-action safety; do not let each language or
framework invent its own database semantics.

Persistence ownership:

  OpenReceive packages own OpenReceive invoice persistence inside the
  merchant's application database. Do not ask app developers to design their own
  invoice table, idempotency table, lifecycle model, or state transition code.
  Each backend/framework package must ship the native persistence shape for its
  ecosystem:

    Rails:
      migrations, ActiveRecord model, generated jobs/tasks.

    Node:
      SQL migrations or ORM templates for common app databases, plus a package
      store adapter contract. Early v0.1 may keep the demo in-memory store, but
      production docs must point developers to package-owned durable schema, not
      hand-rolled invoice tables.

    Django/FastAPI:
      Django migrations/models or SQLAlchemy/SQLModel table definitions and
      Alembic migration templates.

    Laravel/Symfony:
      package migrations, Eloquent/Doctrine models, queue job templates.

    Go/Rust/.NET/JVM/Elixir:
      SQL migrations and package-owned repository/store implementations suited
      to the ecosystem.

  The merchant app owns the database server/account, deployment, backups,
  tenant/user/order/cart references, and business hooks such as
  on_invoice_settlement. OpenReceive owns the invoice/idempotency/settlement
  rows and the durable transitions that make polling, notifications, replay,
  and restart recovery safe.

Required invoice row fields:

  invoice_id
  merchant_scope
  idempotency_key
  idempotency_request_hash
  payment_hash
  invoice
  amount_msats
  transaction_state
  workflow_state
  settlement_action_state
  created_at
  expires_at
  settled_at
  settlement_action_completed_at
  refreshed_from_invoice_id
  metadata
  fiat_quote

Required uniqueness and transition invariants:

  - invoice_id is unique.
  - payment_hash is unique where present.
  - idempotency_key is unique within the canonical idempotency scope:
    merchant_scope + operation + idempotency_key. For v0.1, merchant_scope is
    the app/tenant/merchant boundary supplied by the adapter, operation is
    invoice.create or invoice.refresh, and user/order/cart identifiers belong
    in metadata or in the idempotency_key value itself. Adapters may add a
    stricter custom namespace, but they must pass the canonical default
    conformance vectors.
  - Reusing an idempotency_key with the same request body returns the original
    invoice response.
  - Reusing an idempotency_key with a different request body returns 409.
  - Settlement transitions are idempotent.
  - Settlement action transitions are idempotent.
  - settlement_action_completed_at can be set once.
  - invoice.settled events can replay safely.
  - Notifications never directly run merchant settlement actions.
  - Expired invoices cannot be refreshed in-place; refresh creates a new
    invoice linked to the old one.

Exact storage model:

  {
    "transaction_state": "pending | settled | expired | failed | accepted",
    "workflow_state": "draft | invoice_created | verifying | settlement_action_pending | settlement_action_completed | expiry_pending_verification | expired_closed | failed_closed | cancelled",
    "settlement_action_state": "pending | completed | failed"
  }

Exact workflow lifecycle:

  draft -> invoice_created -> verifying -> settlement_action_pending -> settlement_action_completed
  draft -> invoice_created -> expiry_pending_verification -> expired_closed
  draft -> invoice_created -> failed_closed
  draft -> invoice_created -> cancelled

Notes:

  - workflow_state is OpenReceive's app-layer lifecycle.
  - transaction_state preserves the NIP-47 transaction state where available.
  - verifying means the backend is checking activity through lookup_invoice.
  - settlement_action_pending means backend lookup has proven settlement, but
    the merchant-supplied settlement action has not completed yet.
  - settlement_action_completed means OpenReceive's optional idempotent
    settlement action boundary completed. It does not say what merchant
    business effect happened.
  - If no settlement action hook is configured, adapters may complete the
    boundary as a no-op after backend settlement is proven.
  - checking, settlement_action_pending, settlement_action_completed, and other
    workflow labels must never be written to transaction_state.
  - Every schema, storage adapter, framework migration, and conformance vector
    must use this lifecycle.


Next developer-experience layer:

  The storage architecture is correct only if it does not make developers feel
  like payment-database engineers. After the durable storage contract is green,
  the next v0.1 work is package-native setup and diagnostics that hide the
  persistence machinery behind normal framework workflows.

  Required next work:

    Node:
      - Ship a package CLI with `openreceive init`, `openreceive migrate`,
        `openreceive doctor`, `openreceive poll`, and `openreceive listen`.
      - `init` should generate the smallest server-only config stub, env
        example, poll/listen entrypoints, and framework-specific route wiring.
      - `migrate` should run package-owned OpenReceive migrations without
        requiring developers to copy SQL from docs.
      - `doctor` should verify database connectivity, required tables/indexes,
        migration version, durable store usage in production, NWC preflight,
        secret redaction behavior, and poll/listen runner readiness.
      - Support Node SQLite as a first-class package-owned store alongside
        Postgres. SQLite is the preferred local/small-app path; Postgres is the
        preferred production/reference path. Both must use the same lifecycle,
        idempotency, recovery, and settlement-action conformance vectors.

    Rails:
      - Keep the native generator path as the model: install generator,
        package-owned migration/model/store, `bin/rails db:migrate`,
        `bin/rails openreceive:poll`, and `bin/rails openreceive:listen`.
      - Add a Rails doctor task that checks migration status, store wiring,
        NWC preflight, and worker readiness.

    All backend packages:
      - Refuse production boot or print an unmistakable fail-closed error when
        only in-memory storage is configured.
      - Document a small supported database matrix instead of implying every
        database is supported. A database is supported only when OpenReceive
        ships the store adapter, migration/setup path, and conformance coverage.
      - Keep user-owned fields in metadata or app tables. Do not make app
        developers add columns to OpenReceive-owned invoice rows.


SDK-First Implementation Model
------------------------------

OpenReceive has one primary implementation model:

  The app backend talks directly to NWC through a language-native OpenReceive
  SDK, which either wraps a vetted NWC library or depends on a standalone
  neutral NWC client maintained by this project.

    Rails app          -> openreceive-ruby      -> nwc-ruby / NWC wallet
    Laravel app        -> openreceive-php       -> nwc-client-php -> NWC wallet
    FastAPI app        -> openreceive-py        -> nwc-client-py  -> NWC wallet
    Node app           -> @openreceive/node     -> @getalby/sdk       -> NWC wallet
    Go app             -> openreceive-go        -> audited go-nwc or nwc-client-go -> NWC wallet
    Swift Vapor API    -> OpenReceiveVapor      -> NostrKit or NWCClient -> NWC wallet
    iOS app            -> Node/Express payment API -> NWC wallet

The important product decision: OpenReceive should feel like installing a gem,
Composer package, npm package, PyPI package, SwiftPM package, crate, module, or
NuGet package into the app the developer already deploys. It should not ask a
Rails, PHP, Python, Go, Swift, or .NET developer to operate an extra daemon.

Backend packages must still own the settlement runtime. App developers should
not have to learn and reimplement polling, final expiry lookup, notification
dedupe, settlement-action idempotency, or restart recovery. They provide
durable app-owned storage and choose the native runtime shape; OpenReceive
provides the lifecycle brain.

Frontend and mobile packages are UI clients only:

    Browser UI      -> merchant payment backend -> NWC wallet
    iOS/Android app -> merchant payment backend -> NWC wallet
    Static site     -> small API/proven serverless backend -> NWC wallet

The frontend/mobile decision is only: which backend endpoint do I call? The
server-side implementation is a separate choice owned by the merchant backend.

Inside that backend, the developer may use:

  Node/Express API    -> @openreceive/node     -> @getalby/sdk
  Rails app           -> openreceive-rails     -> nwc-ruby
  FastAPI app         -> openreceive-fastapi   -> nwc-client-py
  Laravel app         -> openreceive-laravel   -> nwc-client-php
  Vapor API           -> OpenReceiveVapor      -> NWCClient

Those implementation details never cross into browser, iOS, Android, Flutter,
or React Native apps. Client apps only receive invoice display data and passive
settlement state updates.

Default iOS demo choice:

  Build the first iOS demo as SwiftUI app -> Node/Express payment API -> NWC
  wallet. Internally, that API uses @openreceive/node and @getalby/sdk.
  Node/Express is common for mobile API backends, aligns with the v0.1
  OpenReceive stack, and avoids blocking the iOS UI demo on the Swift/Vapor
  server adapter. Keep ios-swiftui-vapor-api as the later Swift-native backend
  proof.

Default build strategy:

  - Build the spec, wallet-profile live NWC test harness, conformance smoke
    suite, and one excellent native Node/React stack first.
  - Build Ruby/Rails next using nwc-ruby as the model for in-app background
    workers and notification listeners.
  - Add neutral standalone NWC clients where the target language lacks a fresh,
    dependable client.
  - Keep conformance identical across all backend SDKs and NWC clients.


Package-Owned Polling And Notification Runners
----------------------------------------------

Every serious server-side OpenReceive package must ship two package-owned
backend runners:

  1. A settlement polling runner.
  2. A payment notification listener runner, where the wallet/NWC client
     supports payment_received subscriptions.

These runners are not OpenReceive-hosted daemons, sidecars, gateways, or
separate OpenReceive databases. They are code installed into the merchant's own
backend and run as two separate backend processes, worker roles, scheduler
commands, or job-system roles. Do not model this as multithreading inside one
runtime. The clear production shape is:

  web process       -> mounts OpenReceive HTTP routes
  poll process      -> runs settlement polling/recovery
  listen process    -> runs payment notification subscription where supported

OpenReceive owns:

  - Invoice persistence schema, migrations/templates, and store/repository
    implementations inside the app database.
  - Restart recovery for non-terminal invoices.
  - Canonical polling cadence.
  - Final lookup at local expiry.
  - Grace lookups when wallet truth is inconclusive.
  - A long-running notification listener as a passive accelerator.
  - Settlement detection rules.
  - Duplicate-safe settlement transitions.
  - The idempotent settlement-action boundary.
  - Framework-native runner commands, jobs, tasks, or services for both
    polling and listening.

The merchant application owns:

  - The durable database instance/account and backup policy.
  - Auth, tenant, user, cart, checkout, and order boundaries.
  - The settlement action's business effect.
  - The deployment shape for its normal web, worker, command, or scheduler
    processes.

The required backend package API shape:

  app_openreceive = create/configure OpenReceive with wallet + store + hooks
  app_openreceive.mount_routes()
  app_openreceive.run_settlement_polling_process()
  app_openreceive.run_payment_notification_process()

Packages should expose functions that are easy to call from separate process
entrypoints:

  Node/Express:
    node scripts/openreceive-poll.js
    node scripts/openreceive-listen.js

  Rails:
    generated jobs/tasks for polling and listening.

For production, each package must also expose the native worker shape:

  Node:
    `openreceive poll` and `openreceive listen` CLI commands, plus
    function exports for custom poll/listen scripts.

  Rails:
    generated ActiveJob/Solid Queue/Sidekiq/GoodJob job, plus
    `bin/rails openreceive:poll` and `bin/rails openreceive:listen`.

  Django/FastAPI:
    management/app commands for poll and listen, Celery/RQ/Dramatiq/Arq tasks,
    and scheduler recipes.

  Laravel/Symfony:
    Artisan/console poll and listen commands, queue workers, and
    scheduler/Messenger recipes.

  Go:
    `RunSettlementPollingWorker(ctx, store, wallet)` and
    `RunPaymentNotificationListener(ctx, store, wallet)` for the app's
    ordinary worker binary or service process.

  ASP.NET Core:
    separate `IHostedService` integrations and DI registration.

  Phoenix:
    Oban polling worker and supervised notification listener using the same
    storage contract.

On poll process boot, the polling runner must query app-owned durable storage
for invoices whose workflow is not terminal and whose local expiry plus grace
window has not passed, plus settled invoices whose settlement action has not
completed. It then continues lookup polling until each invoice is settled,
expired, failed, cancelled, or its settlement action boundary completes.

The notification listener runner must hold the NWC payment_received
subscription open where supported. Each notification is only a hint: the runner
must lookup_invoice by payment_hash before marking settlement or running the
settlement action. If notification support is absent, the listener command must
skip or fail clearly while the polling runner remains sufficient for
correctness.

Do not make multiprocess coordination, thread pools, or lock managers part of
the first runner contract. Settlement and settlement-action writes must still
be idempotent and replay-safe, but v0.1 should keep the public installation
story to two obvious package-provided processes: poll and listen.


Serverless And Edge Policy
--------------------------

Do not promise serverless or edge backends as a complete OpenReceive path until
the settlement-loop and crypto constraints are proven.

The problem:

  - A normal serverless function cannot sleep for ten minutes to run the
    canonical polling cadence.
  - A normal serverless function is not a long-running Nostr relay listener.
  - Edge runtimes do not provide secp256k1 Schnorr signing through WebCrypto.
  - Non-JS NWC clients with native crypto bindings generally cannot run at the
    edge.

Allowed stateless patterns:

  - Client-driven polling of a backend lookup endpoint, with backend
    authorization and rate limits.
  - Scheduled/cron functions that poll open invoices from durable storage.
  - Durable primitives such as Cloudflare Durable Objects, queues, or
    long-lived worker-style runtimes that can own invoice state.
  - A small ordinary API service behind the static site.

Edge constraints:

  - Treat edge/serverless NWC as JS-only until proven otherwise.
  - Use noble-curves/noble-secp256k1 or another audited JS/WASM crypto path
    for secp256k1/Schnorr where native bindings are unavailable.
  - Prove NIP-44 ChaCha20/HKDF behavior and bundle size in Cloudflare Workers
    before publishing an edge quickstart.
  - Do not support live checkout in pure frontend code.

v0.1 policy:

  - The first static HTML demo should use a small stateful API or a proven
    scheduled/durable backend, not an assumed one-shot serverless function.
  - Cloudflare Worker, Vercel Edge, and similar examples are later compatibility
    proofs after the JS reference path is green.
  - If a serverless example is included, its README must name who owns polling:
    browser lookup loop, cron job, durable object, queue worker, or external
    small API.


Canonical HTTP Contract
-----------------------

The HTTP API is the application-facing contract exposed by framework adapters,
demos, and merchant backends. OpenReceive does not ship a required HTTP daemon;
instead, adapters mount these routes inside Rails, Express, FastAPI, Django,
Laravel, Spring, ASP.NET Core, Phoenix, Vapor, and similar apps.

This contract is still valuable because it keeps frontend checkout components,
mobile apps, demos, docs, tests, and server adapters aligned. A React checkout,
SwiftUI checkout, static-site widget, or mobile app should be able to talk to
the merchant's backend using the same invoice and event shapes no matter which
server framework created the invoice.

NWC naming policy:

  The canonical OpenReceive HTTP payload should mirror NIP-47 field names as
  closely as practical, except where a NIP-47 name collides with OpenReceive
  app-layer state. Use invoice for the BOLT11 string, expiry for seconds from
  invoice creation, and payment_hash, description, description_hash, metadata,
  created_at, expires_at, and settled_at exactly as NIP-47 names them.
  Map NIP-47 result.state to transaction_state in OpenReceive storage, events,
  and public HTTP payloads so logs and app events cannot confuse wallet truth
  with workflow state.

  Intentional naming exception: OpenReceive uses amount_msats instead of NIP-47
  amount in its public HTTP and SDK payloads. In NIP-47, amount means
  millisatoshis, but in merchant APIs amount is too easy to confuse with fiat,
  sats, or display money. NWC adapters map amount_msats to NIP-47 amount when
  sending requests and map NIP-47 result.amount back to amount_msats.

  Any OpenReceive payload that carries satoshi or millisatoshi data must use
  explicit unit names: amount_sats or amount_msats. Do not use amount,
  amount_sat, or amount_msat outside raw NIP-47 request/response handling.

Money JSON representation:

  - fiat.value is a string decimal.
  - btc_fiat_price is a string decimal.
  - amount_sats is a JSON integer within the safe integer range.
  - amount_msats is a JSON integer within the safe integer range.
  - JSON Schema must set explicit minimum and maximum limits for amount_sats
    and amount_msats.
  - amount_msats minimum for v0.1: 1000. OpenReceive does not accept sub-sat
    invoice amounts.
  - amount_sats minimum for v0.1: 1.
  - amount_msats maximum for v0.1: 9007199254740991.
  - amount_sats maximum for v0.1: 9007199254740, so converting to msats stays
    within the same safe integer boundary.
  - Internal SDKs should use BigInt or equivalent for integer money units and
    Decimal/fixed-point types for fiat and exchange-rate math.
  - Do not rely on JavaScript number for fiat math.
  - If a future contract needs values above the safe integer range, introduce a
    v2 string-integer field instead of silently widening v1 behavior.

  OpenReceive-specific fields are allowed only where they describe app-layer
  behavior that NIP-47 does not model: invoice_id, fiat, fiat_quote, checkout,
  settlement_action_state, routes, auth, and idempotency. Public frontend
  responses may hide sensitive fields such as preimage and expose
  preimage_present instead. They must not include product delivery URLs,
  entitlement data, file paths, or other deliverable-specific fields.

  Do not copy the encrypted NWC event envelope into HTTP responses. HTTP routes
  already name the operation, so method, params, and result_type are not needed
  unless a package is implementing a raw NWC proxy/debug tool.

Authentication and authorization:

  OpenReceive does not define a standalone user-auth system. Framework adapters
  mount into the merchant's app and must integrate with the app's existing auth
  model: Rails sessions/Devise, Laravel auth middleware, NextAuth/Auth.js,
  Express middleware, Django auth, Phoenix plugs, ASP.NET policies, or a
  custom guest-checkout/session-token system.

  The app decides who can create an invoice, who can read an invoice, who can
  subscribe to events, and what app-owned action happens after settlement.
  OpenReceive provides
  hooks, helpers, and safe defaults; it does not decide what "logged in",
  "owns this order", "guest checkout", or "admin" means.

  Typical endpoint policy:

    POST /openreceive/v1/invoices
      Protected by app auth, cart/session token, or guest checkout token.
      Cookie-authenticated POSTs must use CSRF protection.

    GET /openreceive/v1/invoices/{invoice_id}
      Allowed only when the invoice belongs to the current user, cart, order,
      or checkout session.

    POST /openreceive/v1/invoices/lookup
      Server-side or strongly authorized use only. Do not expose payment_hash
      lookup as a public status oracle. The adapter must verify the current
      user, cart, order, checkout session, or backend service principal is
      allowed to inspect the invoice before returning status.

    GET /openreceive/v1/invoices/{invoice_id}/events
      Uses same-site session cookies, a short-lived signed event URL, or a
      polling fallback when EventSource auth is awkward on the platform.
      Signed event URLs must be scoped to one invoice, use a short TTL, avoid
      sensitive query names, and be protected from log/referrer leakage as far
      as the framework allows.

    GET /openreceive/v1/routes, /providers, /rates
      Usually public/cacheable but still rate-limited.

  CORS and browser security defaults:

    - Deny cross-origin credentialed access by default.
    - Allow only configured origins.
    - Never use wildcard CORS together with credentials.
    - Set Referrer-Policy on pages that use signed event URLs.
    - Avoid logging full signed event URLs.
    - Rate-limit invoice creation, invoice reads, and lookup endpoints.
    - Return no-store on invoice, lookup, and event responses.

Base path:

  /openreceive/v1

Core endpoints:

  POST   /openreceive/v1/invoices
  GET    /openreceive/v1/invoices/{invoice_id}
  POST   /openreceive/v1/invoices/lookup
  GET    /openreceive/v1/invoices/{invoice_id}/events
  POST   /openreceive/v1/invoices/{invoice_id}/refresh
  GET    /openreceive/v1/rates
  POST   /openreceive/v1/rates/quote
  GET    /openreceive/v1/routes

Create invoice request:

  POST /openreceive/v1/invoices
  Content-Type: application/json
  Idempotency-Key: fruit-demo-user-123-order-456

  The canonical request body mirrors NIP-47 make_invoice params except that
  OpenReceive uses amount_msats instead of amount for clarity.

  Canonical create requests must contain exactly one amount source:

    amount_msats

  or:

    fiat

  Do not send both amount_msats and fiat in the same create request. If fiat is
  used, the adapter quotes fiat to msats first, stores the locked fiat_quote,

  The simple canonical example and most SDK examples should use amount_msats.

    - Use description for normal checkout.
    - Include product, order, or checkout details in description so the payer
      and merchant can recognize the invoice.
    - description_hash is advanced/raw mode only.
    - Exactly one of description or description_hash may be present.
    - Demos and quickstarts should use description, not description_hash.

QR codes:

  QR rendering is not part of the canonical backend invoice response. The
  backend returns the BOLT11 invoice string in invoice, and OpenReceive
  frontend/mobile packages generate safe QR codes from that string. This is how
  React, Vue, Svelte, Angular, SwiftUI, Android Compose, Flutter, and
  web-component checkouts should work.

  OpenReceive must not rely on application developers to remember QR scanning
  details. Every official QR helper, primitive, component, and downloadable PNG
  renderer must use safe defaults:

    payload: lightning:<invoice>
    background: opaque white
    foreground: black or another tested high-contrast color
    quiet zone: at least 4 modules, white, included inside the rendered image
    error correction: conservative default suitable for Lightning invoices
    exported PNG/SVG: includes the same quiet zone and opaque background

  Frontend packages should expose both:

    OpenReceiveQRCode / equivalent framework component
    createQrSvg(), createQrPng(), and createLightningUri() helpers

  Developers may wrap, size, place, brand, or download the QR output, but the
  OpenReceive renderer owns the payload, quiet zone, and contrast defaults.

  Framework adapters may offer optional QR helpers for no-JavaScript fallback
  pages, server-rendered emails, PDFs, or unusual CMS/plugin environments, but
  those helpers are not required OpenReceive endpoints, should not be included
  in the canonical invoice response, and must share the same QR safety defaults
  and tests as the frontend helpers.

Refresh invoice:

  POST /openreceive/v1/invoices/{invoice_id}/refresh
  Content-Type: application/json
  Idempotency-Key: fruit-demo-user-123-order-456-refresh-1

  Refresh never mutates the original invoice. It creates a new invoice linked
  to the old invoice and leaves the old invoice's audit history intact.

  Refresh is allowed only when the original invoice is expired, failed, or
  otherwise explicitly refreshable by adapter policy. Refreshing an already
  settled invoice returns 409.

  Request:

  {
    "reason": "expired"
  }

  Response:

  {
    "old_invoice_id": "or_inv_old",
    "new_invoice_id": "or_inv_new",
    "reason": "expired",
    "invoice": {
      "invoice_id": "or_inv_new",
      "type": "incoming",
      "transaction_state": "pending",
      "workflow_state": "invoice_created",
      "invoice": "lnbc...",
      "payment_hash": "def456...",
      "amount_msats": 200000,
      "created_at": 1781741500,
      "expires_at": 1781742100
    }
  }

  Storage rules:

    - old_invoice_id keeps its original payment_hash, invoice,
      transaction_state, workflow_state, and settlement action records.
    - new_invoice_id has a new payment_hash and new BOLT11 invoice.
    - refreshed_from_invoice_id on the new row points to old_invoice_id.
    - The idempotency key for refresh is scoped separately from initial create.
    - Replaying the same refresh request with the same idempotency key returns
      the same new invoice.

Events endpoint:

  GET /openreceive/v1/invoices/or_inv_01jz.../events
  Accept: text/event-stream

Use SSE first because it is simple, portable, proxy-friendly, and works in
almost every stack. Framework adapters can bridge the same events into
ActionCable, Phoenix Channels, Socket.IO, SignalR, WebSockets, mobile push, or
managed realtime services.

SSE remains the default transport. Heartbeat comments are only a keepalive and
stale-connection detector; they do not add business semantics. If a framework
already has a mature realtime channel, such as ActionCable, Phoenix Channels,
Socket.IO, SignalR, or Laravel Reverb, adapters may expose that too, but the
payload and resume semantics still come from the canonical event contract.

Required SSE headers:

  Content-Type: text/event-stream
  Cache-Control: no-store

Connection: keep-alive is optional and only meaningful for HTTP/1.1. Do not
require it in the contract because Cloudflare and many modern deployments use
HTTP/2, where that header is a no-op or invalid.

Required SSE behavior:

  - Each event includes id: <monotonic event id>.
  - Event ids are monotonic per invoice event stream.
  - Clients send Last-Event-ID when reconnecting.
  - The server replays missed events for that invoice when possible.
  - The server sends heartbeat comments, for example ": heartbeat", at a
    conservative interval such as 15-30 seconds.
  - Signed event URLs are scoped to one invoice.
  - Signed event URLs expire.
  - Event responses are never cacheable.
  - The browser must verify invoice_id and payment_hash match the current
    invoice before updating UI.
  - Events are passive UI hints; they never directly run merchant settlement
    actions.

Event names:

  invoice.created
  invoice.verifying
  invoice.settled
  invoice.expired
  invoice.failed
  invoice.settlement_action_completed
  invoice.cancelled

Example SSE:

  id: 1
  event: invoice.created
  data: {"invoice_id":"or_inv_01jz...","type":"incoming","transaction_state":"pending","workflow_state":"invoice_created"}

  : heartbeat

  id: 2
  event: invoice.verifying
  data: {"invoice_id":"or_inv_01jz...","type":"incoming","transaction_state":"pending","workflow_state":"verifying"}

  id: 3
  event: invoice.settled
  data: {"invoice_id":"or_inv_01jz...","type":"incoming","transaction_state":"settled","workflow_state":"settlement_action_pending","payment_hash":"abc123...","amount_msats":200000,"settled_at":1781740990,"settlement_action_state":"pending"}

  id: 4
  event: invoice.settlement_action_completed
  data: {"invoice_id":"or_inv_01jz...","type":"incoming","transaction_state":"settled","workflow_state":"settlement_action_completed","payment_hash":"abc123...","amount_msats":200000,"settlement_action_state":"completed","settlement_action_completed_at":1781740991}

  id: 5

  data: {"invoice_id":"or_inv_01jz...","type":"incoming","transaction_state":"expired","workflow_state":"expired_closed"}

  ActionCable, Socket.IO, SSE, WebSocket, SignalR, Phoenix Channels, and
  Laravel Reverb adapters can all carry the canonical invoice.settled event
  payload. NIP-47 payment_received remains the wallet notification name, not
  the OpenReceive browser event name.


Payment State Machine
---------------------

All SDKs, examples, database rows, UI components, and event payloads should
preserve NIP-47 transaction state names wherever they describe the invoice
itself.

Canonical transaction states:

  pending
    Customer has not paid yet.

  settled
    Wallet lookup proves settlement.

  expired
    Invoice expired before settlement.

  failed
    Wallet, provider, network, validation, or internal error occurred.

  accepted
    Reserved for hold-invoice semantics. OpenReceive v0.1 does not create hold
    invoices, but clients should not choke on this NIP-47 state.

OpenReceive workflow states:

  These are app-layer states, not replacements for NIP-47 state. Use them only
  in local storage, UI state machines, or event payloads. They deliberately do
  not reuse pending, settled, expired, failed, or accepted.

  draft
    App has requested a payment intent, but no BOLT11 invoice exists yet.

  invoice_created
    Wallet returned an invoice and payment_hash.

  verifying
    A notification, poll, or user action suggests activity; server is verifying
    by lookup_invoice.

  settlement_action_pending
    Backend lookup has proven settlement, but the optional merchant settlement
    action has not completed yet.

  settlement_action_completed
    OpenReceive's optional idempotent settlement action boundary completed.
    The product, entitlement, credit, download, or other app effect remains
    private app-owned state.

  expiry_pending_verification
    Local expiry time has passed, but wallet truth has not yet conclusively
    returned settled or expired.

  expired_closed
    Wallet lookup confirmed expiry, or the grace verification policy completed
    without settlement.

  failed_closed
    Wallet, provider, network, validation, or internal error occurred and the
    app has closed the invoice workflow as failed.

  cancelled
    Merchant/app intentionally cancelled the payment flow.

Typical workflow transitions:

  draft -> invoice_created
  invoice_created -> verifying
  verifying -> invoice_created
  verifying -> settlement_action_pending
  settlement_action_pending -> settlement_action_completed
  invoice_created -> expiry_pending_verification
  verifying -> expiry_pending_verification
  expiry_pending_verification -> settlement_action_pending
  expiry_pending_verification -> expired_closed
  invoice_created -> failed_closed
  verifying -> failed_closed
  invoice_created -> cancelled

Invariant:

  Merchant settlement actions are never triggered by frontend state alone. A
  settlement action may run only after backend verification through
  lookup_invoice. For NIP-47, require settled_at or transaction_state/state ==
  "settled". Treat preimage
  as corroborating proof only because a receiver can already know the preimage
  when it creates the invoice. Normalize older library-specific variants at
  adapter boundaries only.


NWC Protocol Foundation
-----------------------

NWC URI shape:

  nostr+walletconnect://<wallet-pubkey>?relay=<wss-url>&secret=<client-secret>

The URI parser must support:

  - Wallet service pubkey in the URI path.
  - One or more relay query parameters.
  - Required 32-byte hex client secret.
  - Optional lud16 query parameter.
  - Percent-encoded relay URLs and parameters.

NWC connection bootstrap:

  1. Parse the NWC URI.
  2. Connect to the relay or relays from the URI.
  3. Fetch the wallet service replaceable info event, kind 13194.
  4. Parse the plaintext capabilities in content.
  5. Parse the notifications tag, if present.
  6. Parse the encryption tag, if present.
  7. Choose an encryption mode for requests and notification subscription.

Encryption policy:

  NIP-04 is still widely deployed and must work for initial compatibility.
  NIP-44 v2 should be supported where practical and preferred when the wallet
  service advertises it in the info event. If the info event has no encryption
  tag, assume NIP-04. Requests should include an encryption tag when using
  NIP-44 v2. Unsupported encryption errors must normalize cleanly.

Crypto portability policy:

  NWC clients are real Nostr clients. They must sign events, verify public-key
  material, encrypt/decrypt NIP-04, and support NIP-44 v2 where practical.
  This creates platform-specific dependency work that must be planned before
  promising easy installation in every ecosystem.

  Requirements:

    - Every NWC client must document its secp256k1/Schnorr, ECDH,
      AES-256-CBC for NIP-04, ChaCha20, HKDF, and random-source dependencies.
    - Every backend SDK README must say whether the NWC dependency is pure
      language, JS/WASM, or native.
    - If native dependencies are required, provide install notes for macOS,
      Debian/Ubuntu, Alpine/musl, ARM Linux, and Docker.
    - Official Dockerfiles should include the required system packages.
    - Users who do not use the official Dockerfile must have apt, apk, brew,
      yum/dnf, or equivalent instructions.
    - CI should test the supported native dependency combinations before a
      package is published.
    - Edge/runtime support must be proven separately from ordinary server
      support.

  Crypto soundness challenge:

    The real implementation risk is not only NWC method coverage. Writing
    standalone clients in Python, PHP, JVM, .NET, Elixir, Go, Swift, Dart, or
    other ecosystems means choosing and composing secp256k1/Schnorr, ECDH,
    NIP-04 AES-CBC, NIP-44 ChaCha20/HKDF, randomness, and serialization
    libraries correctly. Deterministic test vectors are mandatory, and the
    nwc-ruby experience shows the right vectors can make a client dependable,
    but vectors only prove input-to-output equivalence. They do not prove nonce
    freshness, constant-time comparisons, entropy quality, or safe key
    lifecycle behavior.

    Before a new standalone NWC client becomes an OpenReceive dependency, it
    must use well-maintained cryptographic primitives, run official NIP-04 and
    NIP-44 vectors where available, include negative/tamper tests, document RNG
    and nonce behavior, pass install tests on target platforms, and receive a
    security review focused on cryptographic misuse. If a maintained audited
    library exists, prefer wrapping it over hand-assembling crypto.

  Practical starting assumptions:

    - JS/TypeScript can target Node first and use a JS/WASM crypto path for
      edge experiments.
    - Ruby may require rbsecp256k1 or another native dependency when using
      nwc-ruby; document and test it explicitly.
    - Python, PHP, JVM, .NET, Go, Rust, Swift, Elixir, and Dart each need a
      specific crypto dependency decision before implementation starts.
    - Do not claim "install a package and it just works everywhere" until the
      package has passing install tests and documented native dependency paths.

OpenReceive receive-checkout methods for v0.1:

  get_info
  make_invoice
  lookup_invoice
  list_transactions, optional
  get_balance, optional diagnostic only

get_balance is server-side diagnostic/admin data. It must never be returned to
the public checkout frontend, included in public invoice responses, exposed
through browser/mobile SDKs, or logged in user-visible responses. If a merchant
dashboard uses get_balance, it must do so on the server with app-owned admin
authorization.

Standalone nwc-client-* protocol libraries should also support:

  pay_invoice

Out of scope for v0.1 standalone clients and for OpenReceive checkout:

  pay_keysend
  multi_pay_keysend
  make_hold_invoice
  cancel_hold_invoice
  settle_hold_invoice
  hold_invoice_accepted

multi_pay_invoice can be considered later, but is not required for v0.1.

Receive-only expectation:

  A receive-only code should support creating invoices and checking settlement,
  but should not support spending funds. OpenReceive checkout APIs never expose
  send-payment methods.

Automatic wallet preflight:

  v0.1 does not require a manual wallet-compatibility ceremony, but every live
  backend SDK must run an automatic preflight before accepting an NWC code for
  checkout:

    1. Parse and redact the NWC URI.
    2. Fetch the wallet service info event.
    3. Confirm make_invoice and lookup_invoice are advertised or proven usable.
    4. Negotiate NIP-04 or NIP-44 encryption.
    5. Warn if the connection appears to have spend capability.
    6. Fail closed if required methods or encryption are unavailable.
    7. Record a non-secret wallet/service capability summary for diagnostics.

  Live test configuration should use:

    OPENRECEIVE_NWC
    OPENRECEIVE_WALLET_PROFILE=rizful|alby|zeus|custom
    expected_capabilities.json

  Rizful is the first supported live fixture, not a hard-coded policy center.

Notifications:

  - NWC notification event kind 23197 is used for NIP-44.
  - NWC notification event kind 23196 is used for NIP-04 compatibility.
  - Subscribe according to the negotiated/advertised encryption mode.
  - Filter notifications by wallet service pubkey and the client pubkey p tag.
  - Notifications can wake invoice lookup immediately.
  - Notifications do not replace polling.
  - Notification handlers must be idempotent.
  - Duplicate notifications, including duplicate NIP-04/NIP-44 notifications
    from wallets that publish both, must not double-credit payments.

NIP-47 event shape:

  - Info event: kind 13194, plaintext capability list in content, optional
    notifications tag, optional encryption tag.
  - Request event: kind 23194, encrypted JSON object with method and params,
    p tag set to the wallet service pubkey, optional expiration tag as unix
    timestamp seconds, encryption tag when using NIP-44 v2.
  - Response event: kind 23195, encrypted JSON object with result_type, error,
    and result; p tag set to client pubkey; e tag should reference the request.
  - Successful responses have error null.
  - Failed responses have error.code and error.message and result null.

NIP-47 to OpenReceive field mapping:

  - result.invoice -> invoice
  - result.amount -> amount_msats
  - result.payment_hash -> payment_hash
  - result.created_at / expires_at / settled_at stay unix timestamps in
    canonical HTTP responses.
  - result.state -> transaction_state, preserving lowercase NIP-47 values such
    as pending, settled, accepted, expired, and failed.
  - OpenReceive-specific invoice_id, fiat_quote, checkout, routes, and
    settlement_action_state fields may be added around the NIP-47 transaction
    object. Deliverable fields must stay in the merchant app.

Long-running listener requirements:

  Every native listener implementation should satisfy these requirements:

    - Dedupe notifications by payment_hash.
    - Persist the last processed event timestamp or cursor where the wallet
      library supports resuming with since.
    - Reconnect with bounded exponential backoff.
    - Keep WebSocket connections alive with ping/heartbeat behavior.
    - Treat notification delivery as at-least-once, not exactly-once.
    - Make the database transition idempotent with a unique payment_hash or an
      update guard such as pending -> settled only.
    - Run notification listeners as long-running worker processes, not inside
      request handlers.
    - Still verify settlement with lookup_invoice before running any merchant
      settlement action.

Settlement detection:

  For NIP-47 incoming invoices, treat invoice as settled only when
  lookup_invoice returns one of:

    settled_at present
    state == "settled"

  Treat preimage as corroborating proof only. In Lightning, the receiver
  generated the preimage and derived the payment_hash when the invoice was
  created, so a wallet can know the preimage before payment. NIP-47 says
  preimage should be present if paid, but real wallet behavior may vary; any
  wallet that returns preimage at creation would create a false-positive if
  OpenReceive treated preimage alone as settlement.

  Normalize legacy/library-specific variants such as settled true, paid true,
  or uppercase SETTLED only at adapter boundaries. Store all available proof,
  including preimage when present, but make settled_at or state == "settled"
  the required settlement signal.

Metadata:

  NIP-47 metadata must fit in wallet/relay payload constraints. OpenReceive
  should keep NWC metadata below 3900 characters after JSON serialization,
  leaving margin under the NIP-47 4096-character limit. Larger application
  metadata stays in the merchant database and is referenced by ids such as
  order_id, cart_id, user_id, or checkout_session_id.

Common wallet/service errors to normalize:

  NIP-47 native:

  NOT_IMPLEMENTED
  RESTRICTED
  UNAUTHORIZED
  RATE_LIMITED
  QUOTA_EXCEEDED
  INTERNAL
  UNSUPPORTED_ENCRYPTION
  INSUFFICIENT_BALANCE
  PAYMENT_FAILED
  OTHER

  OpenReceive normalized/internal:

  NOT_FOUND
  TIMEOUT
  INVALID_REQUEST
  WALLET_UNAVAILABLE
  INVOICE_EXPIRED
  UNSUPPORTED_METHOD

For receive-only checkout, send-payment errors such as INSUFFICIENT_BALANCE and
PAYMENT_FAILED should normally appear only in diagnostics or if a developer
accidentally configures read/write features. Normalize them anyway so SDKs
match the underlying NWC libraries.


Safe Polling
------------

Polling is reliable and should be the default. Notifications make checkout feel
instant, but polling is the recovery path when notification delivery is delayed
or missed. Backend packages must expose this as a settlement polling runner, so
app developers configure storage and runtime rather than hand-coding this loop.

Canonical polling cadence:

  no immediate post-create lookup unless a notification arrives
  0-15 seconds:       poll every 2 seconds
  15-60 seconds:      poll every 5 seconds
  60-180 seconds:     poll every 10 seconds
  180s-expiry:        poll every 20 seconds
  at local expiry:    perform one final lookup
  after expiry:       short grace window when wallet truth is inconclusive

Pseudo-code:

  now = clock.now()
  next_lookup_at = invoice.created_at + 2

  while now < invoice.expires_at:
    sleep_until(min(next_lookup_at, invoice.expires_at))
    now = clock.now()

    if now >= invoice.expires_at:
      break

    result = lookup_invoice(payment_hash)

    if isSettled(result):
      mark_settled()
      publish_invoice_settled()
      run_settlement_action_once()
      return

    next_lookup_at = now + nextDelay(invoice.created_at, now)

  final = lookup_invoice(payment_hash)

  if isSettled(final):
    mark_settled()
    publish_invoice_settled()
    run_settlement_action_once()
    return

  if isWalletExpired(final):
    mark_expired_closed()
    publish_invoice_expired()
    return

  mark_expiry_pending_verification()

  for attempt in graceLookups(invoice.expires_at, clock.now()):
    sleep(attempt.delay)
    result = lookup_invoice(payment_hash)

    if isSettled(result):
      mark_settled()
      publish_invoice_settled()
      run_settlement_action_once()
      return

    if isWalletExpired(result):
      mark_expired_closed()
      publish_invoice_expired()
      return

  mark_expired_closed()
  publish_invoice_expired()

This cadence must live in test vectors so every language agrees.


Fiat Pricing And Quotes
-----------------------

OpenReceive must have first-class fiat to msat quoting. Most merchants think in
USD, EUR, GBP, or local currency, not msats.

Core API:

  quoteFiatToMsats({
    fiat: {
      currency: "USD",
      value: "0.10"
    }
  })

Response:

  {
    "fiat": {
      "currency": "USD",
      "value": "0.10"
    },
    "btc_fiat_price": "50000.00",
    "amount_sats": 200,
    "amount_msats": 200000,
    "source": "static_mock",
    "as_of": 1781740800,
    "expires_at": 1781741400
  }

Rules:

  - fiat.value is a string decimal in HTTP JSON.
  - btc_fiat_price is a string decimal in HTTP JSON.
  - amount_sats and amount_msats are JSON integers constrained to the v0.1
    safe ranges defined in the HTTP contract.
  - Generated clients must test boundary values for amount_sats and
    amount_msats.
  - Use decimal math.
  - Do not use binary floats for money.
  - Lock the fiat quote at invoice creation.
  - Store price source, rate, as_of, expires_at, fiat amount, fiat currency,
    and amount_msats.
  - Enforce a minimum of 1 sat.
  - Round up to a whole sat when wallet/UI behavior needs sat-level display.
  - Amount to NWC make_invoice is in millisatoshis. OpenReceive calls this
    amount_msats and maps it to NIP-47 amount at the adapter boundary.

Formula:

  amount_btc   = fiat_amount / btc_fiat_price
  amount_sats  = amount_btc * 100_000_000
  amount_msats = amount_sats * 1000

Default price source architecture:

  PriceProvider
    getBtcFiatRates(currencies: string[]): RateMap

Initial providers:

  - CoinGecko Simple Price compatible endpoint.
  - OpenReceive exchange-rate mirror endpoint.
  - Megalithic exchange-rate fallback endpoint.
  - CoinGecko direct fallback endpoint.
  - Static deterministic provider for tests and screenshots.

Rate endpoint contract:

  The canonical live response shape is CoinGecko Simple Price compatible:

    {
      "bitcoin": {
        "usd": 62599,
        "eur": 54792,
        "gbp": 47539
      }
    }

  The adapter must accept integer or decimal JSON numbers and convert them to
  Decimal/string internally. Do not rely on JavaScript floating point for final
  money math.

  Official OpenReceive mirror, planned:

    https://openreceive.org/exchange_rates

  Existing compatible fallback:

    https://megalithic.me/exchange_rates

  CoinGecko direct fallback:

Anti-Duplication Strategy
-------------------------

This is the core engineering discipline for the whole project.

Never duplicate:

  - provider lists
  - supported currencies
  - route rankings
  - fiat-to-msat behavior
  - polling cadence
  - settlement detection rules
  - invoice schemas
  - event schemas
  - demo product data
  - live wallet profile behavior
  - package README structure

Use these mechanisms:

  - JSON Schema validates canonical data.
  - OpenAPI generates shared HTTP models.
  - AsyncAPI documents shared event models.
  - Codegen emits provider-data packages.
  - Test vectors enforce behavior across languages.
  - The shared live-wallet harness gives every SDK the same real-wallet
    behavior in integration tests, with Rizful as the first profile.
  - Scaffolding tools create examples from templates.
  - Docs use shared snippets where possible.
  - CI fails when generated files are stale.

Where duplication is acceptable:

  - Thin idiomatic wrappers in each ecosystem.
  - Framework-specific adapters.
  - UI components written in each target UI framework.
  - Small example glue code that teaches local conventions.

Rule of thumb:

  The same business rule in two languages is acceptable only if both load the
  same vector and prove identical output.


NWC Client Libraries
--------------------

The shared implementation burden moves into NWC client libraries and
conformance, not into a daemon. Each serious server-side SDK must either wrap a
recently maintained NWC client or ship a standalone neutral NWC client that can
be used without OpenReceive.

That is an architectural tradeoff, not a free lunch. The cost/risk center for
many-language support is cryptographic correctness inside the NWC clients.
OpenReceive should not write fresh crypto glue in eleven ecosystems merely to
claim coverage. Each new client needs vetted primitives, official vectors,
negative tests, live wallet smoke tests, install tests, and explicit review of
RNG, nonce, constant-time, and key-handling behavior before it becomes a
supported dependency.

These NWC clients are standalone protocol clients, not OpenReceive-only
packages. They may live under the openreceive GitHub organization for
stewardship, CI, release automation, and security response, but their names,
READMEs, examples, and public APIs should make sense to any NWC developer.

Canonical standalone naming:

  Repositories:
    nwc-client-js, if the project ever needs its own JS client - FOR NOW WE USE ALBY SDK
    nwc-client-ruby, if nwc-ruby is mirrored or superseded - FOR NOW WE USE NWC-RUBY
    nwc-client-php
    nwc-client-py
    nwc-client-go
    nwc-client-rs
    nwc-client-swift
    nwc-client-jvm
    nwc-client-dotnet
    nwc-client-elixir
    nwc-client-dart

  Public package names:
    npm:        @nwc-client/core, only if a standalone JS client is needed
    RubyGems:   nwc-client-ruby, only if nwc-ruby is not the long-term package
    Packagist:  nwc-client/php
    PyPI:       nwc-client-py, import name nwc_client
    Go:         github.com/openreceive/nwc-client-go
    crates.io:  nwc-client-rs
    SwiftPM:    repo nwc-client-swift, product NWCClient
    Maven:      org.nwcclient:nwc-client-jvm, subject to namespace setup
    NuGet:      Nwc.Client
    Hex:        nwc_client
    pub.dev:    nwc_client, only for server-side Dart if needed

If a neutral GitHub or package namespace is created later, prefer that for raw
NWC clients. If they stay under the openreceive organization, avoid
OpenReceive-branded repository names and keep the package APIs neutral.

OpenReceive backend SDKs consume these packages. They should not be required
for an application that only wants a raw NWC protocol client.

Each standalone NWC client README must start with raw NWC usage, not
OpenReceive usage. OpenReceive integration belongs in a later section so users
understand the protocol client is useful on its own.

Standalone NWC clients must not depend on OpenReceive SDKs, provider data,
checkout UI packages, demos, or product-specific state machines. Dependency
direction is one-way: OpenReceive depends on NWC clients, never the reverse.

Maintenance gate for third-party NWC clients:

  - Repository pushed within the last 12 months at the time we adopt it.
  - MIT, Apache-2.0, BSD, or similarly permissive license.
  - Supports NIP-47 info-event bootstrap or can be wrapped to support it.
  - Supports NIP-47 receive checkout methods: get_info, make_invoice, and
    lookup_invoice.
  - Supports pay_invoice for general standalone NWC client use.
  - Supports or can be extended to support payment_received notifications.
  - Handles NIP-04 encrypted wallet messages.
  - Supports or can be extended to support NIP-44 v2 negotiation.
  - Has tests or can pass OpenReceive's wallet-profile NWC conformance smoke
    suite.
  - Does not force browser/mobile exposure of NWC secrets.
  - Can run in ordinary server framework jobs, queue workers, cron tasks,
    scheduled/durable serverless runtimes with an explicit polling owner, or
    long-running worker roles.

Candidate maintenance snapshot checked on 2026-06-19:

  Ruby:
    Use MegalithicBTC/nwc-ruby initially. It was pushed 2026-06-13 and already
    targets Rails/server-side receive checkout. The OpenReceive Ruby SDK should
    wrap it first. If desired later, move, mirror, or supersede it as
    nwc-client-ruby while preserving package identity, compatibility, and author
    credit.

  JavaScript / TypeScript:
    Use getAlby/js-sdk for NWC initially. It was pushed 2026-05-23 and is the
    mature JS option for Nostr Wallet Connect. @openreceive/node should wrap the
    receive-side subset and hide send-payment methods from checkout APIs.

  Go:
    Audit rolznz/go-nwc first. It was pushed 2025-06-30, which satisfies the
    current recency gate but is close enough that it must be rechecked before
    adoption. If it lacks notification resilience, NIP-44 support, redaction,
    or conformance coverage, create github.com/openreceive/nwc-client-go and use
    rolznz/go-nwc only as a reference.

  Rust:
    Use rust-nostr/nostr initially. It was pushed 2026-06-15, has substantial
    adoption, and explicitly includes Nostr Wallet Connect support. The
    OpenReceive Rust crate should wrap the receive-checkout subset and provide
    Axum/Actix-friendly workers.

  Swift:
    Audit SparrowTek/NostrKit. It was pushed 2026-04-29 and has NWC support,
    but OpenReceive needs server-side Vapor suitability, Linux compatibility,
    notification behavior, and secret-redaction checks. If those do not pass,
    create nwc-client-swift with SwiftPM product NWCClient.

  PHP:
    Audit dsbaars/nostr-php-nwc. It was pushed 2025-07-07, so it currently
    passes the recency gate, but it is small and must prove receive checkout,
    notifications, PHP 8.2+/8.3+ compatibility, Laravel queue friendliness, and
    Symfony Messenger friendliness. Plan to write nwc-client-php unless the
    audit shows it is ready to adopt or collaborate on.

  Python:
    Write nwc-client-py. The known Python NWC candidate checked from the
    awesome-nwc list was pushed 2024-07-24, which is too stale for a primary
    dependency. The Python client should support asyncio first, with sync
    wrappers for Django/Flask where practical.

  Kotlin / JVM:
    Write nwc-client-jvm. There is no currently vetted Kotlin/JVM NWC
    client for server-side Ktor and Spring Boot. The package should expose
    coroutine APIs, Java-friendly facades, and worker examples for Spring
    scheduling and Ktor background jobs.

  .NET / C#:
    Write Nwc.Client. There is no currently vetted .NET NWC client for
    ASP.NET Core. The package should expose async Task APIs, hosted-service
    notification listeners, dependency injection, and SignalR-friendly events.

  Elixir:
    Write nwc_client. There is no currently vetted Elixir NWC client for
    Phoenix. The package should use OTP supervision for relay connections,
    GenServer/Broadway/Oban-friendly polling, and Phoenix PubSub events.

  Dart:
    Do not use Dart/Flutter as a live wallet-secret runtime for consumer apps.
    Flutter packages are checkout UI only. If a server-side Dart ecosystem
    becomes important, write nwc_client; currently observed Dart NWC packages
    were pushed in 2024 and should be treated as references only.

Minimum NWC client responsibilities:

  - Parse nostr+walletconnect URIs.
  - Support multiple relay parameters and optional lud16.
  - Validate and redact wallet pubkey, relay URLs, and client secret.
  - Connect to one or more Nostr relays over WebSocket.
  - Fetch and parse the wallet service kind 13194 info event.
  - Parse content capabilities, notifications tag, and encryption tag.
  - Choose NIP-04 or NIP-44 v2 according to wallet support.
  - Sign Nostr events correctly.
  - Construct request events as kind 23194 with p tag, optional expiration tag,
    and encryption tag when using NIP-44 v2.
  - Parse response events as kind 23195 with p tag, e tag, result_type, error,
    and result.
  - Encrypt/decrypt NWC request and response payloads.
  - Support NIP-04 and track NIP-44 v2 support.
  - Implement get_info.
  - Implement make_invoice.
  - Implement lookup_invoice.
  - Implement pay_invoice for standalone NWC users.
  - Optionally implement list_transactions for diagnostics.
  - Never expose pay_invoice through the OpenReceive receive-checkout surface.
  - Do not implement pay_keysend or hold-invoice methods for v0.1.
  - Subscribe to payment_received notifications where the language runtime and
    wallet support it.
  - Dedupe notifications by payment_hash.
  - Dedupe duplicate NIP-04/NIP-44 notifications.
  - Reconnect with bounded exponential backoff.
  - Support heartbeat/ping or equivalent dead-connection detection.
  - Persist or expose the last processed event timestamp/cursor where possible.
  - Treat notifications as hints that trigger lookup_invoice, not final proof.
  - Normalize settlement detection into the canonical OpenReceive state model.
  - Normalize wallet/service errors into canonical OpenReceive errors.
  - Enforce NWC metadata below 3900 serialized characters.
  - Provide safe polling helpers with the canonical cadence.
  - Provide idempotency guidance and helper APIs for framework adapters.
  - Provide shared conformance vectors and wallet-profile NWC smoke tests.

Standalone NWC client test scripts:

  Each standalone NWC client repository must include scripts or commands that
  mirror the nwc-ruby real-wallet testing style:

    test:vectors
      Run deterministic protocol, parsing, crypto, settlement, error, and
      redaction vectors without a live wallet.

    test:live:create-invoice
      Read OPENRECEIVE_NWC from a local gitignored env file, create a tiny
      invoice, print the BOLT11/QR, and redact secrets in all output.

    test:live:manual-payment
      Wait while a human pays the invoice, poll lookup_invoice with the
      canonical cadence, and exit successfully only after lookup proves
      settlement.

    test:live:notifications
      Subscribe to payment_received notifications where supported, trigger
      lookup_invoice after notification, and prove duplicate notifications do
      not double-settle.

  Live scripts must skip clearly when OPENRECEIVE_NWC is absent. They must not
  run on untrusted pull requests with secrets. They should be documented in
  each NWC client README because a real wallet, a human payer, and notification
  observation are part of proving the protocol client.

Framework storage, polling runners, and notification listeners:

  OpenReceive libraries should not own an external database service. Each
  framework adapter stores payment state in the host application's normal
  database and runs background work in the host application's normal job system.
  The adapter/package owns the polling runner and notification listener
  implementations and exposes the native way to start each one.

  Rails:
    ActiveRecord tables, generated ActiveJob/Solid Queue/Sidekiq/GoodJob
    polling jobs, `bin/rails openreceive:poll`, `bin/rails openreceive:listen`,

    listening.

  Laravel:

    scheduler, events, broadcasting, Reverb or Echo for passive frontend updates.

  Symfony:
    Doctrine entities, package poll/listen console commands, Messenger workers,

  Django:
    Django models, package poll/listen management commands, Celery/RQ/Huey task,

  FastAPI:
    SQLAlchemy/SQLModel examples, package asyncio poll/listen runners, Celery/Arq/RQ,
    SSE/WebSockets.

  Flask:
    SQLAlchemy examples, package poll/listen CLI commands, Celery/RQ workers,
    SSE/WebSocket extension examples.

  Node:
    Plain async storage interface first, package-owned Postgres and SQLite

    init/migrate/doctor/poll/listen commands, Prisma/Drizzle examples that wrap
    the package-owned schema rather than reinventing it, BullMQ/Bree workers,

    database/sql interfaces, package `RunSettlementPollingWorker(ctx, ...)`

Backend SDK API
---------------

Every backend SDK should expose the same conceptual API with ecosystem-native
syntax.

TypeScript shape:

  const openreceive = new OpenReceive({
    wallet: {
      type: "nwc",
      connectionString: process.env.OPENRECEIVE_NWC
    },
    pricing: {
      source: "coingecko",
      quoteTtlSeconds: 600
    }
  })

  const quote = await openreceive.rates.quoteFiatToMsats({
    fiat: {
      currency: "USD",
      value: "0.10"
    }
  })

  const invoice = await openreceive.invoices.create({
    amount_msats: quote.amount_msats,
    description: "Fruit sticker for user_123, banana",
    expiry: 600,
    metadata: {
      fruit: "banana",
      userId: "user_123"
    },
    idempotencyKey: "fruit-demo-user-123"
  })

  If an SDK exposes direct fiat invoice creation, it must use a separate input
  shape with fiat and no amount_msats, quote atomically, store fiat_quote, and
  reject requests that include both fiat and amount_msats.

  const current = await openreceive.invoices.lookup(invoice.invoice_id)

  await openreceive.invoices.waitForSettlement(invoice.invoice_id, {
    timeoutSeconds: 600
  })

  openreceive.events.on("invoice.settled", async (event) => {
    await unlockFruitSticker(event.invoice_id)
  })

Equivalent naming:

  JS/TS:   openreceive.invoices.create()
  Python:  client.invoices.create()
  Ruby:    client.invoices.create
  PHP:     $client->invoices()->create()
  Rust:    client.invoices().create(...)
  Go:      client.Invoices.Create(...)
  Swift:   client.invoices.create(...)
  Kotlin:  client.invoices.create(...)
  Dart:    client.invoices.create(...)
  C#:      client.Invoices.CreateAsync(...)
  Elixir:  OpenReceive.Invoices.create(...)

Backend SDK responsibilities:

  - Native NWC client or wrapper around a vetted recent NWC dependency.
  - Invoice create/lookup/wait APIs.
  - Price quote helper.
  - Provider route helper.
  - Event subscription helper.
  - Receive-checkout surface that never exposes spend methods.
  - Secret redaction.
  - Error normalization.
  - Settlement-state normalization.
  - Safe polling helper.
  - Notification listener helper where practical.
  - Idempotency-key helper.
  - Framework storage hooks.
  - Framework job/worker hooks.
  - Testkit integration.


Frontend SDK API
----------------

Frontend packages must never require wallet secrets.

Frontend packages do not create live invoices by themselves. They call the
merchant backend's OpenReceive routes, receive display-safe invoice data, and
listen for passive settlement updates from that backend.

They receive:

  invoice_id
  invoice
  payment_hash
  amount_msats
  transaction_state
  workflow_state
  expires_at
  checkout.events_url

  data:

    paymentHash
    routes

    selectedAsset

The view model is the real frontend API. Default components, headless
components, web components, and design-system recipes all consume the same view

Headless browser package:

  @openreceive/browser

Responsibilities:

  - Parse invoice object.
  - Connect to SSE events.
  - Track countdown.
  - Generate lightning: invoice URI payloads.
  - Generate safe QR SVG/PNG output with opaque background, high contrast, and
    at least a 4-module white quiet zone.
  - Copy invoice.
  - Open lightning: URI.
  - Load provider routes.
  - Emit invoice.settled/invoice.expired/error events.
  - Expose a framework-neutral checkout state machine.
  - Expose framework-neutral actions for copy, open-wallet, reload state,
    refresh expired invoice, select route, dismiss, retry, and cancel.

Frontend package philosophy:

  OpenReceive frontend packages should make it easy to build a polished payment
  wizard without forcing a design system. Teams must be able to use Mantine,
  Material UI, Chakra, Radix, Tailwind, Bootstrap, DaisyUI, shadcn-style
  components, Angular Material, PrimeNG, Vuetify, Svelte component libraries,
  or their own design system.

  The strategy is layered:

    1. @openreceive/browser
       Framework-neutral state machine, event handling, provider-route loading,
       route/wizard selection, country storage, theme preference helpers,
       countdowns, QR helpers, copy/open-wallet actions, watcher lifecycle for
       SSE/polling/countdown, shared backend lookup fetcher, shared checkout
       labels, wizard selection reducer/model helpers, country picker/map model
       helpers, shared theme model helpers, shared checkout icon assets/helpers,
       and typed data. No DOM ownership beyond optional helpers.

    2. Framework headless adapters
       React hooks/context, Vue composables, Svelte stores/actions, Angular
       services/signals/directives, and Solid signals. These expose state and
       actions but do not dictate markup.

    3. Accessible unstyled primitives
       Small components such as Root, Modal, OpenReceiveQRCode, CopyButton,
       RouteWizard, ProviderList, PaymentState, Countdown, and Success. They own
       behavior, ARIA, focus management, keyboard handling, and slot structure,
       but use minimal reset styles and accept class/style/slot/component
       overrides. The QR primitive owns the QR quiet zone and contrast defaults
       even when the surrounding UI is fully customized.

    4. Optional default UI
       A pleasant default checkout for demos, quickstarts, and teams without a
       design system. This is not the main customization surface, but it must
       be complete enough to show the real payment wizard, waiting state,
       countdown, copy/open-wallet actions, route/provider suggestions, and
       light/dark theme behavior without demo-local business logic.

    5. Design-system recipes
       Example integrations for Mantine, Material UI, Chakra, Tailwind/Radix,
       Bootstrap, Angular Material, Vuetify, and common Svelte UI libraries.
       These should usually be docs/examples, not hard dependencies or required
       packages.

Backend Framework Adapters
--------------------------

Each backend adapter should provide:

  - Route helper.
  - Invoice creation helper.
  - Event stream helper.
  - Auth middleware integration point.
  - Invoice access authorization hook.
  - Guest checkout/session-token support where idiomatic.
  - Payment model/table migration or documented storage interface.
  - Polling job/worker integration.
  - Notification listener integration where practical.
  - Fulfillment hook.
  - Middleware/security defaults.
  - Test helpers.
  - Demo-ready defaults.

Required adapter auth hooks:

  currentUser(request)
    Return the app's current user/account/customer object, or null for a guest
    checkout flow.

  authorizeInvoiceCreate(request, user, input)
    Decide whether this request may create an invoice.

  authorizeInvoiceAccess(request, user, invoice)
    Decide whether this request may read, refresh, poll, or subscribe to events
    for this invoice.

  buildInvoiceMetadata(request, user, input)
    Attach user_id, order_id, cart_id, tenant_id, checkout_session_id, or other
    app-owned identifiers needed for authorization and the app's private
    post-settlement behavior.

  eventAuthStrategy(request, invoice)
    Choose same-site session cookies, signed event URL, polling fallback, or a
    framework realtime bridge.

Adapters should fail closed when authorization hooks are missing in production
mode. Demos may use a simple signed guest checkout session so the flow works
without user accounts.

Every demo sells the same tiny product:

  Pay $0.10, unlock a fruit sticker.

  {
    "id": "fruit-sticker-pack",
    "name": "Fruit Sticker",
    "price": {
      "currency": "USD",
      "value": "0.10"
    },
    "sticker_files": ["apple.svg", "banana.svg", "orange.svg", "pear.svg"]
  }

  examples/hello-fruit/shared/stickers/apple.svg
  examples/hello-fruit/shared/stickers/banana.svg
  examples/hello-fruit/shared/stickers/orange.svg
  examples/hello-fruit/shared/stickers/pear.svg

User flow:

  1. User opens demo.
  2. User picks fruit if he wants or clicks the suggested fruit which is random
  3. Frontend calls backend: POST /api/checkout.
  4. Backend calls OpenReceive to create invoice for USD 0.10.
  5. Backend stores invoice_id, payment_hash, quote, product metadata.
  6. Frontend displays QR code, copy button, lightning wallet button,
     countdown, transaction/workflow state, and payment route wizard.
  7. Customer pays.
  8. Backend confirms with lookup_invoice.
  9. Backend emits invoice.settled event.
  10. Frontend flips to settled/success state.
  11. Backend serves signed one-time sticker URL.

Optional deterministic fruit:

  fruitIndex = parseInt(preimage.slice(0, 8), 16) % fruits.length

Demo modes:

  Live wallet test mode, Rizful profile:

    OPENRECEIVE_MODE=test_nwc
    OPENRECEIVE_WALLET=nwc
    OPENRECEIVE_WALLET_PROFILE=rizful
    OPENRECEIVE_NWC=nostr+walletconnect://...
    OPENRECEIVE_EXPECTED_CAPABILITIES=./expected_capabilities.json
    OPENRECEIVE_PRICE_SOURCE=static

  Production NWC mode:

    OPENRECEIVE_MODE=production
    OPENRECEIVE_WALLET=nwc
    OPENRECEIVE_NWC=nostr+walletconnect://...
    OPENRECEIVE_PRICE_SOURCE=coingecko

Every demo must support live wallet test mode with at least the Rizful profile.
Tests create real low-value invoices and settlement is confirmed by paying the
fruit-sticker checkout. That is acceptable for this project because the test
wallet is receive-only, the amounts are tiny, and the owner is paying
themselves. Rizful is the first required profile; the harness must allow
additional profiles such as Alby, Zeus, and custom self-hosted wallets without
changing the OpenReceive protocol contract.

    - Basic monitoring for uptime, disk, memory, container restarts, and tunnel
      health.
    - A clear rebuild path from git, GHCR, and host secret files.

  If using a WireGuard public-IP gateway, Cloudflare points at the gateway and
  the gateway forwards only HTTP/HTTPS traffic to the office demo proxy. Do not
  route arbitrary inbound traffic into the office LAN.

  If the office node loses power, internet, or the WireGuard path, public demos
  may go down. unlikiely becuase we have UPSs and bakcup power and multipl einternet connectsion.

Caddy ownership and routing:

  The demo node owns its own proxy. Caddy may bind 80/443 there because the
  node is not the private apex app host and does not host the critical private
  apps. Keep the private app's proxy out of the normal demo path.

  Prefer Caddy for the first office-node deployment because the route map is
  easy to read. Use Docker service-name routing from day one so adding demos
  does not require assigning and tracking host ports.

    demo-nodejs-whatever.openreceive.org {
      reverse_proxy demo-nodejs-whatever:3000
    }

  Put Caddy and demo containers on a shared external Docker network such as
  openreceive_demo_proxy. Each demo compose project joins that network and
  exposes its internal HTTP port only to the Docker network. Do not publish
  per-demo host ports for stable public demos.

  This shared proxy network lets demo containers reach each other by service
  name. That is acceptable for the first low-value demos because containers
  should hold only receive-only NWC credentials, static product data, and
  minimal invoice records, and they still must not reach the private LAN. Do
  not claim container-to-container isolation for this mode.

  If a demo later handles higher-value production mode, user uploads, private
  integrations, or sensitive merchant data, give that demo its own private
  network plus the proxy network, and restrict egress explicitly.

  Canonical routing rules:

    - Caddy is the only public container that publishes host ports 80 and 443.
    - Demo containers use expose, not ports, for their HTTP listener.
    - Demo service names are stable and match the demo slug where practical.
    - Caddy routes to Docker service names, not 127.0.0.1 host ports.
    - Do not use network_mode: "host" for the canonical demo fleet.
    - Do not assign per-demo host ports except in a gitignored local debug
      override.

  Keep one generated Caddyfile, or one generated imported Caddyfile fragment per
  demo, so deployment can add/update routes without hand-editing production
  config.

  Create the proxy network once on the demo node:

    docker network create openreceive_demo_proxy

  Keep the root Caddyfile small and import generated site fragments:

    import /etc/caddy/sites/*.caddy

  Local or preview demos can use Cloudflare Tunnel or VPN-only access. They are
  not stable public demos unless promoted to the dedicated office demo node with
  the requirements above.

  Docker-network Caddy avoids host port allocation and is the default:

    networks:

DNS plan:

  Use Cloudflare DNS. Add explicit proxied records for stable demos:

    django-demo.openreceive.org       -> WireGuard gateway IP or CNAME target
    express-demo.openreceive.org      -> WireGuard gateway IP or CNAME target
    fastapi-demo.openreceive.org      -> WireGuard gateway IP or CNAME target
    rails-react-demo.openreceive.org  -> WireGuard gateway IP or CNAME target

  For stable public demos, point records at the separate demo edge, not the
  private production edge. With the office-node
  plan, Cloudflare records point at the public WireGuard gateway IP or a stable
  CNAME that reaches that gateway; the gateway forwards only HTTP/HTTPS over
  WireGuard to Caddy on the office demo node. Choose one DNS convention and keep
  the inventory file as the source of truth.

  If using the office Ubuntu demo node, point Cloudflare records at the public
  IP or stable tunnel endpoint that reaches that node. Keep Cloudflare proxy
  enabled for public demos, and keep the WireGuard/tunnel details out of public
  docs.

  Do not point stable demo records at the private production edge. That path is
  for the private apex app, not the demo architecture.

  Do not create a broad *.openreceive.org wildcard for the demo fleet. The apex
  openreceive.org app is private and separate, and broad wildcard routing makes
  it too easy for docs, previews, app subdomains, or future private services to
  land on the wrong infrastructure.

  If preview deployments need wildcard DNS, scope it under a dedicated preview
  zone:

    *.preview.openreceive.org -> preview/demo edge IP

  For local-machine previews, prefer Cloudflare Tunnel or another explicit
  tunnel endpoint under preview.openreceive.org. Do not point public DNS
  directly at an ordinary home/dev machine. An isolated office Ubuntu demo node
  with stable internet, backup power, and WireGuard/public-IP routing is allowed
  as the separate demo node when it satisfies the office-node requirements.
  VPN-only demos are fine for private QA, but they are not public hosted demos.

  Keep these DNS targets separate:

    openreceive.org                 -> existing private app edge
    www.openreceive.org             -> existing private app edge, if used
    django-demo.openreceive.org     -> WireGuard demo edge
    express-demo.openreceive.org    -> WireGuard demo edge
    demos.openreceive.org           -> demo gallery route
    docs.openreceive.org            -> docs host/edge

TLS plan:

  Cloudflare is the public TLS edge for the current setup. Keep Cloudflare proxy
  enabled for stable public demo subdomains unless a specific demo needs DNS
  only for debugging.

  Origin TLS for demos belongs on Caddy on the office Ubuntu demo node.
  Cloudflare Full Strict accepts valid public CA certificates such as Let's
  Encrypt, but proxied Cloudflare DNS breaks the default Caddy ACME paths:
  TLS-ALPN-01 cannot reach Caddy because Cloudflare terminates TLS, and HTTP-01
  is fragile behind redirects, proxying, and Cloudflare-only origin firewalls.

  Canonical path:

    Use Caddy with the Cloudflare DNS module and issue/renew certificates via
    ACME DNS-01. Store a narrowly scoped Cloudflare DNS API token on the demo
    host, limited to the demo hostnames/zone permissions needed for TXT
    challenges.

  Acceptable alternatives:

    - Use a Cloudflare Origin CA certificate on Caddy for proxied-only demo
      origins.
    - Temporarily set a hostname to DNS-only for manual bootstrap, issue the
      certificate, then return it to proxied + Full Strict. This is a manual
      fallback, not the normal renewal plan.

  Keep the Cloudflare demo records proxied so browsers see Cloudflare's public
  edge certificate and Cloudflare connects onward to Caddy. Caddy's certificate
  is for the Cloudflare-to-origin leg. Manage Cloudflare DNS records manually in
  the Cloudflare dashboard for v0.1.

  Cloudflare cache and realtime rules:

    - Bypass cache for /openreceive/*, /api/*, /events, and text/event-stream.
    - Keep polling fallback even when SSE is enabled.
    - Send no-store on invoice/lookup/event responses.
    - Cache only static demo assets, docs, and screenshots.

  Recommendation:

    Keep Cloudflare for public DNS/TLS edge, but run public demos on a separate
    demo node from the start. Add explicit Cloudflare demo subdomain records and
    demo-node origin routes.

Deployment workflow:

  CI means a hosted automation runner, such as GitHub Actions, does the boring
  repeatable work after each push or release. For OpenReceive demos, CI should
  build and test images; the office demo node should only pull already-built
  images and restart containers.

  Start with the same deploy path both locally and in CI:

    local first:
      Run ./demos/deploy/scripts/deploy-demo express-demo from a trusted
      operator machine. The script SSHes to the office demo node, pulls the
      selected image, updates compose/Caddy config, reloads Caddy, and runs the
      smoke test.

    CI later:
      GitHub Actions runs tests, builds images, and pushes them to GHCR. A
      protected deploy step then runs the same deploy-demo or promote-demo
      script, either from an operator machine over SSH or from a self-hosted
      runner on the office demo node.

  Because the preferred demo node lives behind WireGuard, do not assume a
  GitHub-hosted runner can SSH directly to it. Start with manual promotion, or
  run a protected self-hosted GitHub Actions runner on the office demo node for
  deploy jobs only. GitHub-hosted runners can still do tests, package builds,
  image builds, and GHCR pushes.

  Safe CI/deploy split:

    GitHub-hosted runner:
      Validate code, run tests, build demo images, push immutable GHCR images.
      This runner does not need office network access.

    Office self-hosted runner or manual operator command:
      Pull tested GHCR image digests, update compose/Caddy config, reload
      Caddy, run smoke tests, and roll back if needed.

  Do not give untrusted pull requests access to the office self-hosted runner,
  WireGuard keys, deploy SSH keys, or NWC secrets.
  Deployment workflows must be restricted to protected branches, tags,
  workflow_dispatch, or a protected GitHub environment.

  Self-hosted runner hard rules:

    - Never mount the host Docker socket into an untrusted or broad CI runner.
      Docker socket access is effectively root on the host.
    - Run any deploy runner in a locked-down VM, container boundary, or separate
      deploy user with minimal SSH/deploy rights.
    - Keep build/test runners separate from deploy runners.
    - Do not store NWC secrets in the same environment that builds untrusted
      code.
    - Do not let pull_request workflows from forks or untrusted branches reach
      deploy credentials, WireGuard keys, GHCR write tokens, or host secrets.

  - GitHub Actions builds demo images.
  - Images are tagged with git SHA and optionally latest for stable branch.
  - Images are pushed to GHCR.
  - A deploy job or operator command runs on the office demo node, or reaches it
    through a trusted SSH path.
  - The host pulls updated compose files or receives a generated deploy bundle.
  - The host runs docker compose pull and docker compose up -d for changed demos.
  - Caddy config is generated or updated from the demo registry.
  - Caddy is reloaded after config validation succeeds.
  - The Cloudflare DNS inventory is checked against the demo subdomain registry.
  - A smoke test hits /healthz and the public checkout URL.
  - Failed smoke tests roll back to the previous image tag.

  Minimal GitHub Actions mental model:

    on pull request:
      run tests and build changed demos. Run live-wallet smoke tests only where
      the trusted test secret is available; do not deploy stable demos

    on merge to main:
      run tests, build changed demos, push GHCR images, optionally deploy
      live-wallet test demos after smoke tests pass

    on tagged release:
      publish packages, rebuild affected demos, promote tested image digests to
      stable demo subdomains

Deployment churn policy:

  Expect SDK packages and demos to change often. The demo platform must make
  frequent updates cheap without turning the demo node into a hand-managed build
  machine.

  Use two deployment lanes:

    preview lane:
      Every pull request or branch may build changed demos with noindex and
      short-lived preview URLs. Trusted previews may use the live-wallet test
      NWC secret; untrusted previews build without NWC smoke tests. Preview
      deploys can use local tunnels, VPN-only access, or
      preview.openreceive.org. They do not imply that a stack is ready for the
      stable public demo list.

    stable lane:
      Merges to main build immutable demo images tagged by git SHA and image
      digest. Stable public demo subdomains are promoted to a tested image
      digest after smoke tests pass. Live-wallet test demos may auto-promote once
      the pipeline is reliable; production-wallet demos require explicit
      manual approval.

  Package update rules:

    - A package release or SDK change should trigger rebuilds for every demo
      that consumes that package.
    - Demos should pin package versions or workspace references explicitly, so
      a public demo always reports the OpenReceive/NWC client versions it is
      running.
    - Each demo should expose a non-secret build metadata endpoint or footer
      containing demo git SHA, image digest, SDK package versions, mode, and
      deployed_at.
    - Dependabot/Renovate-style package update PRs are acceptable, but they must
      run the same demo smoke tests as human-authored package changes.
    - Do not install or compile packages interactively on the demo node. Build
      images in CI, push to GHCR, then pull immutable images on the demo node.

  Deploy safety rules:

    - Deploy one demo independently without redeploying the whole fleet.
    - Use a per-demo deploy lock so two CI jobs cannot update the same compose
      project at the same time.
    - Write a deployment manifest with demo slug, image digest, package
      versions, git SHA, config mode, deployed_at, and previous image digest.
    - Run smoke tests after each deploy and automatically roll back to the
      previous image digest on failure.
    - Keep the proxy config stable; adding or updating a demo should usually be
      a compose-label change plus a Cloudflare DNS registry change.
    - Batch large demo-matrix updates during a maintenance window if they touch
      many language runtimes at once.

Receive-only NWC secrecy:

  A receive-only NWC code cannot spend funds, but it is still a credential. It
  can create invoices, inspect invoice/payment metadata, and potentially spam
  your wallet or leak business activity. Treat it like an API key.

  Rules for OpenReceive itself:

    - Never commit OPENRECEIVE_NWC.
    - Never include OPENRECEIVE_NWC in frontend bundles.
    - Never expose OPENRECEIVE_NWC in HTML, JS, source maps, screenshots, logs,
      telemetry, error pages, or health endpoints.
    - Always redact nostr+walletconnect:// URIs in logs.
    - Always provide .env.example files with placeholder values.
    - Every framework demo should include an automated check that the built
      frontend assets do not contain nostr+walletconnect:// or OPENRECEIVE_NWC.

  Rules for OpenReceive users:

    - Put receive-only NWC in a server-side environment variable.
    - Do not put it in React/Vue/Svelte/Angular public env vars.
    - Do not put it in mobile apps.
    - Do not paste it into support tickets, GitHub issues, screenshots, or
      analytics tools.
    - Rotate it if it was committed, logged, or shared.
    - Use a separate receive-only NWC code per app or environment when possible.

Test Strategy
-------------

Testing has to be more serious than the number of libraries. The tests are the
thing that keep the project from becoming a pile of similar but incompatible
SDKs.

Test layers:

1. Schema validation

  - v0.1: validate invoice, invoice storage lifecycle, payment event, rate
    quote, and error JSON Schemas.
  - Later: validate provider registry, supported currencies, demo product data,
    OpenAPI, and AsyncAPI.

2. Unit tests

  - fiat-to-msat conversion
  - decimal rounding
  - invoice state transitions
  - NWC URI parsing
  - secret redaction
  - provider route selection, later
  - event parsing
  - QR payload, quiet-zone, and contrast helpers
  - idempotency key handling

3. Shared test vectors

  - fiat-to-msats
  - amount field safe-integer boundaries
  - description vs description_hash validation
  - invoice lifecycle transitions
  - settlement detection
  - polling backoff
  - refresh creates a linked replacement invoice
  - provider routing, later
  - NWC URI parsing
  - NWC info event parsing
  - NWC encryption negotiation
  - NWC request/response event shape
  - error normalization
  - idempotency behavior
  - idempotency key reuse with different request body returns 409
  - metadata size limit

4. Deterministic mock wallet

  Post-v0.1. Do not block the first Express payment path on this service.

  Location:

    tools/mock-wallet

  Purpose:

    Provide a deterministic NWC-compatible wallet/service for conformance tests
    before any SDK talks to a real wallet. The mock wallet does not replace
    live wallet profile tests; it makes schema, vector, error, duplicate-event,
    replay, expiration, idempotency, and notification behavior reproducible.

    The mock wallet does not generate payable Lightning invoices. It may return
    invoice fixtures: fake or syntactically valid-but-unpayable BOLT11-like
    strings with deterministic payment_hash, amount, expiry, and description.
    Its job is to prove OpenReceive contract behavior, not Lightning routing or
    real wallet compatibility.

  Useful for testing:

    idempotency behavior
    database lifecycle transitions
    duplicate notifications
    replay safety
    settlement detection logic
    polling cadence
    refresh creates-new-invoice behavior
    frontend SSE/WebSocket updates
    UI state changes after scripted payment
    SDK conformance against fixed vectors

  Not useful for proving:

    real wallets accept the NWC connection
    real BOLT11 invoice creation works
    payment actually routes over Lightning
    Rizful, Alby, Zeus, or other wallet notification behavior works
    NIP-04/NIP-44 compatibility with real wallets
    the human payment flow works

  Required behavior:

    parse test NWC URIs
    expose get_info, make_invoice, and lookup_invoice
    return deterministic invoice fixtures from make_invoice
    emit payment_received notifications from scripted fixtures
    support duplicate notification replay
    support expired, failed, pending, and settled invoice fixtures
    support NIP-04 compatibility fixtures
    support NIP-44 fixtures once the reference implementation supports them
    reject metadata over 3900 serialized characters
    provide deterministic errors and timing controls
    never require real wallet funds

5. Live wallet NWC test harness

  Location:

    tools/live-nwc-test

  Required behavior:

    parse OPENRECEIVE_NWC from a gitignored env file
    parse OPENRECEIVE_WALLET_PROFILE=rizful|alby|zeus|custom
    load expected_capabilities.json when present
    run automatic preflight before invoice creation
    create a low-value fruit-sticker invoice
    print QR/copyable invoice for a human operator to pay
    look up the invoice before payment
    poll with the canonical cadence
    subscribe to notifications where the client supports it
    confirm settlement only through lookup_invoice
    redact the NWC string in every log and error path
    skip live NWC smoke tests clearly when OPENRECEIVE_NWC is not present

  Required live scenarios:

    invoice created through the configured wallet profile
    preflight capability summary recorded without secrets
    settled by polling after the owner manually pays the invoice
    settled by notification plus lookup after the owner manually pays the
    invoice when notifications are available
    duplicate idempotency key
    rate source failure using the static price provider
    receive-only NWC success
    multiple relay URI parsing
    missing encryption tag falls back to NIP-04
    NIP-44 advertised and selected when the wallet profile advertises it
    metadata over 3900 serialized characters rejected before NWC request

  The deterministic mock wallet covers forced timeouts, forced notification
  loss, duplicate notifications, and scripted settlement. Live wallet profile
  tests prove real payable invoice creation, real wallet compatibility,
  notification behavior, Lightning settlement, and the human payment path.

6. Conformance tests

  Every backend SDK and every standalone NWC client must pass the same shared
  vectors plus the same wallet-profile NWC smoke flow when OPENRECEIVE_NWC is
  available. Third-party NWC dependencies must be wrapped by an OpenReceive
  adapter that passes that same suite.

7. Demo E2E tests

  Use Playwright for web demos:

    open demo
    create invoice
    see QR with white quiet zone and copy button
    see provider wizard, later
    owner pays the live-wallet-profile invoice
    UI changes to settled
    sticker download appears

  Mobile demos need platform-specific smoke tests once they exist.

8. Security tests

  - Secret scanning.
  - Ensure NWC strings are redacted in logs.
  - Ensure frontend bundles do not contain OPENRECEIVE_NWC.
  - Ensure frontend bundles do not contain nostr+walletconnect://.
  - Ensure source maps do not contain OPENRECEIVE_NWC.
  - Ensure mounted framework APIs reject missing/invalid auth when auth is
    configured.
  - Ensure one user/session cannot read or subscribe to another user's invoice.
  - Ensure signed event URLs expire and are scoped to one invoice.
  - Ensure guest checkout tokens are unguessable and scoped to the checkout
    session.
  - Ensure settlement actions cannot be triggered by client-only state changes.

9. Package smoke tests

  For every publishable package:

    install from local build artifact
    import package
    run minimal create/lookup against the configured NWC profile when
    OPENRECEIVE_NWC is
    available
    run provider data import after provider-data packages exist

Security Model
--------------

Threats:

  - Merchant accidentally exposes NWC secret to browser.
  - Merchant commits a receive-only NWC code to git.
  - Merchant stores NWC secrets in GitHub Actions unnecessarily.
  - Read/write NWC leaks and can spend funds.
  - Duplicate notification double-credits a customer.
  - Client forges "settled" state.
  - Old invoices keep polling forever and hit rate limits.
  - Provider registry makes stale or overconfident claims.
  - Public demo is abused for spam or invoice generation.
  - Logs, errors, screenshots, or telemetry leak wallet details.

Controls:

  - Receive-only NWC default.
  - Automatic wallet preflight before live checkout; no manual compatibility
    ceremony or central wallet registry required.
  - Checkout APIs never expose send-payment methods.
  - .env.example templates committed; real env files ignored.
  - Host-side secret files stored outside the repo with chmod 600.
  - Secret redaction utility in every SDK.
  - Build-time frontend bundle scan for OPENRECEIVE_NWC and
    nostr+walletconnect://.
  - Backend-only settlement verification.
  - Idempotent settlement action boundary.
  - Unique payment_hash constraints.
  - Unique invoice_id constraints.
  - Scoped idempotency_key uniqueness and request-body hash checks.
  - Bounded polling.
  - Provider registry linter.
  - Demo rate limits.
  - Secret scanning CI.
  - Security docs from day one.

Public demo safety:

  - Low invoice amount.
  - Rate limit by IP/session.
  - Optional captcha only if abused.
  - No user-uploaded post-settlement content.
  - No dynamic file paths.
  - Daily receive cap.
  - Clear "test payment" disclosure.
  - Separate demo wallet.
  - Logs without secrets.

Important Design Decisions To Make Early
----------------------------------------

1. First NWC client adoption decisions

  Good candidates:

    Ruby:
      Use MegalithicBTC/nwc-ruby first. It is current, server-side, and already
      models the Rails worker/listener shape OpenReceive wants.

    JavaScript/TypeScript:
      Use getAlby/js-sdk first. It is current, mature, and close to the first
      Node/React reference stack.

    Go:
      Audit rolznz/go-nwc before depending on it. If it falls short on
      notifications, NIP-44, tests, or maintenance, write nwc-client-go.

    Rust:
      Use rust-nostr/nostr first and wrap only the receive-checkout surface.

    PHP, Python, JVM, .NET, Elixir:
      Plan to write standalone NWC clients unless a fresh, audited,
      permissively licensed client passes the full conformance suite.

  Recommendation:

    Start the native reference stack in TypeScript/Node for v0.1.0 because the
    first frontend and backend demo material is closest to that ecosystem. Use
    Ruby/Rails immediately after to prove the non-JS story with nwc-ruby. Build
    Python and PHP NWC clients next because those ecosystems are important and
    cannot depend on a stale or uncertain NWC layer.

2. First source of truth for NWC

  Recommendation:

    Use proven NWC libraries where possible for early delivery, wrap them behind
    OpenReceive interfaces, and keep protocol behavior covered by shared
    vectors plus live-wallet-profile NWC smoke tests. Do not hand-roll Nostr
    encryption unless the existing library path blocks the project.

3. Hosted test demo policy

  Recommendation:

    Hosted public demos default to live wallet test mode with the Rizful profile
    first. These are real mainnet test purchases for tiny sticker amounts,
    which is acceptable because the owner is paying themselves. There is no
    manual compatibility ceremony, but every SDK must run automatic preflight,
    fail if make_invoice or lookup_invoice are unavailable, warn on spend
    capability, and record a non-secret capability summary.

4. Provider registry claims

  Recommendation:

    Treat the provider registry as maintained data with evidence, verification
    dates, and conservative wording. The wizard should help users route a
    payment, not promise availability.

5. Package naming

  Recommendation:

    Reserve names early, but publish only the v0.1 packages that are actually
    tested.

   npm run check:release, and npm run test:ci.
9. Do not add new SDK families, demos, ecommerce integrations, mobile kits, or
   provider-data variants until the current release surface is easier to
   install, diagnose, package, and document.


Acceptance Checklist
--------------------

This plan is complete enough when these statements are true:

  - There is one product contract, not one per language.
  - Provider data is canonical and validated after the post-v0.1 data phase.
  - Price conversion is canonical and tested.
  - Polling behavior is canonical and tested.
  - Settlement detection is canonical and tested.
  - A normal server app can run the whole flow without an extra OpenReceive
    daemon.
  - The first JS packages prove the SDK shape.
  - Rails proves the framework-native worker/listener shape.
  - Hello Fruit proves the UX.
  - Live wallet profile mode exercises the real wallet path for demos and
    tests, with Rizful as the first profile.
  - Production NWC mode is possible without changing app code.
  - The frontend never sees wallet secrets.
  - Pure frontend live checkout is rejected by docs, examples, and tests.
  - Frontend packages are headless-first; design-system recipes come after the
    first Express path is green.
  - Framework adapters integrate with app-owned auth and fail closed when
    invoice access authorization is missing in production.
  - Every new SDK has a conformance gate.
  - Every standalone NWC client has a conformance gate.
  - Every new demo uses the same shared product data.
  - CI prevents generated/data drift.


Final Mental Model
------------------

OpenReceive should not become a loose pile of payment SDKs.

It should be:

  one spec,
  one live-wallet profile harness,
  one deterministic mock wallet after the first payment path is green,
  one provider registry,
  one price model,
  one test suite,
  one fruit-sticker demo,
  many idiomatic backend SDKs,
  many UI packages,
  enough standalone NWC clients to make the major server frameworks work
  without extra infrastructure.

That gives the project broad ecosystem coverage without drowning in duplicated
logic.
