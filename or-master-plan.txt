OpenReceive Master Plan
=======================

Purpose
-------

OpenReceive is an open source receive-payments kit for websites, server apps,
mobile apps, and native clients. Its job is intentionally narrow:

1. Let a merchant create one Bitcoin Lightning BOLT11 invoice.
2. Let the customer pay that invoice from Lightning, on-chain BTC, crypto,
   fiat rails, wallets, exchanges, or swap services.
3. Let the merchant backend verify settlement.
4. Let the app unlock content, credits, files, memberships, stickers, or
   other products only after verified settlement.

OpenReceive is not a bank, exchange, payment processor, or wallet backend.
The merchant brings a wallet they control through Nostr Wallet Connect (NWC /
NIP-47), such as Rizful, Alby Hub, LNbits, Zeus, or a self-hosted wallet that
follows the spec. OpenReceive libraries create and track receive-only invoices
inside the merchant's own server-side app, job system, or API backend.

The main architecture:

  spec/                  = the truth
  packages/nwc-*         = standalone neutral NWC protocol clients where needed
  packages/*             = idiomatic SDKs, framework adapters, and UI kits
  examples/*             = proof it works in real stacks
  demos/*                = hosted public deployments
  tools/*                = codegen, conformance, data validation, release

The strongest rule is that the hard behavior must exist once and be tested
everywhere. Provider data, schemas, price conversion rules, polling cadence,
payment-state semantics, and demo assets must not be copied by hand into each
language. They live in specs, shared data, generated types, and test vectors.

Brand And Naming
----------------

Use the product name OpenReceive.


The product, packages, docs, org, domains, and public assets should use the
correct spelling: OpenReceive / openreceive.

Recommended public names:

  GitHub org:        openreceive
  Main repo:         openreceive/openreceive
  Apex/private app:  openreceive.org, private Rails/Kamal app, out of scope
  Docs:              docs.openreceive.org
  Hosted demos:      demos.openreceive.org
  Playground:        play.openreceive.org
  Registry CDN:      registry.openreceive.org
  Demo subdomains:   django-demo.openreceive.org, express-demo.openreceive.org
  Runtime model:     no required OpenReceive daemon; run inside the app backend

If the misspelled domain was purchased, keep it only as a redirect.

Existing openreceive.org app:

  openreceive.org is already a private Rails app deployed with Kamal behind
  Cloudflare and an existing Hetzner Load Balancer. That app lives in a private
  repo and must stay completely outside this open source project.

  Hard boundary:

    - Do not put open source demo code in the private Rails app.
    - Do not put demo deployment config in the private Rails repo.
    - Do not require the private Rails app to route, proxy, or host demos.
    - Do not attach demo containers to the private app's Kamal deployment.
    - Do not put private landing pages, SEO pages, product pages, admin code,
      analytics code, or app-specific business logic in the public repo.
    - Do not require the public repo to contain the private site's renderer,
      CMS, routes, or import job implementation.

  The open source project may use subdomains such as docs.openreceive.org,
  demos.openreceive.org, and django-demo.openreceive.org, but those should be
  deployed as separate services with separate DNS records, compose projects,
  environment files, and source control from the private apex Rails app. They
  must not share app-virginia for stable public demos because that host runs
  critical production apps. The private app can link to demos manually if
  desired, but it is not part of their deployment path.

  Documentation is the one intentional bridge: canonical technical docs live
  in the public GitHub repo as markdown/data, and the private openreceive.org
  app may import or render those docs. The import job, SEO presentation,
  landing pages, and private site code stay in the private repo.


Product Definition
------------------

OpenReceive should make this promise:

  "Add Bitcoin Lightning receive payments to any app with a tiny backend
  integration and a polished frontend checkout."

The core product must support:

  - Creating a BOLT11 invoice.
  - Looking up invoice settlement.
  - Safely polling until settled or expired.
  - Listening to NWC notifications as an accelerator.
  - Converting fiat amounts to millisatoshis.
  - Locking a quote at invoice creation.
  - Showing the customer QR/copy/open-wallet UI.
  - Showing route suggestions for paying from other assets.
  - Pushing passive settlement updates to the frontend.
  - Unlocking fulfillment only after backend verification.

The reference flow:

1. Merchant creates or obtains a receive-only NWC connection string.
2. Backend stores that NWC string server-side only.
3. Backend SDK calls make_invoice through a native NWC client.
4. Backend stores invoice_id, payment_hash, invoice, amount_msats, expiry,
   quote, metadata, and NIP-47 state.
5. Frontend receives only non-secret invoice display data.
6. Frontend renders QR, copy button, lightning URI, countdown, payment routes,
   and state.
7. Backend listens for notifications where practical and also polls
   lookup_invoice through a background job, queue worker, scheduled task, or
   long-running framework worker.
8. Backend confirms settlement by lookup_invoice result with settled_at,
   preimage, or state == "settled".
9. Backend emits payment_received to the matching browser session.
10. Backend fulfills the product.

There is no pure-frontend OpenReceive checkout mode. A browser app, mobile app,
static site, or desktop shell can render the checkout UI, but invoice creation,
NWC access, settlement lookup, polling, notification handling, and fulfillment
must always happen in a server-side component controlled by the merchant.


Non-Negotiable Rules
--------------------

Security and correctness:

  - NWC secrets never go in browser code.
  - NWC secrets never go in mobile apps shipped to users.
  - Pure frontend apps are not supported for live checkout.
  - Static sites must call a serverless function, small API, CMS/plugin
    backend, or merchant server before showing an invoice.
  - Receive-only NWC is the default and recommended mode.
  - OpenReceive receive-checkout APIs never send payments.
  - Standalone nwc-client-* protocol libraries may support pay_invoice for
    general NWC users, but OpenReceive checkout adapters hide or refuse send
    methods by default.
  - The frontend never unlocks a product by itself.
  - Notifications are never final proof by themselves.
  - Settlement truth comes from lookup_invoice but also nwc notifications can provide the same data faster.
  - Store payment_hash for every invoice.
  - Store preimage when available.
  - Use idempotency keys for invoice creation.
  - Use exact decimal/fixed-point money math, not floating point.
  - Stop polling when invoice expiry is reached.
  - Redact wallet secrets from logs, errors, telemetry, and screenshots.
  - Public demos use rate limits and low invoice amounts.

Architecture:

  - One canonical product contract.
  - One canonical provider registry.
  - One canonical price conversion algorithm.
  - One canonical payment state machine.
  - One canonical Rizful receive-only test wallet policy.
  - One canonical fruit-sticker demo product.
  - One conformance suite used by every SDK.
  - No required OpenReceive daemon, sidecar, gateway, or extra service.
  - Every backend SDK owns native NWC integration directly or through a
    recently maintained NWC client dependency.
  - Generated types are allowed; duplicated business rules are not.

Release strategy:

  - Build one excellent end-to-end flow before broad package expansion.
  - Start with native SDK mode for real framework developer experience.
  - Build or adopt NWC clients per language, but only adopt third-party clients
    that are recently maintained, permissively licensed, and pass conformance.
  - Do not publish 20 half-working libraries before the spec is stable.


SDK-First Implementation Model
------------------------------

OpenReceive has one primary implementation model:

  The app backend talks directly to NWC through a language-native OpenReceive
  SDK, which either wraps a vetted NWC library or depends on a standalone
  neutral NWC client maintained by this project.

    Rails app          -> openreceive-ruby      -> nwc-ruby / NWC wallet
    Laravel app        -> openreceive-php       -> nwc-client-php -> NWC wallet
    FastAPI app        -> openreceive-py        -> nwc-client-py  -> NWC wallet
    Node app           -> @openreceive/node     -> @getalby/sdk       -> NWC wallet
    Go app             -> openreceive-go        -> audited go-nwc or nwc-client-go -> NWC wallet
    Swift Vapor API    -> OpenReceiveVapor      -> NostrKit or NWCClient -> NWC wallet
    iOS app            -> Node/Express payment API -> NWC wallet

The important product decision: OpenReceive should feel like installing a gem,
Composer package, npm package, PyPI package, SwiftPM package, crate, module, or
NuGet package into the app the developer already deploys. It should not ask a
Rails, PHP, Python, Go, Swift, or .NET developer to operate an extra daemon.

Frontend and mobile packages are UI clients only:

    Browser UI      -> merchant payment backend -> NWC wallet
    iOS/Android app -> merchant payment backend -> NWC wallet
    Static site     -> serverless/API backend   -> NWC wallet

The frontend/mobile decision is only: which backend endpoint do I call? The
server-side implementation is a separate choice owned by the merchant backend.

Inside that backend, the developer may use:

  Node/Express API    -> @openreceive/node     -> @getalby/sdk
  Rails app           -> openreceive-rails     -> nwc-ruby
  FastAPI app         -> openreceive-fastapi   -> nwc-client-py
  Laravel app         -> openreceive-laravel   -> nwc-client-php
  Vapor API           -> OpenReceiveVapor      -> NWCClient

Those implementation details never cross into browser, iOS, Android, Flutter,
or React Native apps. Client apps only receive invoice display data and passive
settlement state updates.

Default iOS demo choice:

  Build the first iOS demo as SwiftUI app -> Node/Express payment API -> NWC
  wallet. Internally, that API uses @openreceive/node and @getalby/sdk.
  Node/Express is common for mobile API backends, aligns with the v0.1
  OpenReceive stack, and avoids blocking the iOS UI demo on the Swift/Vapor
  server adapter. Keep ios-swiftui-vapor-api as the later Swift-native backend
  proof.

Default build strategy:

  - Build the spec, Rizful-backed NWC test harness, conformance smoke suite,
    and one excellent native Node/React stack first.
  - Build Ruby/Rails next using nwc-ruby as the model for in-app background
    workers and notification listeners.
  - Add neutral standalone NWC clients where the target language lacks a fresh,
    dependable client.
  - Keep conformance identical across all backend SDKs and NWC clients.


Canonical HTTP Contract
-----------------------

The HTTP API is the application-facing contract exposed by framework adapters,
demos, and merchant backends. OpenReceive does not ship a required HTTP daemon;
instead, adapters mount these routes inside Rails, Express, FastAPI, Django,
Laravel, Spring, ASP.NET Core, Phoenix, Vapor, and similar apps.

This contract is still valuable because it keeps frontend checkout components,
mobile apps, demos, docs, tests, and server adapters aligned. A React checkout,
SwiftUI checkout, static-site widget, or mobile app should be able to talk to
the merchant's backend using the same invoice/state/event shapes no matter
which server framework created the invoice.

NWC naming policy:

  The canonical OpenReceive HTTP payload should mirror NIP-47 field names as
  closely as practical. Use invoice for the BOLT11 string, expiry for seconds
  from invoice creation, state for the NIP-47 transaction state, and
  payment_hash, description, description_hash, metadata, created_at,
  expires_at, and settled_at exactly as NIP-47 names them.

  Intentional naming exception: OpenReceive uses amount_msats instead of NIP-47
  amount in its public HTTP and SDK payloads. In NIP-47, amount means
  millisatoshis, but in merchant APIs amount is too easy to confuse with fiat,
  sats, or display money. NWC adapters map amount_msats to NIP-47 amount when
  sending requests and map NIP-47 result.amount back to amount_msats.

  Any OpenReceive payload that carries satoshi or millisatoshi data must use
  explicit unit names: amount_sats or amount_msats. Do not use amount,
  amount_sat, or amount_msat outside raw NIP-47 request/response handling.

  OpenReceive-specific fields are allowed only where they describe app-layer
  behavior that NIP-47 does not model: invoice_id, fiat, fiat_quote, checkout,
  fulfillment, routes, auth, and idempotency. Public frontend responses may
  hide sensitive fields such as preimage and expose preimage_present instead.

  Do not copy the encrypted NWC event envelope into HTTP responses. HTTP routes
  already name the operation, so method, params, and result_type are not needed
  unless a package is implementing a raw NWC proxy/debug tool.

Authentication and authorization:

  OpenReceive does not define a standalone user-auth system. Framework adapters
  mount into the merchant's app and must integrate with the app's existing auth
  model: Rails sessions/Devise, Laravel auth middleware, NextAuth/Auth.js,
  Express middleware, Django auth, Phoenix plugs, ASP.NET policies, or a
  custom guest-checkout/session-token system.

  The app decides who can create an invoice, who can read an invoice, who can
  subscribe to events, and who receives fulfillment. OpenReceive provides
  hooks, helpers, and safe defaults; it does not decide what "logged in",
  "owns this order", "guest checkout", or "admin" means.

  Typical endpoint policy:

    POST /openreceive/v1/invoices
      Protected by app auth, cart/session token, or guest checkout token.

    GET /openreceive/v1/invoices/{invoice_id}
      Allowed only when the invoice belongs to the current user, cart, order,
      or checkout session.

    GET /openreceive/v1/invoices/{invoice_id}/events
      Uses same-site session cookies, a short-lived signed event URL, or a
      polling fallback when EventSource auth is awkward on the platform.

    GET /openreceive/v1/routes, /providers, /rates
      Usually public/cacheable but still rate-limited.

Base path:

  /openreceive/v1

Core endpoints:

  POST   /openreceive/v1/invoices
  GET    /openreceive/v1/invoices/{invoice_id}
  POST   /openreceive/v1/invoices/lookup
  GET    /openreceive/v1/invoices/{invoice_id}/events
  POST   /openreceive/v1/invoices/{invoice_id}/refresh
  GET    /openreceive/v1/rates
  POST   /openreceive/v1/rates/quote
  GET    /openreceive/v1/routes
  GET    /openreceive/v1/providers
  GET    /openreceive/v1/health
  GET    /openreceive/v1/capabilities

QR codes:

  QR rendering is not part of the canonical backend invoice response. The
  backend returns the BOLT11 invoice string in invoice, and OpenReceive
  frontend/mobile packages generate safe QR codes from that string. This is how
  React, Vue, Svelte, Angular, SwiftUI, Android Compose, Flutter, and
  web-component checkouts should work.

  OpenReceive must not rely on application developers to remember QR scanning
  details. Every official QR helper, primitive, component, and downloadable PNG
  renderer must use safe defaults:

    payload: lightning:<invoice>
    background: opaque white
    foreground: black or another tested high-contrast color
    quiet zone: at least 4 modules, white, included inside the rendered image
    error correction: conservative default suitable for Lightning invoices
    exported PNG/SVG: includes the same quiet zone and opaque background

  Frontend packages should expose both:

    OpenReceiveQRCode / equivalent framework component
    createQrSvg(), createQrPng(), and createLightningUri() helpers

  Developers may wrap, size, place, brand, or download the QR output, but the
  OpenReceive renderer owns the payload, quiet zone, and contrast defaults.

  Framework adapters may offer optional QR helpers for no-JavaScript fallback
  pages, server-rendered emails, PDFs, or unusual CMS/plugin environments, but
  those helpers are not required OpenReceive endpoints, should not be included
  in the canonical invoice response, and must share the same QR safety defaults
  and tests as the frontend helpers.

Lookup invoice request:

  Frontend checkout flows normally use invoice_id because it is scoped to the
  merchant app's order/session/auth model:

    GET /openreceive/v1/invoices/or_inv_01jz...

  Backend SDKs and server-to-server code may also use a NIP-47-shaped lookup
  request where the framework auth policy allows it:

    POST /openreceive/v1/invoices/lookup

    {
      "payment_hash": "abc123..."
    }

  The lookup request may use payment_hash or invoice, matching NIP-47
  lookup_invoice params. One of payment_hash or invoice is required.

Payment State Machine
---------------------

All SDKs, examples, database rows, UI components, and event payloads should
preserve NIP-47 transaction state names wherever they describe the invoice
itself.

Canonical transaction states:

  pending
    Customer has not paid yet.

  settled
    Wallet lookup proves settlement.

  expired
    Invoice expired before settlement.

  failed
    Wallet, provider, network, validation, or internal error occurred.

  accepted
    Reserved for hold-invoice semantics. OpenReceive v0.1 does not create hold
    invoices, but clients should not choke on this NIP-47 state.

OpenReceive workflow states:

  These are app-layer states, not replacements for NIP-47 state. Use them only
  in local storage, UI state machines, or nested objects such as fulfillment.

  draft
    App has requested a payment intent, but no BOLT11 invoice exists yet.

  invoice_created
    Wallet returned an invoice and payment_hash.

  checking
    A notification or poll suggests activity; server is verifying by lookup.

  fulfilled
    App delivered the product, file, credit, pass, or entitlement.

  cancelled
    Merchant/app intentionally cancelled the payment flow.

Typical workflow transitions:

  draft -> invoice_created
  invoice_created -> checking
  invoice_created -> cancelled
  checking -> invoice_created
  checking -> expired
  checking -> failed
  transaction state settled -> fulfillment state fulfilled

Invariant:

  Product fulfillment is never triggered by frontend state alone. Fulfillment
  happens only after backend verification through lookup_invoice. For NIP-47,
  prefer settled_at, preimage, or state == "settled"; normalize older
  library-specific variants at adapter boundaries only.


NWC Protocol Foundation
-----------------------

NWC URI shape:

  nostr+walletconnect://<wallet-pubkey>?relay=<wss-url>&secret=<client-secret>

The URI parser must support:

  - Wallet service pubkey in the URI path.
  - One or more relay query parameters.
  - Required 32-byte hex client secret.
  - Optional lud16 query parameter.
  - Percent-encoded relay URLs and parameters.

NWC connection bootstrap:

  1. Parse the NWC URI.
  2. Connect to the relay or relays from the URI.
  3. Fetch the wallet service replaceable info event, kind 13194.
  4. Parse the plaintext capabilities in content.
  5. Parse the notifications tag, if present.
  6. Parse the encryption tag, if present.
  7. Choose an encryption mode for requests and notification subscription.

Encryption policy:

  NIP-04 is still widely deployed and must work for initial compatibility.
  NIP-44 v2 should be supported where practical and preferred when the wallet
  service advertises it in the info event. If the info event has no encryption
  tag, assume NIP-04. Requests should include an encryption tag when using
  NIP-44 v2. Unsupported encryption errors must normalize cleanly.

OpenReceive receive-checkout methods for v0.1:

  get_info
  make_invoice
  lookup_invoice
  list_transactions, optional
  get_balance, optional diagnostic only

Standalone nwc-client-* protocol libraries should also support:

  pay_invoice

Out of scope for v0.1 standalone clients and for OpenReceive checkout:

  pay_keysend
  multi_pay_keysend
  make_hold_invoice
  cancel_hold_invoice
  settle_hold_invoice
  hold_invoice_accepted

multi_pay_invoice can be considered later, but is not required for v0.1.

Receive-only expectation:

  A receive-only code should support creating invoices and checking settlement,
  but should not support spending funds. OpenReceive checkout APIs never expose
  send-payment methods, so v0.1 does not need a separate wallet-compatibility
  confirmation step before using an NWC code. Use the shared Rizful
  receive-only/no-spend NWC code for project testing.

Notifications:

  - NWC notification event kind 23197 is used for NIP-44.
  - NWC notification event kind 23196 is used for NIP-04 compatibility.
  - Subscribe according to the negotiated/advertised encryption mode.
  - Filter notifications by wallet service pubkey and the client pubkey p tag.
  - Notifications can wake invoice lookup immediately.
  - Notifications do not replace polling.
  - Notification handlers must be idempotent.
  - Duplicate notifications, including duplicate NIP-04/NIP-44 notifications
    from wallets that publish both, must not double-credit payments.

NIP-47 event shape:

  - Info event: kind 13194, plaintext capability list in content, optional
    notifications tag, optional encryption tag.
  - Request event: kind 23194, encrypted JSON object with method and params,
    p tag set to the wallet service pubkey, optional expiration tag as unix
    timestamp seconds, encryption tag when using NIP-44 v2.
  - Response event: kind 23195, encrypted JSON object with result_type, error,
    and result; p tag set to client pubkey; e tag should reference the request.
  - Successful responses have error null.
  - Failed responses have error.code and error.message and result null.

NIP-47 to OpenReceive field mapping:

  - result.invoice -> invoice
  - result.amount -> amount_msats
  - result.payment_hash -> payment_hash
  - result.created_at / expires_at / settled_at stay unix timestamps in
    canonical HTTP responses.
  - result.state -> state, preserving lowercase NIP-47 values such as pending,
    settled, accepted, expired, and failed.
  - OpenReceive-specific invoice_id, fiat_quote, checkout, routes, and
    fulfillment fields may be added around the NIP-47 transaction object.

Long-running listener requirements:

  Every native listener implementation should satisfy these requirements:

    - Dedupe notifications by payment_hash.
    - Persist the last processed event timestamp or cursor where the wallet
      library supports resuming with since.
    - Reconnect with bounded exponential backoff.
    - Keep WebSocket connections alive with ping/heartbeat behavior.
    - Treat notification delivery as at-least-once, not exactly-once.
    - Make the database transition idempotent with a unique payment_hash or an
      update guard such as pending -> settled only.
    - Run notification listeners as long-running worker processes, not inside
      request handlers.
    - Still verify settlement with lookup_invoice before fulfillment.

Settlement detection:

  For NIP-47 incoming invoices, treat invoice as settled when lookup_invoice
  returns one of:

    settled_at present
    preimage present
    state == "settled"

  Normalize legacy/library-specific variants such as settled true, paid true,
  or uppercase SETTLED only at adapter boundaries. Store the strongest
  available proof. Prefer preimage when present.

Metadata:

  NIP-47 metadata must fit in wallet/relay payload constraints. OpenReceive
  should keep NWC metadata below 3900 characters after JSON serialization,
  leaving margin under the NIP-47 4096-character limit. Larger application
  metadata stays in the merchant database and is referenced by ids such as
  order_id, cart_id, user_id, or checkout_session_id.

Common wallet/service errors to normalize:

  NIP-47 native:

  NOT_IMPLEMENTED
  RESTRICTED
  UNAUTHORIZED
  RATE_LIMITED
  QUOTA_EXCEEDED
  INTERNAL
  UNSUPPORTED_ENCRYPTION
  INSUFFICIENT_BALANCE
  PAYMENT_FAILED
  OTHER

  OpenReceive normalized/internal:

  NOT_FOUND
  TIMEOUT
  INVALID_REQUEST
  WALLET_UNAVAILABLE
  INVOICE_EXPIRED
  UNSUPPORTED_METHOD

For receive-only checkout, send-payment errors such as INSUFFICIENT_BALANCE and
PAYMENT_FAILED should normally appear only in diagnostics or if a developer
accidentally configures read/write features. Normalize them anyway so SDKs
match the underlying NWC libraries.


Safe Polling
------------

Polling is reliable and should be the default. Notifications make checkout feel
instant, but polling is the recovery path when notification delivery is delayed
or missed.

Canonical polling cadence:

  immediate first lookup
  0-30 seconds:       poll every 1 second
  30-120 seconds:     poll every 3 seconds
  120-300 seconds:    poll every 6 seconds
  300s-expiry:        poll every 12 seconds
  after expiry:       stop and mark expired

Pseudo-code:

  while now < invoice.expires_at:
    result = lookup_invoice(payment_hash)

    if isSettled(result):
      mark_settled()
      publish_payment_received()
      fulfill()
      return

    sleep(min(nextDelay(invoice.created_at, now), invoice.expires_at - now))

  mark_expired()
  publish_invoice_expired()

This cadence must live in test vectors so every language agrees.


Fiat Pricing And Quotes
-----------------------

OpenReceive must have first-class fiat to msat quoting. Most merchants think in
USD, EUR, GBP, or local currency, not msats.

Core API:

  quoteFiatToMsats({
    fiat: {
      currency: "USD",
      value: "0.10"
    }
  })

Response:

  {
    "fiat": {
      "currency": "USD",
      "value": "0.10"
    },
    "btc_fiat_price": "50000.00",
    "amount_sats": 200,
    "amount_msats": 200000,
    "source": "static_mock",
    "as_of": 1781740800,
    "expires_at": 1781741400
  }

Rules:

  - Use decimal math.
  - Do not use binary floats for money.
  - Lock the fiat quote at invoice creation.
  - Store price source, rate, as_of, expires_at, fiat amount, fiat currency,
    and amount_msats.
  - Enforce a minimum of 1 sat.
  - Round up to a whole sat when wallet/UI behavior needs sat-level display.
  - Amount to NWC make_invoice is in millisatoshis. OpenReceive calls this
    amount_msats and maps it to NIP-47 amount at the adapter boundary.

Formula:

  amount_btc   = fiat_amount / btc_fiat_price
  amount_sats  = amount_btc * 100_000_000
  amount_msats = amount_sats * 1000

Default price source architecture:

  PriceProvider
    getBtcFiatRates(currencies: string[]): RateMap

Initial providers:

  - CoinGecko Simple Price compatible endpoint.
  - OpenReceive exchange-rate mirror endpoint.
  - Megalithic exchange-rate fallback endpoint.
  - CoinGecko direct fallback endpoint.
  - Static deterministic provider for tests and screenshots.

Rate endpoint contract:

  The canonical live response shape is CoinGecko Simple Price compatible:

    {
      "bitcoin": {
        "usd": 62599,
        "eur": 54792,
        "gbp": 47539
      }
    }

  The adapter must accept integer or decimal JSON numbers and convert them to
  Decimal/string internally. Do not rely on JavaScript floating point for final
  money math.

  Official OpenReceive mirror, planned:

    https://openreceive.org/exchange_rates

  Existing compatible fallback:

    https://megalithic.me/exchange_rates

  CoinGecko direct fallback:

    https://api.coingecko.com/api/v3/simple/price?ids=bitcoin&vs_currencies=usd,aed,ars,aud,bdt,bhd,bmd,brl,cad,chf,clp,cny,czk,dkk,eur,gbp,gel,hkd,huf,idr,ils,inr,jpy,krw,kwd,lkr,mmk,mxn,myr,ngn,nok,nzd,php,pkr,pln,rub,sar,sek,sgd,thb,try,twd,uah,vef,vnd,zar

  The open source repo should treat https://openreceive.org/exchange_rates as
  an external public rate mirror. Its implementation may live in the private
  Rails/Kamal app, but the open source code depends only on the JSON contract,
  not on that private app's source or deployment.

Supported fiat/quote currencies for v0.1:

  usd, aed, ars, aud, bdt, bhd, bmd, brl, cad, chf, clp, cny, czk,
  dkk, eur, gbp, gel, hkd, huf, idr, ils, inr, jpy, krw, kwd, lkr,
  mmk, mxn, myr, ngn, nok, nzd, php, pkr, pln, rub, sar, sek, sgd,
  thb, try, twd, uah, vef, vnd, zar

Rate source order:

  1. static_mock in tests, docs screenshots, and deterministic demo mode.
  2. openreceive_mirror for production demos once available.
  3. megalithic_mirror as a compatible fallback.
  4. coingecko_direct as the final upstream fallback.

Staleness handling:

  The exchange-rate response does not need to contain timestamps. The
  OpenReceive service records fetch time as as_of, applies cache_seconds, and
  locks the quote at invoice creation with invoice_quote_ttl_seconds.

Recommended config file:

  spec/data/rates/price-sources.json

Recommended default:

  cache_seconds: 30
  invoice_quote_ttl_seconds: 600
  primary: https://openreceive.org/exchange_rates
  secondary: https://megalithic.me/exchange_rates
  tertiary: CoinGecko Simple Price direct URL
  test/dev: static mock price, e.g. BTC/USD = 50000.00

The static mock is mandatory. It keeps docs, tests, screenshots, and demos
stable.

Anti-Duplication Strategy
-------------------------

This is the core engineering discipline for the whole project.

Never duplicate:

  - provider lists
  - supported currencies
  - route rankings
  - fiat-to-msat behavior
  - polling cadence
  - settlement detection rules
  - invoice schemas
  - event schemas
  - demo product data
  - Rizful test wallet behavior
  - package README structure

Use these mechanisms:

  - JSON Schema validates canonical data.
  - OpenAPI generates shared HTTP models.
  - AsyncAPI documents shared event models.
  - Codegen emits provider-data packages.
  - Test vectors enforce behavior across languages.
  - The shared Rizful receive-only NWC code gives every SDK the same real
    wallet behavior in integration tests.
  - Scaffolding tools create examples from templates.
  - Docs use shared snippets where possible.
  - CI fails when generated files are stale.

Where duplication is acceptable:

  - Thin idiomatic wrappers in each ecosystem.
  - Framework-specific adapters.
  - UI components written in each target UI framework.
  - Small example glue code that teaches local conventions.

Rule of thumb:

  The same business rule in two languages is acceptable only if both load the
  same vector and prove identical output.


NWC Client Libraries
--------------------

The shared implementation burden moves into NWC client libraries and
conformance, not into a daemon. Each serious server-side SDK must either wrap a
recently maintained NWC client or ship a standalone neutral NWC client that can
be used without OpenReceive.

These NWC clients are standalone protocol clients, not OpenReceive-only
packages. They may live under the openreceive GitHub organization for
stewardship, CI, release automation, and security response, but their names,
READMEs, examples, and public APIs should make sense to any NWC developer.

Canonical standalone naming:

  Repositories:
    nwc-client-js, if the project ever needs its own JS client - FOR NOW WE USE ALBY SDK
    nwc-client-ruby, if nwc-ruby is mirrored or superseded - FOR NOW WE USE NWC-RUBY
    nwc-client-php
    nwc-client-py
    nwc-client-go
    nwc-client-rs
    nwc-client-swift
    nwc-client-jvm
    nwc-client-dotnet
    nwc-client-elixir
    nwc-client-dart

  Public package names:
    npm:        @nwc-client/core, only if a standalone JS client is needed
    RubyGems:   nwc-client-ruby, only if nwc-ruby is not the long-term package
    Packagist:  nwc-client/php
    PyPI:       nwc-client-py, import name nwc_client
    Go:         github.com/openreceive/nwc-client-go
    crates.io:  nwc-client-rs
    SwiftPM:    repo nwc-client-swift, product NWCClient
    Maven:      org.nwcclient:nwc-client-jvm, subject to namespace setup
    NuGet:      Nwc.Client
    Hex:        nwc_client
    pub.dev:    nwc_client, only for server-side Dart if needed

If a neutral GitHub or package namespace is created later, prefer that for raw
NWC clients. If they stay under the openreceive organization, avoid
OpenReceive-branded repository names and keep the package APIs neutral.

OpenReceive backend SDKs consume these packages. They should not be required
for an application that only wants a raw NWC protocol client.

Each standalone NWC client README must start with raw NWC usage, not
OpenReceive usage. OpenReceive integration belongs in a later section so users
understand the protocol client is useful on its own.

Standalone NWC clients must not depend on OpenReceive SDKs, provider data,
checkout UI packages, demos, or product-specific state machines. Dependency
direction is one-way: OpenReceive depends on NWC clients, never the reverse.

Maintenance gate for third-party NWC clients:

  - Repository pushed within the last 12 months at the time we adopt it.
  - MIT, Apache-2.0, BSD, or similarly permissive license.
  - Supports NIP-47 info-event bootstrap or can be wrapped to support it.
  - Supports NIP-47 receive checkout methods: get_info, make_invoice, and
    lookup_invoice.
  - Supports pay_invoice for general standalone NWC client use.
  - Supports or can be extended to support payment_received notifications.
  - Handles NIP-04 encrypted wallet messages.
  - Supports or can be extended to support NIP-44 v2 negotiation.
  - Has tests or can pass OpenReceive's Rizful-backed NWC conformance smoke
    suite.
  - Does not force browser/mobile exposure of NWC secrets.
  - Can run in ordinary server framework jobs, queue workers, cron tasks,
    serverless functions, or long-running worker roles.

Candidate maintenance snapshot checked on 2026-06-19:

  Ruby:
    Use MegalithicBTC/nwc-ruby initially. It was pushed 2026-06-13 and already
    targets Rails/server-side receive checkout. The OpenReceive Ruby SDK should
    wrap it first. If desired later, move, mirror, or supersede it as
    nwc-client-ruby while preserving package identity, compatibility, and author
    credit.

  JavaScript / TypeScript:
    Use getAlby/js-sdk for NWC initially. It was pushed 2026-05-23 and is the
    mature JS option for Nostr Wallet Connect. @openreceive/node should wrap the
    receive-side subset and hide send-payment methods from checkout APIs.

  Go:
    Audit rolznz/go-nwc first. It was pushed 2025-06-30, which satisfies the
    current recency gate but is close enough that it must be rechecked before
    adoption. If it lacks notification resilience, NIP-44 support, redaction,
    or conformance coverage, create github.com/openreceive/nwc-client-go and use
    rolznz/go-nwc only as a reference.

  Rust:
    Use rust-nostr/nostr initially. It was pushed 2026-06-15, has substantial
    adoption, and explicitly includes Nostr Wallet Connect support. The
    OpenReceive Rust crate should wrap the receive-checkout subset and provide
    Axum/Actix-friendly workers.

  Swift:
    Audit SparrowTek/NostrKit. It was pushed 2026-04-29 and has NWC support,
    but OpenReceive needs server-side Vapor suitability, Linux compatibility,
    notification behavior, and secret-redaction checks. If those do not pass,
    create nwc-client-swift with SwiftPM product NWCClient.

  PHP:
    Audit dsbaars/nostr-php-nwc. It was pushed 2025-07-07, so it currently
    passes the recency gate, but it is small and must prove receive checkout,
    notifications, PHP 8.2+/8.3+ compatibility, Laravel queue friendliness, and
    Symfony Messenger friendliness. Plan to write nwc-client-php unless the
    audit shows it is ready to adopt or collaborate on.

  Python:
    Write nwc-client-py. The known Python NWC candidate checked from the
    awesome-nwc list was pushed 2024-07-24, which is too stale for a primary
    dependency. The Python client should support asyncio first, with sync
    wrappers for Django/Flask where practical.

  Kotlin / JVM:
    Write nwc-client-jvm. There is no currently vetted Kotlin/JVM NWC
    client for server-side Ktor and Spring Boot. The package should expose
    coroutine APIs, Java-friendly facades, and worker examples for Spring
    scheduling and Ktor background jobs.

  .NET / C#:
    Write Nwc.Client. There is no currently vetted .NET NWC client for
    ASP.NET Core. The package should expose async Task APIs, hosted-service
    notification listeners, dependency injection, and SignalR-friendly events.

  Elixir:
    Write nwc_client. There is no currently vetted Elixir NWC client for
    Phoenix. The package should use OTP supervision for relay connections,
    GenServer/Broadway/Oban-friendly polling, and Phoenix PubSub events.

  Dart:
    Do not use Dart/Flutter as a live wallet-secret runtime for consumer apps.
    Flutter packages are checkout UI only. If a server-side Dart ecosystem
    becomes important, write nwc_client; currently observed Dart NWC packages
    were pushed in 2024 and should be treated as references only.

Minimum NWC client responsibilities:

  - Parse nostr+walletconnect URIs.
  - Support multiple relay parameters and optional lud16.
  - Validate and redact wallet pubkey, relay URLs, and client secret.
  - Connect to one or more Nostr relays over WebSocket.
  - Fetch and parse the wallet service kind 13194 info event.
  - Parse content capabilities, notifications tag, and encryption tag.
  - Choose NIP-04 or NIP-44 v2 according to wallet support.
  - Sign Nostr events correctly.
  - Construct request events as kind 23194 with p tag, optional expiration tag,
    and encryption tag when using NIP-44 v2.
  - Parse response events as kind 23195 with p tag, e tag, result_type, error,
    and result.
  - Encrypt/decrypt NWC request and response payloads.
  - Support NIP-04 and track NIP-44 v2 support.
  - Implement get_info.
  - Implement make_invoice.
  - Implement lookup_invoice.
  - Implement pay_invoice for standalone NWC users.
  - Optionally implement list_transactions for diagnostics.
  - Never expose pay_invoice through the OpenReceive receive-checkout surface.
  - Do not implement pay_keysend or hold-invoice methods for v0.1.
  - Subscribe to payment_received notifications where the language runtime and
    wallet support it.
  - Dedupe notifications by payment_hash.
  - Dedupe duplicate NIP-04/NIP-44 notifications.
  - Reconnect with bounded exponential backoff.
  - Support heartbeat/ping or equivalent dead-connection detection.
  - Persist or expose the last processed event timestamp/cursor where possible.
  - Treat notifications as hints that trigger lookup_invoice, not final proof.
  - Normalize settlement detection into the canonical OpenReceive state model.
  - Normalize wallet/service errors into canonical OpenReceive errors.
  - Enforce NWC metadata below 3900 serialized characters.
  - Provide safe polling helpers with the canonical cadence.
  - Provide idempotency guidance and helper APIs for framework adapters.
  - Provide shared conformance vectors and Rizful-backed NWC smoke tests.

  Node:

Frontend SDK API
----------------

Frontend packages must never require wallet secrets.

Frontend packages do not create live invoices by themselves. They call the
merchant backend's OpenReceive routes, receive display-safe invoice data, and
listen for passive settlement updates from that backend.

They receive:

  invoice_id
  invoice
  payment_hash
  amount_msats
  state
  expires_at
  checkout.events_url
  checkout.routes_url

They expose a checkout view model:

  data:
    transaction
    invoice
    amountMsats
    expiresAt
    expiresIn
    state
    paymentHash
    routes
    assets
    selectedAsset
    selectedCountry
    providers
    activeProvider
    error

  actions:
    copyInvoice()
    openWallet()
    refreshState()
    selectAsset(asset)
    selectCountry(country)
    selectProvider(provider)
    retry()
    cancel()

  events:
    created
    checking
    payment_received
    expired
    failed
    copied
    route_selected

The view model is the real frontend API. Default components, headless
components, web components, and design-system recipes all consume the same view
model.

Headless browser package:

  @openreceive/browser

Responsibilities:

  - Parse invoice object.
  - Connect to SSE events.
  - Track countdown.
  - Generate lightning: invoice URI payloads.
  - Generate safe QR SVG/PNG output with opaque background, high contrast, and
    at least a 4-module white quiet zone.
  - Copy invoice.
  - Open lightning: URI.
  - Load provider routes.
  - Emit payment_received/expired/error events.
  - Expose a framework-neutral checkout state machine.
  - Expose framework-neutral actions for copy, open-wallet, refresh, select
    route, dismiss, retry, and cancel.

Frontend package philosophy:

  OpenReceive frontend packages should make it easy to build a polished payment
  wizard without forcing a design system. Teams must be able to use Mantine,
  Material UI, Chakra, Radix, Tailwind, Bootstrap, DaisyUI, shadcn-style
  components, Angular Material, PrimeNG, Vuetify, Svelte component libraries,
  or their own design system.

  The strategy is layered:

    1. @openreceive/browser
       Framework-neutral state machine, event handling, provider-route loading,
       countdowns, QR helpers, copy/open-wallet actions, and typed data. No DOM
       ownership beyond optional helpers.

    2. Framework headless adapters
       React hooks/context, Vue composables, Svelte stores/actions, Angular
       services/signals/directives, and Solid signals. These expose state and
       actions but do not dictate markup.

    3. Accessible unstyled primitives
       Small components such as Root, Modal, OpenReceiveQRCode, CopyButton,
       RouteWizard, ProviderList, PaymentState, Countdown, and Success. They own
       behavior, ARIA, focus management, keyboard handling, and slot structure,
       but use minimal reset styles and accept class/style/slot/component
       overrides. The QR primitive owns the QR quiet zone and contrast defaults
       even when the surrounding UI is fully customized.

    4. Optional default UI
       A pleasant default checkout for demos, quickstarts, and teams without a
       design system. This is not the main customization surface.

    5. Design-system recipes
       Example integrations for Mantine, Material UI, Chakra, Tailwind/Radix,
       Bootstrap, Angular Material, Vuetify, and common Svelte UI libraries.
       These should usually be docs/examples, not hard dependencies or required
       packages.

Backend Framework Adapters
--------------------------

Each backend adapter should provide:

  - Route helper.
  - Invoice creation helper.
  - Event stream helper.
  - Auth middleware integration point.
  - Invoice access authorization hook.
  - Guest checkout/session-token support where idiomatic.
  - Payment model/table migration or documented storage interface.
  - Polling job/worker integration.
  - Notification listener integration where practical.
  - Fulfillment hook.
  - Middleware/security defaults.
  - Test helpers.
  - Demo-ready defaults.

Required adapter auth hooks:

  currentUser(request)
    Return the app's current user/account/customer object, or null for a guest
    checkout flow.

  authorizeInvoiceCreate(request, user, input)
    Decide whether this request may create an invoice.

  authorizeInvoiceAccess(request, user, invoice)
    Decide whether this request may read, refresh, poll, or subscribe to events
    for this invoice.

  buildInvoiceMetadata(request, user, input)
    Attach user_id, order_id, cart_id, tenant_id, checkout_session_id, or other
    app-owned identifiers needed for fulfillment and authorization.

  eventAuthStrategy(request, invoice)
    Choose same-site session cookies, signed event URL, polling fallback, or a
    framework realtime bridge.

Adapters should fail closed when authorization hooks are missing in production
mode. Demos may use a simple signed guest checkout session so the flow works
without user accounts.

Express:

  app.use("/openreceive", openreceiveExpress({
    requireUser: requireUser,
    client,
    buildMetadata: (req) => ({
      userId: req.user.id,
      cartId: req.session.cartId
    }),
    authorizeInvoiceAccess: async (req, invoice) => {
      return invoice.metadata.userId === req.user.id
    },
    fulfill: async ({ invoice, metadata }) => {
      await unlockSticker(metadata.userId, metadata.fruit)
    }
  }))

Rails:

  OpenReceive.configure do |config|
    config.wallet = :nwc
    config.nwc = ENV.fetch("OPENRECEIVE_NWC")
    config.authenticate = ->(controller) {
      controller.authenticate_user!
    }
    config.authorize_invoice = ->(controller, invoice) {
      invoice.user_id == controller.current_user.id
    }
    config.metadata = ->(controller, params) {
      {
        user_id: controller.current_user.id,
        order_id: params[:order_id]
      }
    }
    config.fulfill = ->(invoice) {
      FruitStickerUnlock.create!(invoice_id: invoice.id)
    }
  end

  mount OpenReceive::Rails::Engine => "/openreceive"

Rails adapter requirements:

  - The Rails package should keep checkout APIs receive-only by construction:
    no pay_invoice, pay_keysend, or other spend methods on the OpenReceive
    checkout surface.
  - If the implementation uses nwc-ruby or rbsecp256k1, document native build
    dependencies for macOS, Debian/Ubuntu, Alpine, and Docker/Kamal images.
  - The NWC notification listener should deploy as a Kamal role using the same
    app image with a different command, not as a Kamal accessory.
  - Listener handlers must be idempotent at the database level because deploys,
    reconnects, restarts, and multiple listener hosts can replay the same
    payment_received event.
  - Rails examples should show an AppState or equivalent cursor for resuming
    notifications with since after restart.

    },
    "fulfillment": {
      "type": "digital_download",
      "items": [
        "apple.svg",

        "pear.svg"
      ]
    }

  examples/hello-fruit/shared/stickers/banana.svg
  examples/hello-fruit/shared/stickers/orange.svg
  examples/hello-fruit/shared/stickers/pear.svg

User flow:

  1. User opens demo.
  2. User picks fruit if he wants or clicks the suggested fruit which is random
  3. Frontend calls backend: POST /api/checkout.
  4. Backend calls OpenReceive to create invoice for USD 0.10.
  5. Backend stores invoice_id, payment_hash, quote, product metadata.
  6. Frontend displays QR code, copy button, lightning wallet button,
     countdown, state, and payment route wizard.
  7. Customer pays.
  8. Backend confirms with lookup_invoice.
  9. Backend emits payment_received event.
  10. Frontend flips to settled/success state.
  11. Backend serves signed one-time sticker URL.

Optional deterministic fruit:

  fruitIndex = parseInt(preimage.slice(0, 8), 16) % fruits.length

Demo modes:

  Rizful test NWC mode:

    OPENRECEIVE_MODE=test_nwc
    OPENRECEIVE_WALLET=nwc
    OPENRECEIVE_NWC=nostr+walletconnect://...
    OPENRECEIVE_PRICE_SOURCE=static

  Production NWC mode:

    OPENRECEIVE_MODE=production
    OPENRECEIVE_WALLET=nwc
    OPENRECEIVE_NWC=nostr+walletconnect://...
    OPENRECEIVE_PRICE_SOURCE=coingecko

Every demo must support Rizful test NWC mode. Tests create real low-value
invoices and settlement is confirmed by paying the fruit-sticker checkout.
That is acceptable for this project because the test wallet is receive-only,
the amounts are tiny, and the owner is paying themselves.

  reason to host public OpenReceive demos there. Demo containers are low
  traffic but operationally noisy: many runtimes, frequent deploys, routing
  changes, screenshot/debug sessions, and occasional memory spikes. Production
  isolation matters more than saving one small VPS bill.

  Observed 2026-06-19: app-virginia had 4 vCPU, about 15.2 GB RAM, roughly
  3.9 GB used, 6 Docker containers, low load, and kamal-proxy bound to public
  ports 80 and 443. That confirms capacity exists, but it does not change the
  isolation decision.

  app-virginia rule:

    Do not deploy stable public demos to app-virginia.
    Do not bind another public Caddy/Traefik to app-virginia ports 80/443.
    Do not route demo traffic through the private app's kamal-proxy.
    Do not use app-virginia as part of the normal demo architecture.

Caddy ownership and routing:

  The demo node owns its own proxy. Caddy may bind 80/443 there because the
  node is not app-virginia and does not host the critical private apps. Keep the
  private app's kamal-proxy out of the normal demo path.

  Prefer Caddy for the first office-node deployment because the route map is
  easy to read. Use Docker service-name routing from day one so adding demos
  does not require assigning and tracking host ports.

    demo-nodejs-whatever.openreceive.org {
      reverse_proxy demo-nodejs-whatever:3000
    }

  Put Caddy and demo containers on a shared external Docker network such as
  openreceive_demo_proxy. Each demo compose project joins that network and
  exposes its internal HTTP port only to the Docker network. Do not publish
  per-demo host ports for stable public demos.

  Canonical routing rules:

    - Caddy is the only public container that publishes host ports 80 and 443.
    - Demo containers use expose, not ports, for their HTTP listener.
    - Demo service names are stable and match the demo slug where practical.
    - Caddy routes to Docker service names, not 127.0.0.1 host ports.
    - Do not use network_mode: "host" for the canonical demo fleet.
    - Do not assign per-demo host ports except in a gitignored local debug
      override.

  Keep one generated Caddyfile, or one generated imported Caddyfile fragment per
  demo, so deployment can add/update routes without hand-editing production
  config.

  Create the proxy network once on the demo node:

    docker network create openreceive_demo_proxy

  Keep the root Caddyfile small and import generated site fragments:

    import /etc/caddy/sites/*.caddy

  Local or preview demos can use Cloudflare Tunnel or VPN-only access. They are
  not stable public demos unless promoted to the dedicated office demo node with
  the requirements above.

TLS plan:

  Cloudflare is the public TLS edge for the current setup. Keep Cloudflare proxy
  enabled for stable public demo subdomains unless a specific demo needs DNS
  only for debugging.

  Origin TLS for demos belongs on Caddy on the office Ubuntu demo node. Use
  Caddy-managed Let's Encrypt certificates for demo hostnames by default. This
  works with Cloudflare Full Strict because Full Strict accepts a valid public
  CA certificate such as Let's Encrypt; a Cloudflare Origin Certificate is not
  required for the default demo setup.

  Keep the Cloudflare demo records proxied so browsers see Cloudflare's public
  edge certificate and Cloudflare connects onward to Caddy. Caddy's certificate
  is for the Cloudflare-to-origin leg. Manage Cloudflare DNS records manually in
  the Cloudflare dashboard for v0.1.

  Cloudflare cache and realtime rules:

    - Bypass cache for /openreceive/*, /api/*, /events, and text/event-stream.
    - Keep polling fallback even when SSE is enabled.
    - Send no-store on invoice/state/event responses.
    - Cache only static demo assets, docs, and screenshots.

  Recommendation:

    Keep Cloudflare for public DNS/TLS edge, but run public demos on a separate
    demo node from the start. Add explicit Cloudflare demo subdomain records and
    demo-node origin routes.

Deployment workflow:

  CI means a hosted automation runner, such as GitHub Actions, does the boring
  repeatable work after each push or release. For OpenReceive demos, CI should
  build and test images; the office demo node should only pull already-built
  images and restart containers.

  Start with the same deploy path both locally and in CI:

    local first:
      Run ./demos/deploy/scripts/deploy-demo express-demo from a trusted
      operator machine. The script SSHes to the office demo node, pulls the
      selected image, updates compose/Caddy config, reloads Caddy, and runs the
      smoke test.

    CI later:
      GitHub Actions runs tests, builds images, and pushes them to GHCR. A
      protected deploy step then runs the same deploy-demo or promote-demo
      script, either from an operator machine over SSH or from a self-hosted
      runner on the office demo node.

  Because the preferred demo node lives behind WireGuard, do not assume a
  GitHub-hosted runner can SSH directly to it. Start with manual promotion, or
  run a protected self-hosted GitHub Actions runner on the office demo node for
  deploy jobs only. GitHub-hosted runners can still do tests, package builds,
  image builds, and GHCR pushes.

  Safe CI/deploy split:

    GitHub-hosted runner:
      Validate code, run tests, build demo images, push immutable GHCR images.
      This runner does not need office network access.

    Office self-hosted runner or manual operator command:
      Pull tested GHCR image digests, update compose/Caddy config, reload
      Caddy, run smoke tests, and roll back if needed.

  Do not give untrusted pull requests access to the office self-hosted runner,
  WireGuard keys, deploy SSH keys, or NWC secrets.
  Deployment workflows must be restricted to protected branches, tags,
  workflow_dispatch, or a protected GitHub environment.

  - GitHub Actions builds demo images.
  - Images are tagged with git SHA and optionally latest for stable branch.
  - Images are pushed to GHCR.
  - A deploy job or operator command runs on the office demo node, or reaches it
    through a trusted SSH path.
  - The host pulls updated compose files or receives a generated deploy bundle.
  - The host runs docker compose pull and docker compose up -d for changed demos.
  - Caddy config is generated or updated from the demo registry.
  - Caddy is reloaded after config validation succeeds.
  - The Cloudflare DNS inventory is checked against the demo subdomain registry.
  - A smoke test hits /healthz and the public checkout URL.
  - Failed smoke tests roll back to the previous image tag.

  Minimal GitHub Actions mental model:

    on pull request:
      run tests and build changed demos. Run Rizful NWC smoke tests only where
      the trusted test secret is available; do not deploy stable demos

    on merge to main:
      run tests, build changed demos, push GHCR images, optionally deploy
      Rizful test demos after smoke tests pass

    on tagged release:
      publish packages, rebuild affected demos, promote tested image digests to
      stable demo subdomains

Deployment churn policy:

  Expect SDK packages and demos to change often. The demo platform must make
  frequent updates cheap without turning the demo node into a hand-managed build
  machine.

  Use two deployment lanes:

    preview lane:
      Every pull request or branch may build changed demos with noindex and
      short-lived preview URLs. Trusted previews may use the Rizful test NWC
      secret; untrusted previews build without NWC smoke tests. Preview
      deploys can use local tunnels, VPN-only access, or
      preview.openreceive.org. They do not imply that a stack is ready for the
      stable public demo list.

    stable lane:
      Merges to main build immutable demo images tagged by git SHA and image
      digest. Stable public demo subdomains are promoted to a tested image
      digest after smoke tests pass. Rizful test demos may auto-promote once
      the pipeline is reliable; production-wallet demos require explicit
      manual approval.

  Package update rules:

    - A package release or SDK change should trigger rebuilds for every demo
      that consumes that package.
    - Demos should pin package versions or workspace references explicitly, so
      a public demo always reports the OpenReceive/NWC client versions it is
      running.
    - Each demo should expose a non-secret build metadata endpoint or footer
      containing demo git SHA, image digest, SDK package versions, mode, and
      deployed_at.
    - Dependabot/Renovate-style package update PRs are acceptable, but they must
      run the same demo smoke tests as human-authored package changes.
    - Do not install or compile packages interactively on the demo node. Build
      images in CI, push to GHCR, then pull immutable images on the demo node.

  Deploy safety rules:

    - Deploy one demo independently without redeploying the whole fleet.
    - Use a per-demo deploy lock so two CI jobs cannot update the same compose
      project at the same time.
    - Write a deployment manifest with demo slug, image digest, package
      versions, git SHA, config mode, deployed_at, and previous image digest.
    - Run smoke tests after each deploy and automatically roll back to the
      previous image digest on failure.
    - Keep the proxy config stable; adding or updating a demo should usually be
      a compose-label change plus a Cloudflare DNS registry change.
    - Batch large demo-matrix updates during a maintenance window if they touch
      many language runtimes at once.

Receive-only NWC secrecy:

  A receive-only NWC code cannot spend funds, but it is still a credential. It
  can create invoices, inspect invoice/payment metadata, and potentially spam
  your wallet or leak business activity. Treat it like an API key.

  Rules for OpenReceive itself:

    - Never commit OPENRECEIVE_NWC.
    - Never include OPENRECEIVE_NWC in frontend bundles.
    - Never expose OPENRECEIVE_NWC in HTML, JS, source maps, screenshots, logs,
      telemetry, error pages, or health endpoints.
    - Always redact nostr+walletconnect:// URIs in logs.
    - Always provide .env.example files with placeholder values.
    - Every framework demo should include an automated check that the built
      frontend assets do not contain nostr+walletconnect:// or OPENRECEIVE_NWC.

  Rules for OpenReceive users:

    - Put receive-only NWC in a server-side environment variable.
    - Do not put it in React/Vue/Svelte/Angular public env vars.
    - Do not put it in mobile apps.
    - Do not paste it into support tickets, GitHub issues, screenshots, or
      analytics tools.
    - Rotate it if it was committed, logged, or shared.
    - Use a separate receive-only NWC code per app or environment when possible.

Test Strategy
-------------

Testing has to be more serious than the number of libraries. The tests are the
thing that keep the project from becoming a pile of similar but incompatible
SDKs.

Test layers:

1. Schema validation

  - Validate every JSON Schema.
  - Validate provider registry.
  - Validate supported currencies.
  - Validate demo product data.
  - Validate OpenAPI.
  - Validate AsyncAPI.

2. Unit tests

  - fiat-to-msat conversion
  - decimal rounding
  - invoice state transitions
  - NWC URI parsing
  - secret redaction
  - provider route selection
  - event parsing
  - QR payload, quiet-zone, and contrast helpers
  - idempotency key handling
  - webhook retry state

3. Shared test vectors

  - fiat-to-msats
  - settlement detection
  - polling backoff
  - provider routing
  - NWC URI parsing
  - NWC info event parsing
  - NWC encryption negotiation
  - NWC request/response event shape
  - error normalization
  - idempotency behavior
  - metadata size limit

4. Rizful receive-only NWC test harness

  Location:

    tools/live-nwc-test

  Required behavior:

    parse OPENRECEIVE_NWC from a gitignored env file
    create a low-value fruit-sticker invoice
    look up the invoice before payment
    poll with the canonical cadence
    subscribe to notifications where the client supports it
    confirm settlement only through lookup_invoice
    redact the NWC string in every log and error path
    skip Rizful NWC smoke tests clearly when OPENRECEIVE_NWC is not present

  Required live scenarios:

    invoice created through Rizful
    settled by polling after the owner pays the invoice
    settled by notification plus lookup when notifications are available
    duplicate idempotency key
    rate source failure using the static price provider
    receive-only NWC success
    multiple relay URI parsing
    missing encryption tag falls back to NIP-04
    NIP-44 advertised and selected when Rizful advertises it
    metadata over 3900 serialized characters rejected before NWC request

  Intentionally not required:

    A local fake wallet, fake auto-settlement, forced wallet timeouts, forced
    notification loss, or provider-specific compatibility confirmation. NWC
    codes are treated as compatible; the project proves its own adapters by
    using the Rizful receive-only test code.

5. Conformance tests

  Every backend SDK and every standalone NWC client must pass the same shared
  vectors plus the same Rizful-backed NWC smoke flow when OPENRECEIVE_NWC is
  available. Third-party NWC dependencies must be wrapped by an OpenReceive
  adapter that passes that same suite.

6. Demo E2E tests

  Use Playwright for web demos:

    open demo
    create invoice
    see QR with white quiet zone and copy button
    see provider wizard
    owner pays the Rizful-backed invoice
    UI changes to settled
    sticker download appears

  Mobile demos need platform-specific smoke tests once they exist.

7. Security tests

  - Secret scanning.
  - Ensure NWC strings are redacted in logs.
  - Ensure frontend bundles do not contain OPENRECEIVE_NWC.
  - Ensure frontend bundles do not contain nostr+walletconnect://.
  - Ensure source maps do not contain OPENRECEIVE_NWC.
  - Ensure mounted framework APIs reject missing/invalid auth when auth is
    configured.
  - Ensure one user/session cannot read or subscribe to another user's invoice.
  - Ensure signed event URLs expire and are scoped to one invoice.
  - Ensure guest checkout tokens are unguessable and scoped to the checkout
    session.
  - Ensure fulfillment cannot be triggered by client-only state changes.

8. Package smoke tests

  For every publishable package:

    install from local build artifact
    import package
    run minimal create/lookup against Rizful NWC when OPENRECEIVE_NWC is
    available
    run provider data import if applicable

Security Model
--------------

Threats:

  - Merchant accidentally exposes NWC secret to browser.
  - Merchant commits a receive-only NWC code to git.
  - Merchant stores NWC secrets in GitHub Actions unnecessarily.
  - Read/write NWC leaks and can spend funds.
  - Duplicate notification double-credits a customer.
  - Client forges "settled" state.
  - Old invoices keep polling forever and hit rate limits.
  - Provider registry makes stale or overconfident claims.
  - Public demo is abused for spam or invoice generation.
  - Logs, errors, screenshots, or telemetry leak wallet details.

Controls:

  - Receive-only NWC default.
  - No wallet compatibility registry or preflight ceremony; NWC codes are
    treated as compatible, and checkout APIs never expose send-payment methods.
  - .env.example templates committed; real env files ignored.
  - Host-side secret files stored outside the repo with chmod 600.
  - Secret redaction utility in every SDK.
  - Build-time frontend bundle scan for OPENRECEIVE_NWC and
    nostr+walletconnect://.
  - Backend-only settlement verification.
  - Idempotent fulfillment.
  - Unique payment_hash constraints.
  - Bounded polling.
  - Provider registry linter.
  - Demo rate limits.
  - Secret scanning CI.
  - Security docs from day one.

Public demo safety:

  - Low invoice amount.
  - Rate limit by IP/session.
  - Optional captcha only if abused.
  - No user-uploaded fulfillment.
  - No dynamic file paths.
  - Daily receive cap.
  - Clear "test payment" disclosure.
  - Separate demo wallet.
  - Logs without secrets.

Important Design Decisions To Make Early
----------------------------------------

1. First NWC client adoption decisions

  Good candidates:

    Ruby:
      Use MegalithicBTC/nwc-ruby first. It is current, server-side, and already
      models the Rails worker/listener shape OpenReceive wants.

    JavaScript/TypeScript:
      Use getAlby/js-sdk first. It is current, mature, and close to the first
      Node/React reference stack.

    Go:
      Audit rolznz/go-nwc before depending on it. If it falls short on
      notifications, NIP-44, tests, or maintenance, write nwc-client-go.

    Rust:
      Use rust-nostr/nostr first and wrap only the receive-checkout surface.

    PHP, Python, JVM, .NET, Elixir:
      Plan to write standalone NWC clients unless a fresh, audited,
      permissively licensed client passes the full conformance suite.

  Recommendation:

    Start the native reference stack in TypeScript/Node for v0.1.0 because the
    first frontend and backend demo material is closest to that ecosystem. Use
    Ruby/Rails immediately after to prove the non-JS story with nwc-ruby. Build
    Python and PHP NWC clients next because those ecosystems are important and
    cannot depend on a stale or uncertain NWC layer.

2. First source of truth for NWC

  Recommendation:

    Use proven NWC libraries where possible for early delivery, wrap them behind
    OpenReceive interfaces, and keep protocol behavior covered by shared
    vectors plus Rizful-backed NWC smoke tests. Do not hand-roll Nostr
    encryption unless the existing library path blocks the project.

3. Hosted test demo policy

  Recommendation:

    Hosted public demos default to the shared Rizful receive-only test NWC
    code. These are real mainnet test purchases for tiny sticker amounts, which
    is acceptable because the owner is paying themselves. No separate wallet
    compatibility confirmation is required beyond using the NWC code.

4. Provider registry claims

  Recommendation:

    Treat the provider registry as maintained data with evidence, verification
    dates, and conservative wording. The wizard should help users route a
    payment, not promise availability.

5. Package naming

  Recommendation:

    Reserve names early, but publish only the v0.1 packages that are actually
    tested.

Acceptance Checklist
--------------------

This plan is complete enough when these statements are true:

  - There is one product contract, not one per language.
  - Provider data is canonical and validated.
  - Price conversion is canonical and tested.
  - Polling behavior is canonical and tested.
  - Settlement detection is canonical and tested.
  - A normal server app can run the whole flow without an extra OpenReceive
    daemon.
  - JS packages prove the SDK shape.
  - Rails proves the framework-native worker/listener shape.
  - Hello Fruit proves the UX.
  - Rizful test NWC mode exercises the real wallet path for demos and tests.
  - Production NWC mode is possible without changing app code.
  - The frontend never sees wallet secrets.
  - Pure frontend live checkout is rejected by docs, examples, and tests.
  - Frontend packages are headless-first and proven with at least one external
    design-system recipe.
  - Framework adapters integrate with app-owned auth and fail closed when
    invoice access authorization is missing in production.
  - Every new SDK has a conformance gate.
  - Every standalone NWC client has a conformance gate.
  - Every new demo uses the same shared product data.
  - CI prevents generated/data drift.


Final Mental Model
------------------

OpenReceive should not become a loose pile of payment SDKs.

It should be:

  one spec,
  one Rizful test wallet policy,
  one provider registry,
  one price model,
  one test suite,
  one fruit-sticker demo,
  many idiomatic backend SDKs,
  many UI packages,
  enough standalone NWC clients to make the major server frameworks work
  without extra infrastructure.

That gives the project broad ecosystem coverage without drowning in duplicated
logic.
