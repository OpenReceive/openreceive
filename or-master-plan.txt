OpenReceive Master Plan
=======================

Status: v0.1-v2 architecture
Updated: 2026-06-22

Product Boundary
----------------

OpenReceive is a receive-checkout toolkit. It creates BOLT11 invoices through a
server-side receive-only NWC wallet, shows display-safe checkout UI, and verifies
settlement before app-owned fulfillment. It is not a wallet, exchange,
processor, custodian, broker, money transmitter, or sender API.

Non-negotiable boundaries:

- NWC secrets stay server-side.
- Browser, mobile, source maps, screenshots, docs, tests, and demo assets never
  expose NWC secrets.
- OpenReceive receive APIs never expose send-payment methods.
- Notifications are passive hints only.
- Settlement requires `settled_at`, `state == "settled"`, or
  `transaction_state == "settled"`. A preimage alone is not final proof.
- Public payloads use `amount_msats`.
- Fiat math uses exact integer/decimal arithmetic.

Architecture
------------

The default deployment is the app's web process:

```
browser/mobile -> /openreceive/v1 HTTP routes -> core -> OpenReceive KV store
                                                -> receive-only NWC lookup

optional scheduler -> POST /openreceive/v1/poll or openreceive poll --once
```

There is no required daemon, NWC listener, webhook bridge, SSE route, in-memory
event bus, or process-local coordination. The durable store is the only
coordination point across processes and replicas.

Core owns all lifecycle logic:

- invoice create idempotency and conflict detection
- pure state transitions
- per-invoice lookup cooldown
- global store-backed lookup token bucket
- bounded background sweep
- settlement-action store-backed claim
- duplicate-safe, at-least-once settlement hook delivery

Persistence Ownership
---------------------

OpenReceive owns its persistence. Apps do not map invoice rows through
ActiveRecord, Prisma, Drizzle, Eloquent, Doctrine, app migrations, or arbitrary
user tables.

Store contract:

- `putIfAbsent(record)`
- `put(record, expectedRev)`
- `get(invoiceId)`
- `getByPaymentHash(paymentHash)`
- `getByBolt11Invoice(invoice)`
- `getByIdempotencyScope(scopeKey)`
- `listOpen({ now, limit })`
- `getMeta(key)`
- `casMeta(key, value, expectedRev)`

Records are `{ rev, row }`, where `row` is the canonical invoice storage row.
SQL transports use a frozen package-owned schema with control columns for
uniqueness and recovery plus an opaque `data` blob. Meta rows hold ownership,
schema version, lookup bucket, and sweep clocks.

Store selection:

- `OPENRECEIVE_STORE=local-sqlite`
- `OPENRECEIVE_STORE=sqlite:/path/to/openreceive.sqlite3`
- `OPENRECEIVE_STORE=postgres://...`
- `OPENRECEIVE_STORE=memory:` for tests only
- `OPENRECEIVE_NAMESPACE=default` or another app-specific namespace

MySQL, Redis/Upstash, and Cloudflare Durable Object storage are target
transports but are not v0.1 certified until package-owned adapters pass the
storage KV conformance vector. MongoDB, Prisma-native models, Drizzle-native
models, arbitrary user tables, S3/object storage, and Cloudflare Workers KV are
not v0.1 supported storage targets.

HTTP Contract
-------------

Routes mount under `/openreceive/v1`:

- `POST /invoices`
- `GET /invoices/{invoice_id}`
- `POST /invoices/lookup`
- `POST /invoices/{invoice_id}/refresh`
- `POST /poll`
- `GET /rates`
- `POST /rates/quote`
- `GET /routes`
- `GET /providers`
- `GET /health`
- `GET /capabilities`

`/poll` is a protected one-shot recovery route for schedulers and operator
tooling. Lookup endpoints are gated status refreshes and may return stored state
without a wallet call.

There is no `/events` route in the default v0.1-v2 contract. Browser checkout
state is updated through lookup polling and display-safe backend responses.

Settlement Actions
------------------

Settlement hooks are at least once. OpenReceive prevents concurrent duplicate
execution with a store-backed claim, but a crash after the app hook succeeds and before
OpenReceive records completion can replay the hook. Apps must deduplicate by
`payment_hash` or use conditional fulfillment writes.

Framework Package Shape
-----------------------

Backends are transport-first, not framework/ORM-first:

- Core: storage contract, transitions, gated lookup, sweep, settlement lease.
- Node: receive-only NWC client, Postgres/SQLite KV stores, store URI resolver,
  CLI `init`, `doctor`, `migrate`, and `poll --once`.
- Express: route handlers, Fetch bridge, raw Node bridge, auth/CSRF hooks,
  gated lookup, protected poll, settlement action hook.
- Next: App Router wrapper around the shared Fetch bridge.
- Browser/elements/react/vue/svelte/angular: display-safe checkout, QR, copy,
  open-wallet, countdown, payment wizard, lookup polling, and refresh/retry
  controls. They never create invoices or hold NWC secrets.

Framework packages should not ship ORM invoice models, migrations, listener
processes, worker services, or frontend live checkout code.

Demos
-----

Hello Fruit demos prove the reference path:

- Express + React
- Static HTML + small API
- Next.js fullstack

The JS demos mount OpenReceive routes, use package-owned Postgres KV storage,
depend on server-side `OPENRECEIVE_NWC`, and expose an optional
`npm run openreceive:poll` script for one-shot recovery. They do not define
OpenReceive worker services or notification listeners.

Rails demos remain initial proof work, but they use the package-owned SQLite
store selected by `OPENRECEIVE_STORE` and do not ship app invoice models or
OpenReceive invoice migrations.

Docs And Contracts
------------------

Contract source of truth:

- `spec/openapi/openreceive-http.v1.yaml`
- `spec/asyncapi/openreceive-events.v1.yaml` for server-side lifecycle event
  names only
- `spec/schemas/**`
- `spec/test-vectors/**`

When a schema changes, update generated contracts and test vectors in the same
change. The docs must keep the route-driven, poll-only, store-owned architecture
as the default story.

Release Gate
------------

Before declaring v0.1 synchronized:

```
npm run typecheck
npm run check:generated
npm run test:js
npm run validate
npm run test:package-smoke
npm run test:ruby
```

Live wallet checks remain opt-in and must skip clearly when `OPENRECEIVE_NWC`
is not configured.
